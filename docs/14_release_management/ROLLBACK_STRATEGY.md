# ロールバック戦略

ServiceMatrix Rollback Strategy

Version: 1.0
Status: Active
Classification: Internal Operations Document
Last Updated: 2026-03-02

---

## 1. 概要

本ドキュメントは、ServiceMatrix プロジェクトにおけるロールバック（変更取り消し）の戦略・手順・判断基準を定義する。
デプロイ後に問題が検出された場合、迅速かつ安全にシステムを安定状態に戻す手順を定める。

---

## 2. ロールバック基本方針

### 2.1 原則

| 原則 | 説明 |
|------|------|
| 速度優先 | ロールバックは問題解決より速度を優先する |
| 準備必須 | すべてのデプロイ前にロールバック手順を準備する |
| 自動優先 | 可能な限りロールバックを自動化する |
| データ保護 | データの整合性を最優先にする |
| 記録必須 | ロールバック実施は必ず監査ログに記録する |
| 事後分析 | ロールバック後に必ず根本原因分析を実施する |

### 2.2 ロールバック対象

| 対象 | ロールバック手段 | 難易度 |
|------|---------------|--------|
| アプリケーションコード | 旧コンテナイメージへの切り替え | 低 |
| 設定変更 | 旧設定値への復元 | 低 |
| データベーススキーマ | ダウンマイグレーション実行 | 中〜高 |
| インフラ変更 | IaC（Terraform）での旧状態適用 | 中 |
| 依存サービス変更 | 旧バージョンへの切り替え | 中 |

## 3. ロールバック判断基準

### 3.1 自動ロールバックトリガー

以下の条件を検出した場合、システムが自動的にロールバックを実行する。

| トリガー条件 | 閾値 | 検出方法 |
|------------|------|---------|
| ヘルスチェック連続失敗 | 3回連続（30秒間隔） | Kubernetes Probe |
| エラー率急増 | デプロイ前比 300%以上増加 | Prometheus アラート |
| 応答時間超過（継続） | P95 > 500ms が5分継続 | Prometheus アラート |
| Podクラッシュループ | 3回以上の連続再起動 | Kubernetes 自動検出 |

### 3.2 手動ロールバック判断基準

以下の状況では担当者がロールバックを判断する。

| 状況 | 推奨対応 | 判断者 |
|------|---------|--------|
| P1インシデント発生（デプロイ起因疑い） | 即時ロールバック | DevOps + Incident Manager |
| データ不整合の検出 | 即時ロールバック + DBA協議 | DevOps + DBA |
| 主要機能の動作不良 | ロールバック検討 | DevOps + Change Manager |
| パフォーマンス劣化（P95 > 200ms） | ロールバック検討 | DevOps + テックリード |
| セキュリティ問題の検出 | 即時ロールバック | DevOps + セキュリティチーム |
| ユーザーからの大量クレーム | ロールバック検討 | Change Manager |

### 3.3 ロールバック不要の判断基準

以下の状況ではロールバックを行わず修正版をデプロイする。

| 状況 | 理由 |
|------|------|
| 軽微な表示バグ（機能影響なし） | ロールバックのリスクが修正コストを上回る |
| パッチ修正で対応可能な問題 | 修正版の方が適切 |
| ロールバックでデータ整合性が損なわれる | データ保護優先 |
| デプロイから24時間以上経過 | ロールバックリスクが高い |

## 4. ロールバック手順

### 4.1 アプリケーションロールバック手順

```
[ステップ1] ロールバック判断
  担当: DevOps + Change Manager
  内容: ロールバック要否の判断、影響範囲確認

[ステップ2] ロールバック通知
  担当: DevOps
  内容: Slack に開始通知、関係者への連絡

[ステップ3] Kubernetes デプロイメントのロールバック
  担当: DevOps
  コマンド:
    kubectl rollout undo deployment/servicematrix -n production
    kubectl rollout status deployment/servicematrix -n production

[ステップ4] ヘルスチェック確認
  担当: DevOps
  内容: /health エンドポイントの200 OK確認（全Pod）

[ステップ5] スモークテスト実行
  担当: DevOps
  内容: 主要APIエンドポイントの疎通確認

[ステップ6] エラー率・応答時間確認
  担当: DevOps
  内容: 監視ダッシュボードで指標が正常範囲内か確認

[ステップ7] ロールバック完了通知
  担当: DevOps
  内容: 完了を Slack・関係者に通知、インシデント記録更新

[ステップ8] 根本原因分析
  担当: テックリード + 開発チーム
  期限: ロールバック後 24時間以内
  内容: 問題の根本原因特定、再発防止策の策定
```

### 4.2 データベースロールバック手順

データベースロールバックは高リスクのため、DBAとの協議必須。

```
[ステップ1] ロールバック可否判断（DBA協議）
  確認: ダウンマイグレーションスクリプトの存在確認
  確認: データ損失リスクの評価
  確認: ロールバック後のデータ整合性確認方法の準備

[ステップ2] アプリケーションの一時停止（必要に応じ）
  目的: データ整合性保護

[ステップ3] フルバックアップ取得（ロールバック前）
  目的: ロールバック失敗時のリカバリ手段確保

[ステップ4] ダウンマイグレーション実行
  コマンド: npm run db:migrate:down -- --to {migration_id}
  監視: マイグレーション進捗・エラーの監視

[ステップ5] データ整合性検証
  内容: 重要テーブルのレコード数・データ内容の検証
  ツール: 事前に準備した検証クエリの実行

[ステップ6] アプリケーション再起動
  内容: 旧バージョンアプリケーションの起動確認
```

### 4.3 緊急ロールバックの判断タイムライン

| 経過時間 | アクション |
|---------|---------|
| 問題検出後 0〜5分 | 状況確認・ロールバック要否判断 |
| 5〜10分 | ロールバック開始（判断した場合） |
| 10〜20分 | ロールバック完了・正常動作確認 |
| 20〜60分 | インシデント記録・関係者通知 |
| 24時間以内 | 根本原因分析完了・再発防止策策定 |

## 5. ロールバック後の対応

### 5.1 必須実施事項

| 対応 | 期限 | 担当 |
|------|------|------|
| インシデント記録の更新 | 即時 | Incident Manager |
| 影響ユーザーへの通知 | 1時間以内 | Change Manager |
| 根本原因分析（RCA）実施 | 24時間以内 | テックリード |
| 再発防止策の策定 | 48時間以内 | 開発チーム |
| ポストモーテムドキュメント作成 | 7日以内 | テックリード |
| 変更記録への記録 | 即時 | DevOps |

### 5.2 ポストモーテムテンプレート

```
# ポストモーテムレポート

日時: {YYYY-MM-DD}
バージョン: {ロールバックしたバージョン}
影響時間: {開始〜終了の時刻と影響時間}
影響範囲: {影響を受けたサービス・ユーザー数}

## タイムライン
- HH:MM: 問題検出
- HH:MM: ロールバック開始
- HH:MM: ロールバック完了

```

## 6. ロールバックテスト

### 6.1 ロールバック訓練

ロールバック手順の有効性を定期的に検証する。

| 訓練種別 | 頻度 | 環境 | 目的 |
|---------|------|------|------|
| アプリケーションロールバック訓練 | 四半期 | Staging | 手順確認・時間測定 |
| DBロールバック訓練 | 半年 | Staging | 高リスク手順の習熟 |
| 緊急ロールバック訓練 | 年次 | Staging | インシデント対応シミュレーション |

### 6.2 訓練目標値

| 指標 | 目標値 |
|------|--------|
| アプリロールバック完了時間 | 10分以内 |
| DBロールバック完了時間 | 30分以内 |
| 正常動作確認完了時間 | 15分以内 |
| 訓練成功率 | 100% |
