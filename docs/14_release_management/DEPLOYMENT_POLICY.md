# デプロイポリシー

ServiceMatrix Deployment Policy

Version: 1.0
Status: Active
Classification: Internal Operations Document
Last Updated: 2026-03-02

---

## 1. 概要

本ドキュメントは、ServiceMatrix プロジェクトにおけるデプロイの方針・手順・権限・制約を定義する。
すべてのデプロイはITIL 4の変更管理プロセスおよびCI/CDパイプラインに従って実施される。

---

## 2. デプロイ原則

### 2.1 基本原則

| 原則 | 説明 |
|------|------|
| CI/CD優先 | すべてのデプロイはCI/CDパイプライン経由で実施する |
| 自動化 | 手動デプロイは例外時のみ許可し、必ず記録する |
| 変更管理連携 | デプロイは承認済みの変更要求（RFC）に紐づける |
| 品質ゲート | 品質ゲート（QG-01〜06）を通過したコードのみデプロイ可能 |
| ロールバック準備 | すべてのデプロイにロールバック手順を準備する |
| 監査記録 | すべてのデプロイ操作を監査ログに記録する |
| ゼロダウンタイム | 本番環境はゼロダウンタイムデプロイを目指す |

### 2.2 デプロイ禁止事項

| 禁止事項 | 理由 |
|---------|------|
| main ブランチへの直接プッシュ | ガバナンス違反 |
| 承認なしの本番デプロイ | 変更管理違反 |
| テスト未通過のコードのデプロイ | 品質違反 |
| 営業時間外の計画デプロイ（緊急時除く） | 対応体制確保のため |
| 手動での本番DB操作（緊急時除く） | データ整合性リスク |
| デプロイ記録の省略 | 監査要件違反 |

## 3. デプロイ環境

### 3.1 環境定義

| 環境 | 略称 | 用途 | デプロイ頻度 | 承認要件 |
|------|------|------|------------|---------|
| Development | dev | 統合開発・テスト | 毎PR | なし（CI自動） |
| Staging | stg | リリース前検証 | main マージ毎 | なし（CI自動） |
| Production | prod | 本番運用 | リリース計画に従う | Change Manager承認必須 |

### 3.2 環境別設定管理

| 設定項目 | 管理方法 | アクセス権 |
|---------|---------|-----------|
| 環境変数 | GitHub Secrets / Kubernetes Secrets | DevOps + Admin |
| 設定ファイル | リポジトリ内（秘密情報除く） | 開発者 |
| シークレット | Vault / GitHub Secrets | DevOps + Admin |
| インフラ設定 | IaC（Terraform / Helm） | DevOps |

## 4. デプロイフロー

### 4.1 通常デプロイフロー

```
開発者 PR 作成
  |
  v
CI/CD パイプライン（QG-01〜03）
  |
  v
Development 自動デプロイ
  |
  v
E2E テスト + セキュリティスキャン（QG-04）
  |
  v
Staging 自動デプロイ
  |
  v
[リリース判定]
  |
  v
Change Manager 承認（QG-05）
  |
  v
リリース PR 作成（release-please）
  |
  v
本番デプロイ（CI/CD 自動実行）
  |
  v
デプロイ後検証（スモークテスト・監視確認）
```

### 4.2 緊急デプロイフロー（Hotfix）

```
P1 インシデント発生
  |
  v
Incident Manager が緊急変更承認
  |
  v
hotfix ブランチ作成
  |
  v
最小限のテスト実施（QG-06縮退版）
  |
  v
テックリード + Change Manager 緊急承認
  |
  v
本番デプロイ（手動承認トリガー）
  |
  v
事後報告・テスト補完（48時間以内）
```

## 5. デプロイ権限マトリクス

### 5.1 環境別デプロイ権限

| 操作 | Developer | QAエンジニア | DevOps | Change Manager | Admin |
|------|-----------|------------|--------|---------------|-------|
| Development デプロイ（CI経由） | Yes | Yes | Yes | Yes | Yes |
| Development デプロイ（手動） | No | No | Yes | No | Yes |
| Staging デプロイ（CI経由） | Yes（PR merge後） | Yes | Yes | Yes | Yes |
| Staging デプロイ（手動） | No | No | Yes | No | Yes |
| Production デプロイ（CI経由） | No | No | No | Yes（承認後） | Yes |
| Production デプロイ（手動） | No | No | No | No | Yes（緊急時） |
| ロールバック実行 | No | No | Yes | Yes | Yes |
| DB マイグレーション | No | No | Yes | No | Yes |

## 6. デプロイウィンドウ

### 6.1 本番デプロイ可能時間帯

| 曜日 | 可能時間帯 | 備考 |
|------|---------|------|
| 月〜木 | 10:00〜16:00（JST） | コアタイム内、対応チームが揃っている |
| 金 | 10:00〜14:00（JST） | 週末前リスク回避のため短縮 |
| 土・日 | 禁止（緊急時除く） | オンコール体制が必要 |
| 祝日 | 禁止（緊急時除く） | オンコール体制が必要 |

### 6.2 ブラックアウト期間（デプロイ禁止期間）

| 期間 | 理由 |
|------|------|
| 期末決算期（3月・9月最終週） | 業務最繁忙期 |
| 大型連休前後（GW・年末年始前後2日） | 人員確保困難 |
| システム障害復旧中 | 影響拡大リスク |
| J-SOX 監査期間中（原則） | Change Freeze |

## 7. デプロイ手順

### 7.1 本番デプロイ前チェックリスト

| カテゴリ | チェック項目 | 確認者 |
|---------|------------|--------|
| 変更管理 | 承認済みRFCの存在確認 | Change Manager |
| 品質 | 全テスト通過確認 | QAエンジニア |
| セキュリティ | セキュリティスキャン通過確認 | セキュリティチーム |
| バックアップ | DBフルバックアップ取得確認 | DevOps |
| ロールバック | ロールバック手順の準備確認 | DevOps |
| 通知 | 関係者への事前通知完了確認 | Change Manager |
| 監視 | 監視ダッシュボードの正常確認 | DevOps |
| 担当者 | デプロイ担当者と連絡先の確認 | DevOps |

### 7.2 デプロイ実行手順

| ステップ | 作業内容 | 担当 | タイムアウト |
|---------|---------|------|------------|
| 1 | デプロイ開始の Slack通知 | DevOps | - |
| 2 | GitHubActions 手動承認トリガー | Change Manager | - |
| 3 | コンテナイメージデプロイ | CI/CD自動 | 15分 |
| 4 | DBマイグレーション実行 | CI/CD自動 | 10分 |
| 5 | ヘルスチェック確認 | CI/CD自動 | 5分 |
| 6 | スモークテスト実行 | CI/CD自動 | 10分 |
| 7 | エラー率・応答時間確認 | DevOps | 15分 |
| 8 | デプロイ完了通知 | DevOps | - |

### 7.3 デプロイ後監視期間

| 時間帯 | 監視頻度 | 対応者 |
|--------|---------|--------|
| デプロイ後 0〜30分 | 5分毎 | DevOps（担当者待機） |
| デプロイ後 30分〜2時間 | 15分毎 | DevOps（待機継続） |
| デプロイ後 2〜24時間 | 1時間毎（アラート監視） | DevOps（通常体制） |

## 8. デプロイ手法

### 8.1 採用デプロイ手法

| 手法 | 現在の採用状況 | 説明 |
|------|--------------|------|
| ローリングアップデート | 採用中 | 旧バージョンを段階的に新バージョンに置換 |
| Blue-Green デプロイ | 計画中（Phase 3） | 本番環境を2つ用意し切り替え |
| カナリアリリース | 計画中（Phase 4） | 一部トラフィックのみ新バージョンへ |
| フィーチャーフラグ | 検討中 | 機能単位での段階的リリース |

### 8.2 ローリングアップデート設定（現在）

| パラメータ | 設定値 | 説明 |
|-----------|--------|------|
| maxSurge | 1 | 最大追加Pod数 |
| maxUnavailable | 0 | 最大停止Pod数（ゼロダウンタイム） |
| readinessProbe | /health（200 OK） | 新Pod投入前確認 |
| minReadySeconds | 30 | 安定確認待機時間 |
| progressDeadlineSeconds | 600 | タイムアウト（10分） |

## 9. データベースデプロイ

### 9.1 マイグレーション方針

| 方針 | 説明 |
|------|------|
| 後方互換性 | マイグレーションは後方互換性を維持して設計する |
| 展開前適用 | DBマイグレーションはアプリデプロイ前に実行する |
| ロールバック設計 | すべてのマイグレーションにダウン処理を実装する |
| 段階適用 | 大規模マイグレーションは段階的に分割して実施する |
| バックアップ | マイグレーション実行前に必ずフルバックアップを取得する |

### 9.2 マイグレーション禁止パターン

| パターン | リスク | 代替手法 |
|---------|-------|---------|
| 本番データの即時削除 | データ損失 | ソフトデリート → 後日物理削除 |
| NOT NULL制約の即時追加 | デプロイ失敗 | デフォルト値付き追加 → 後日制約追加 |
| 大テーブルの一括更新 | 長時間ロック | バッチ処理で分割更新 |
| インデックスの同期作成 | テーブルロック | CONCURRENTLY オプション使用 |
