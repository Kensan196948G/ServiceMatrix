# 品質ゲート定義

ServiceMatrix Quality Gate Definition

Version: 1.0
Status: Active
Classification: Internal Quality Document
Last Updated: 2026-03-02

---

## 1. 概要

本ドキュメントは、ServiceMatrix プロジェクトにおける品質ゲートを定義する。
品質ゲートは、ソフトウェアの品質が一定基準を満たさない限り
次のフェーズに進むことを許可しない統制ポイントである。

---

## 2. 品質ゲート一覧

### 2.1 ゲート構成

```
[GQ-1] コミット品質ゲート
  |
  v
[GQ-2] PR品質ゲート
  |
  v
[GQ-3] ビルド品質ゲート
  |
  v
[GQ-4] ステージング品質ゲート
  |
  v
[GQ-5] リリース品質ゲート
  |
  v
[GQ-6] 本番デプロイ品質ゲート
```

### 2.2 ゲート概要

| ゲートID | ゲート名 | タイミング | 判定方法 |
|---------|---------|-----------|---------|
| GQ-1 | コミット品質ゲート | git commit 時 | 自動（pre-commit hook） |
| GQ-2 | PR品質ゲート | PR作成・更新時 | 自動（CI/CD） + 人間レビュー |
| GQ-3 | ビルド品質ゲート | ビルド完了時 | 自動（CI/CD） |
| GQ-4 | ステージング品質ゲート | Stagingデプロイ後 | 自動 + 手動検証 |
| GQ-5 | リリース品質ゲート | リリース判定時 | 自動 + 人間承認 |
| GQ-6 | 本番デプロイ品質ゲート | 本番デプロイ前後 | 自動 + 人間確認 |

## 3. GQ-1: コミット品質ゲート

### 3.1 チェック項目

| チェック | ツール | 閾値 | ブロック |
|---------|--------|------|---------|
| Lintチェック | ESLint | エラー0件 | Yes |
| フォーマットチェック | Prettier | 違反0件 | Yes |
| 型チェック | TypeScript (tsc) | エラー0件 | Yes |
| シークレットスキャン | GitGuardian | 検出0件 | Yes |
| コミットメッセージ形式 | commitlint | Conventional Commits準拠 | Yes |

### 3.2 実装方法

pre-commit hook（husky + lint-staged）で自動実行する。

```
pre-commit hook 実行フロー:
  1. lint-staged によるステージ済みファイルのLint
  2. Prettier によるフォーマット確認
  3. tsc --noEmit による型チェック
  4. シークレットスキャン
  5. commitlint によるメッセージ検証
```

## 4. GQ-2: PR品質ゲート

### 4.1 自動チェック項目

| チェック | ツール | 閾値 | ブロック |
|---------|--------|------|---------|
| ユニットテスト成功率 | Vitest | 100% | Yes |
| コードカバレッジ（行） | c8 | 80%以上 | Yes |
| コードカバレッジ（分岐） | c8 | 75%以上 | Yes |
| 統合テスト成功率 | Vitest + Supertest | 100% | Yes |
| SASTスキャン | CodeQL + Semgrep | Critical/High 0件 | Yes |
| SCAスキャン | Dependabot / Snyk | Critical 0件 | Yes |
| Markdownlint | markdownlint-cli | エラー0件 | Yes（docsのみ） |
| バンドルサイズ | webpack-bundle-analyzer | 閾値以下 | Yes |
| ライセンスチェック | license-checker | 非互換0件 | Yes |

### 4.2 人間レビュー要件

| 要件 | 条件 | 必要承認数 |
|------|------|-----------|
| コードレビュー | 全PR | 最低1名 |
| セキュリティレビュー | 認証・認可・暗号化の変更 | セキュリティチーム1名 |
| アーキテクチャレビュー | 構造的変更 | テックリード1名 |
| ドキュメントレビュー | docs/ 配下の変更 | 1名 |
| SoD（職務分離）チェック | 全PR | PR作成者以外が承認 |

### 4.3 PR品質スコア

PRの品質を定量的にスコア化する。

| カテゴリ | 配点 | 評価基準 |
|---------|------|---------|
| テスト | 30点 | カバレッジ率×30 |
| セキュリティ | 25点 | 脆弱性なし:25, Low:20, Medium:10 |
| コード品質 | 20点 | Lint違反なし:20, 軽微:15 |
| ドキュメント | 15点 | 変更に対応するドキュメント更新あり:15 |
| レビュー | 10点 | 全承認:10, 一部:5 |

合格基準: 80点以上

## 5. GQ-3: ビルド品質ゲート

### 5.1 チェック項目

| チェック | 条件 | ブロック |
|---------|------|---------|
| ビルド成功 | エラーなしでビルド完了 | Yes |
| ビルド時間 | 5分以内 | Warning（10分超でブロック） |
| Dockerイメージビルド | イメージ生成成功 | Yes |
| コンテナスキャン | Critical脆弱性0件 | Yes |
| アーティファクトサイズ | Dockerイメージ500MB以下 | Warning |
| ヘルスチェック | コンテナ起動後のヘルスチェック成功 | Yes |

## 6. GQ-4: ステージング品質ゲート

### 6.1 自動チェック項目

| チェック | ツール | 閾値 | ブロック |
|---------|--------|------|---------|
| E2Eテスト成功率 | Playwright | 98%以上 | Yes |
| DASTスキャン | OWASP ZAP | Critical/High 0件 | Yes |
| パフォーマンステスト | k6 | P95応答時間 < 200ms | Yes |
| スモークテスト | カスタムスクリプト | 全項目パス | Yes |
| ヘルスチェック | HTTP GET /health | 200 OK | Yes |
| WebSocket接続テスト | カスタムスクリプト | 接続・メッセージ配信成功 | Yes |

### 6.2 手動検証項目

| 検証 | 担当 | 基準 |
|------|------|------|
| 主要フロー動作確認 | QA | 全クリティカルフロー正常 |
| UI表示確認 | QA | レイアウト崩れなし |
| レスポンシブ確認 | QA | モバイル・タブレット表示正常 |
| アクセシビリティ | QA | WCAG 2.1 AA 主要項目準拠 |

## 7. GQ-5: リリース品質ゲート

### 7.1 リリース判定基準

| カテゴリ | チェック項目 | 閾値 | 必須 |
|---------|------------|------|------|
| テスト | 全テストスイート成功 | 100%（E2E: 98%） | Yes |
| セキュリティ | SAST + DAST + SCA クリア | Critical/High 0件 | Yes |
| パフォーマンス | ベースライン比較 | 20%以上の悪化なし | Yes |
| カバレッジ | コードカバレッジ | 80%以上（行） | Yes |
| ドキュメント | リリースノート作成 | 作成済み | Yes |
| 変更管理 | 全変更のRFC承認済み | 未承認0件 | Yes |
| 既知バグ | Critical/High未解決 | 0件 | Yes |
| SLA | SLA違反リスク評価 | リスク低 | Yes |
| 承認 | Change Manager承認 | 承認済み | Yes |
| 承認 | テックリード承認 | 承認済み | Yes |

### 7.2 リリース承認フロー

```
リリース候補生成
  |
  v
自動品質チェック実行
  |
  v
[全チェック合格?]
  +-- No  --> 問題修正 → 再チェック
  +-- Yes --> リリースレポート生成
                |
                v
            テックリードレビュー
                |
                v
            Change Manager承認
                |
                v
            [高リスク変更含む?]
              +-- Yes --> Admin追加承認
              +-- No  --> リリース承認完了
```

### 7.3 リリースブロック条件

以下のいずれかに該当する場合、リリースをブロックする。

| 条件 | 理由 |
|------|------|
| Critical/High脆弱性の存在 | セキュリティリスク |
| ユニット/統合テスト失敗 | コード品質不足 |
| E2Eテスト成功率98%未満 | ユーザー影響リスク |
| パフォーマンス回帰検出 | SLA違反リスク |
| 未承認の変更要求の存在 | ガバナンス違反 |
| カバレッジ閾値未達 | テスト不足 |
| リリースノート未作成 | 運用準備不足 |

## 8. GQ-6: 本番デプロイ品質ゲート

### 8.1 デプロイ前チェック

| チェック | 担当 | 条件 |
|---------|------|------|
| デプロイウィンドウ確認 | DevOps | メンテナンスウィンドウ内 |
| バックアップ完了確認 | DevOps | フルバックアップ取得済み |
| ロールバック手順確認 | DevOps | ロールバック手順準備済み |
| 監視ダッシュボード確認 | DevOps | 監視ダッシュボード表示正常 |
| 関係者通知 | Change Manager | 関係者への事前通知完了 |

### 8.2 デプロイ後チェック

| チェック | 自動/手動 | 閾値 | 対応 |
|---------|----------|------|------|
| ヘルスチェック | 自動 | 全エンドポイント200 OK | 失敗時ロールバック |
| スモークテスト | 自動 | 全項目パス | 失敗時ロールバック |
| エラー率監視 | 自動 | 前回比増加なし | 増加時アラート |
| 応答時間監視 | 自動 | P95 < 200ms | 閾値超過時アラート |
| ログ監視 | 自動 | 異常ログパターンなし | 検出時調査 |
| ユーザー影響確認 | 手動 | 主要フロー正常 | 問題時ロールバック判定 |

### 8.3 ロールバック判断基準

| 条件 | 判断 | 判断者 |
|------|------|--------|
| ヘルスチェック連続3回失敗 | 即時ロールバック | 自動 |
| エラー率5%超過 | 即時ロールバック | 自動 |
| P95応答時間500ms超過（5分継続） | ロールバック検討 | DevOps |
| ユーザー報告のクリティカル問題 | ロールバック検討 | Incident Manager |
| データ不整合の検出 | 即時ロールバック | DevOps + DBA |

## 9. 品質ゲート例外プロセス

### 9.1 例外申請

品質ゲートの例外（一時的な基準緩和）が必要な場合。

| 項目 | 要件 |
|------|------|
| 申請 | Change Issue で例外理由を明示 |
| 承認 | テックリード + Change Manager |
| 有効期限 | 最大30日（延長はAdmin承認） |
| 条件 | 代替の品質保証手段を明示 |
| 記録 | 監査ログに例外適用を記録 |
| フォローアップ | 期限内に正規基準への復帰計画を提出 |

### 9.2 例外対象外（緩和不可）

| 項目 | 理由 |
|------|------|
| Critical セキュリティ脆弱性 | セキュリティリスクは許容不可 |
| SoD違反 | コンプライアンス要件 |
| 監査ログ記録の欠如 | J-SOX準拠要件 |
| 認証・認可の不具合 | セキュリティの根幹 |

## 10. 品質メトリクスダッシュボード

### 10.1 追跡メトリクス

| メトリクス | 計算方法 | 目標 |
|-----------|---------|------|
| 品質ゲート通過率 | 初回通過PR数 / 全PR数 | 80%以上 |
| 平均修正時間 | ゲート失敗〜再通過の平均時間 | 2時間以内 |
| カバレッジトレンド | 週次カバレッジ推移 | 上昇傾向 |
| セキュリティ修正時間 | 脆弱性検出〜修正の平均時間 | Critical: 24h, High: 7d |
| テスト実行時間トレンド | CI全テスト実行時間の推移 | 15分以内を維持 |
| フレーキーテスト率 | 非決定的テスト件数 / 全テスト件数 | 2%以下 |
| リリースブロック率 | GQ-5でブロックされたリリース数 / 全リリース数 | 10%以下 |
| 本番インシデント率 | リリース後72時間以内のインシデント数 | 0件 |

### 10.2 アラート設定

| メトリクス | Warning閾値 | Critical閾値 | 通知先 |
|-----------|-----------|-------------|--------|
| 品質ゲート通過率 | 70%未満 | 60%未満 | テックリード |
| カバレッジ率 | 78%未満 | 75%未満 | 開発チーム |
| テスト実行時間 | 12分超過 | 15分超過 | DevOps |
| フレーキーテスト率 | 3%超過 | 5%超過 | 開発チーム |
| セキュリティ修正遅延 | Critical 12h超過 | Critical 24h超過 | セキュリティチーム |

## 11. AI Agent による品質ゲート支援

### 11.1 AI支援範囲

| 支援内容 | 自律レベル | 説明 |
|---------|-----------|------|
| 品質スコア自動計算 | L2（補助） | PRの品質スコアを自動算出しコメント |
| 修正提案 | L1（助言） | ゲート失敗時の修正案を提示 |
| トレンド分析 | L2（補助） | 品質メトリクスの傾向分析・異常検知 |
| リリース判定支援 | L1（助言） | リリース可否の推奨を提示（最終判断は人間） |

### 11.2 AI判断の監査

AI Agent による品質判断はすべて監査ログに記録される。

| 記録項目 | 内容 |
|---------|------|
| 入力 | 判断に使用したメトリクス・データ |
| ロジック | 適用したルール・閾値 |
| 出力 | 判断結果・推奨事項 |
| タイムスタンプ | 判断実行日時（UTC） |
| 信頼度 | 判断の信頼度スコア |
