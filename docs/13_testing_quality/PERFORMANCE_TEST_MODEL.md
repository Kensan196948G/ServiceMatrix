# パフォーマンステストモデル

ServiceMatrix Performance Test Model

Version: 1.0
Status: Active
Classification: Internal Quality Document
Last Updated: 2026-03-02

---

## 1. 概要

本ドキュメントは、ServiceMatrix プロジェクトにおけるパフォーマンステストの
モデル・手法・基準値を定義する。
SLA要件に基づくパフォーマンス目標を設定し、
システムの応答性・スループット・スケーラビリティを体系的に検証する。

---

## 2. パフォーマンステスト種別

### 2.1 テスト種別定義

| テスト種別 | 目的 | 実行タイミング |
|-----------|------|--------------|
| ベースラインテスト | 通常負荷での基準値測定 | リリース前 |
| 負荷テスト | 想定最大負荷での動作検証 | リリース前 |
| ストレステスト | 限界負荷での耐久性検証 | 四半期 |
| スパイクテスト | 急激な負荷増加への対応検証 | 四半期 |
| 耐久テスト（ソークテスト） | 長時間稼働での安定性検証 | 月次 |
| スケーラビリティテスト | スケールアウト時の性能向上確認 | 半年 |

### 2.2 テスト対象

| 対象 | 内容 | 優先度 |
|------|------|--------|
| REST API | 全エンドポイントの応答時間・スループット | P1 |
| WebSocket | リアルタイム通知の遅延・接続数 | P1 |
| データベース | クエリ応答時間・接続数 | P1 |
| ダッシュボード | 画面描画時間・データ取得時間 | P2 |
| バックグラウンドジョブ | SLA計測・自動分類・通知の処理時間 | P2 |
| 外部連携 | GitHub API 呼び出しの応答時間 | P2 |

## 3. パフォーマンス基準値（SLA連動）

| エンドポイントカテゴリ | P50 | P95 | P99 | 最大 |
|----------------------|-----|-----|-----|------|

| 単一リソース取得 | 20ms | 100ms | 200ms | 500ms |
| リソース作成 | 50ms | 200ms | 500ms | 1000ms |
| リソース更新 | 30ms | 150ms | 300ms | 800ms |
| リソース削除 | 30ms | 150ms | 300ms | 800ms |
| 検索（全文検索） | 100ms | 500ms | 1000ms | 2000ms |
| レポート生成 | 500ms | 2000ms | 5000ms | 10000ms |
| ダッシュボードデータ | 50ms | 200ms | 500ms | 1000ms |

### 3.2 スループット基準

| シナリオ | 目標スループット | 同時ユーザー |
|---------|----------------|-------------|
| 通常運用 | 500 req/sec | 50 |
| ピーク時 | 1000 req/sec | 100 |
| 最大負荷 | 2000 req/sec | 200 |

| 指標 | 基準値 |
|------|--------|
| 単純クエリ応答時間（P95） | 10ms以下 |
| 結合クエリ応答時間（P95） | 50ms以下 |

## 4. テスト技術スタック

### 4.1 ツール選定

| ツール | 用途 | 選定理由 |
|--------|------|---------|
| k6 | 負荷テスト実行 | JavaScript ベース、CI/CD統合容易 |
| Grafana | メトリクス可視化 | k6連携、リアルタイムダッシュボード |
| InfluxDB | メトリクス保存 | 時系列データの効率的な保存 |
| Lighthouse | フロントエンドパフォーマンス | Web Vitals 自動測定 |
| pg_stat_statements | DBクエリ分析 | PostgreSQL クエリ統計 |

### 4.2 テスト環境要件

| 項目 | 要件 |
|------|------|
| テスト環境 | Staging（本番同等構成） |
| データ量 | 本番同等（匿名化済み） |
| ネットワーク | テストクライアントとStagingが同一リージョン |
| 隔離 | パフォーマンステスト中は他テストを停止 |
| モニタリング | テスト中のリソース使用状況を記録 |

## 6. フロントエンドパフォーマンス

### 6.1 Web Vitals 基準

| 指標 | 説明 | 目標値 | 測定ツール |
|------|------|--------|-----------|
| LCP (Largest Contentful Paint) | 最大コンテンツの表示時間 | 2.5秒以下 | Lighthouse |
| FID (First Input Delay) | 最初の入力応答遅延 | 100ms以下 | Lighthouse |
| CLS (Cumulative Layout Shift) | レイアウトのずれ | 0.1以下 | Lighthouse |
| FCP (First Contentful Paint) | 最初のコンテンツ表示 | 1.8秒以下 | Lighthouse |
| TTFB (Time to First Byte) | 最初のバイト受信 | 800ms以下 | Lighthouse |
| TTI (Time to Interactive) | 操作可能になるまで | 3.5秒以下 | Lighthouse |

### 6.2 バンドルサイズ基準

| 対象 | 最大サイズ | 測定方法 |
|------|-----------|---------|
| 初期バンドル（gzip） | 200KB | webpack-bundle-analyzer |
| 個別ページチャンク | 100KB | 動的インポート分析 |
| 画像アセット（個別） | 200KB | ビルドプロセス |
| 総アセットサイズ | 1MB | Lighthouse |

### 6.3 ページ別パフォーマンス基準

| ページ | LCP目標 | TTI目標 |
|--------|---------|---------|
| ログインページ | 1.5秒 | 2.0秒 |
| ダッシュボード | 2.5秒 | 3.5秒 |
| インシデント一覧 | 2.0秒 | 3.0秒 |
| インシデント詳細 | 1.5秒 | 2.5秒 |
| CMDB一覧 | 2.5秒 | 3.5秒 |
| 監査ログ | 2.0秒 | 3.0秒 |
| レポート画面 | 3.0秒 | 4.0秒 |

## 7. データベースパフォーマンス

### 7.1 クエリパフォーマンス監視

| 監視項目 | 閾値 | アラート |
|---------|------|---------|
| スロークエリ（100ms超） | 発生率1%以下 | 1%超過でWarning |
| スロークエリ（1000ms超） | 発生率0件 | 1件でも発生時にCritical |
| テーブルスキャン | 発生率0件（大テーブル） | 発生時にWarning |
| コネクション使用率 | 70%以下 | 80%でWarning、90%でCritical |
| デッドロック | 0件/日 | 1件でもCritical |

### 7.2 データ量別パフォーマンス基準

| データ規模 | インシデント件数 | CI数 | 応答時間目標（P95） |
|-----------|----------------|------|-------------------|
| 小規模 | 1,000件 | 100 | 一覧: 50ms |
| 中規模 | 10,000件 | 1,000 | 一覧: 100ms |
| 大規模 | 100,000件 | 10,000 | 一覧: 200ms |
| 超大規模 | 1,000,000件 | 100,000 | 一覧: 500ms |

### 7.3 インデックス戦略の検証

パフォーマンステスト時に以下を確認する。

| 確認項目 | 方法 |
|---------|------|
| クエリプラン | EXPLAIN ANALYZE による実行計画確認 |
| インデックス使用率 | pg_stat_user_indexes による統計確認 |
| 未使用インデックス | pg_stat_user_indexes で scan 回数 0 のインデックス検出 |
| インデックスサイズ | pg_relation_size による肥大化検出 |

## 8. パフォーマンス回帰検出

### 8.1 ベースライン管理

| 項目 | 管理方法 |
|------|---------|
| ベースライン測定 | リリース毎に全エンドポイントのP50/P95/P99を記録 |
| 保存先 | InfluxDB + Grafana ダッシュボード |
| 保持期間 | 12ヶ月 |
| 比較方法 | 前回リリースのベースラインと比較 |

### 8.2 回帰判定基準

| 指標 | 回帰判定閾値 | アクション |
|------|-------------|-----------|
| P95応答時間 | 前回比20%以上悪化 | リリースブロック |
| P99応答時間 | 前回比30%以上悪化 | リリースブロック |
| スループット | 前回比15%以上低下 | リリースブロック |
| エラー率 | 前回比で増加 | リリースブロック |
| メモリ使用量 | 前回比30%以上増加 | 調査必要 |
| CPU使用率 | 前回比30%以上増加 | 調査必要 |

## 9. テスト結果レポート

### 9.1 レポート項目

| セクション | 内容 |
|-----------|------|
| エグゼクティブサマリ | 合否判定、主要メトリクス、推奨事項 |
| 応答時間分析 | エンドポイント別P50/P95/P99、時系列推移 |
| スループット分析 | 秒間リクエスト数、成功/失敗率 |
| リソース使用状況 | CPU、メモリ、ディスクI/O、ネットワーク |
| エラー分析 | エラー種別、発生頻度、根本原因 |
| ボトルネック分析 | 律速箇所の特定と改善提案 |
| ベースライン比較 | 前回リリースとの差分 |
| 推奨事項 | パフォーマンス改善の具体的提案 |

### 9.2 配信先と頻度

| レポート | 頻度 | 配信先 |
|---------|------|--------|
| リリーステストレポート | リリース毎 | 開発チーム + DevOps |
| 週次パフォーマンスサマリ | 週次 | テックリード |
| 月次パフォーマンス傾向 | 月次 | マネジメント |
| 回帰検出アラート | 随時 | 開発チーム（即時通知） |
