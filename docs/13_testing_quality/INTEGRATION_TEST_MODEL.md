# 統合テストモデル

ServiceMatrix Integration Test Model

Version: 1.0
Status: Active
Classification: Internal Quality Document
Last Updated: 2026-03-02

---

## 1. 概要

本ドキュメントは、ServiceMatrix プロジェクトにおける統合テストの
設計モデル・実装方針・実行戦略を定義する。
統合テストは、複数のコンポーネント間の連携を検証し、
システム全体の整合性を保証する。

---

## 2. 統合テストの範囲

### 2.1 統合テストの定義

統合テストは以下の連携ポイントを検証する。

| 連携種別 | 対象 | 検証内容 |
|---------|------|---------|
| API-DB連携 | RESTエンドポイント → データベース | CRUD操作の正確性 |
| API-外部サービス連携 | RESTエンドポイント → GitHub API | 外部API呼び出しの正確性 |
| サービス間連携 | サービスA → サービスB | メッセージング・データフロー |
| 認証・認可連携 | API Gateway → 認証サービス | トークン検証・権限チェック |
| イベント連携 | Webhook受信 → イベントハンドラ | イベント処理の正確性 |
| キャッシュ連携 | サービス → Redis | キャッシュの整合性 |

### 2.2 統合テストと他テストの境界

| テスト種別 | 対象範囲 | 外部依存 | 実行速度 |
|-----------|---------|---------|---------|
| ユニットテスト | 単一モジュール | すべてモック | ms単位 |
| 統合テスト | 複数モジュール連携 | DB実体・外部はモック | 秒単位 |
| E2Eテスト | システム全体 | すべて実体 | 分単位 |

## 3. 技術スタックと環境

### 3.1 テスト技術スタック

| 項目 | 技術 | 用途 |
|------|------|------|
| テストランナー | Vitest | テスト実行 |
| HTTP テスト | Supertest | API エンドポイントテスト |
| コンテナ管理 | Testcontainers | PostgreSQL / Redis コンテナ |
| 外部APIモック | MSW (Mock Service Worker) | GitHub API 等のモック |
| データベースツール | Prisma / Drizzle | マイグレーション・シード |
| アサーション | Vitest built-in | 結果検証 |

### 3.2 テスト環境構成

```
統合テスト実行環境
+--------------------------------------------+
|  Vitest テストランナー                       |
|                                            |
|  +----------+  +----------+  +-----------+ |
|  | Supertest|  | MSW      |  | テスト    | |
|  | (API)    |  | (外部API)|  | ヘルパー  | |
|  +----+-----+  +----+-----+  +-----+-----+ |
|       |             |              |        |
|  +----v-------------v--------------v-----+  |
|  |        アプリケーションサーバ          |  |
|  |        (テストモード起動)              |  |
|  +----+------------------+---------------+  |
|       |                  |                  |
+-------|------------------|------------------+
        |                  |
   +----v----+       +-----v-----+
   |PostgreSQL|       |   Redis   |
   |(container)|      |(container)|
   +----------+       +-----------+
```

### 3.3 テストコンテナのライフサイクル

| フェーズ | タイミング | 処理内容 |
|---------|-----------|---------|
| 起動 | テストスイート開始前（beforeAll） | コンテナ起動、マイグレーション実行 |
| シード | 各テストファイル開始前 | テストデータ投入 |
| クリーン | 各テスト終了後（afterEach） | トランザクションロールバック |
| 破棄 | テストスイート終了後（afterAll） | コンテナ停止・削除 |

## 4. テスト設計パターン

### 4.1 API統合テストパターン

| パターン | 説明 | 適用場面 |
|---------|------|---------|
| Request-Response | HTTPリクエスト送信→レスポンス検証 | RESTエンドポイント |
| Data Round-Trip | 作成→取得→更新→削除の一連検証 | CRUDエンドポイント |
| Error Propagation | エラー発生→適切なHTTPステータス返却 | エラーハンドリング |
| Auth Integration | 認証トークン付リクエスト→権限検証 | 認証・認可 |
| Pagination | ページネーションパラメータ→正しいデータ返却 | 一覧取得API |
| Webhook Processing | Webhookペイロード受信→処理結果検証 | Webhook受信エンドポイント |

### 4.2 データベース統合テストパターン

| パターン | 説明 | 適用場面 |
|---------|------|---------|
| Transaction Rollback | テスト毎にトランザクションをロールバック | データ分離 |
| Seed and Verify | テストデータ投入→クエリ結果検証 | 複雑なクエリ |
| Migration Test | マイグレーション実行→スキーマ検証 | スキーマ変更時 |
| Constraint Test | 制約違反→適切なエラー | データ整合性 |
| Concurrent Access | 並行アクセス→データ一貫性確認 | 排他制御 |

### 4.3 外部サービス統合テストパターン

| パターン | 説明 | 適用場面 |
|---------|------|---------|
| Mock Response | MSWで正常レスポンスを返却 | 正常系テスト |
| Error Simulation | MSWでエラーレスポンスを返却 | 異常系テスト |
| Timeout Simulation | MSWで遅延レスポンスを返却 | タイムアウト検証 |
| Rate Limit Simulation | MSWで429レスポンスを返却 | レート制限対応 |
| Retry Verification | リトライ回数・間隔の検証 | リトライロジック |

## 5. 統合テスト対象マトリクス

### 5.1 コンポーネント別テスト対象

| コンポーネント | テスト対象 | 優先度 |
|--------------|-----------|--------|
| Incident Manager | インシデントCRUD、状態遷移、SLA計測 | P1 |
| Change Manager | 変更要求CRUD、承認フロー、リスク評価 | P1 |
| Problem Manager | 問題CRUD、根本原因分析、既知エラー登録 | P2 |
| Request Manager | サービス要求CRUD、カタログ管理 | P2 |
| CMDB Core | CI登録・更新、依存関係管理、影響分析 | P1 |
| SLA Engine | SLA計測、閾値監視、違反検出 | P1 |
| PR Governance | PR作成・承認、SoD検証、マージ制御 | P1 |
| Audit Trail | 監査ログ記録、検索、エクスポート | P1 |
| Agent Coordinator | AI提案生成、人間承認ゲート、実行制御 | P1 |
| Webhook Receiver | Webhook受信、署名検証、イベント振り分け | P2 |
| Auth Service | OAuth認証、セッション管理、RBAC | P1 |

### 5.2 連携ポイント別テスト対象

| 連携 | テスト内容 | 件数目安 |
|------|-----------|---------|
| API → PostgreSQL | CRUD操作、トランザクション、制約検証 | 各APIに3-5件 |
| API → GitHub API | Issue作成、PR操作、Webhook処理 | 各操作に2-3件 |
| API → Redis | キャッシュ読み書き、TTL、無効化 | 5-10件 |
| Service → Service | イベント伝播、データ変換 | 各連携に3-5件 |
| Auth → API | 認証・認可チェック | 各ロールに2-3件 |

## 6. テストデータ管理

### 6.1 テストデータ戦略

| 戦略 | 説明 | 適用場面 |
|------|------|---------|
| ファクトリパターン | テストデータをプログラマティックに生成 | 標準的なテスト |
| フィクスチャ | 固定的なテストデータをファイルで管理 | 複雑なシナリオ |
| シードスクリプト | DBシードによる初期データ投入 | 一括テストデータ |
| ランダム生成 | Faker.jsによる動的データ生成 | 境界値・ファジング |

### 6.2 テストデータの独立性保証

| 手法 | 説明 |
|------|------|
| トランザクションロールバック | 各テスト終了後にDBをロールバック |
| テスト間データ分離 | 各テストが独自のデータセットを使用 |
| 一意識別子生成 | テストデータにUUIDを使用して衝突回避 |
| クリーンアップフック | afterEach / afterAll でのデータ削除 |

### 6.3 テストデータテンプレート

```
インシデントテストデータ:
  必須フィールド:
    title: "テストインシデント_{UUID}"
    description: "統合テスト用インシデント"
    priority: P1 | P2 | P3 | P4
    status: Open
    reporter: "test-user-{UUID}"

  オプションフィールド:
    assignee: "test-assignee-{UUID}"
    category: "Infrastructure" | "Application" | "Security"
    affected_ci: ["CI-{UUID}"]

変更要求テストデータ:
  必須フィールド:
    title: "テストRFC_{UUID}"
    description: "統合テスト用変更要求"
    type: Standard | Normal | Emergency
    risk_level: Low | Medium | High | Critical
    requester: "test-user-{UUID}"

  オプションフィールド:
    affected_services: ["SVC-{UUID}"]
    scheduled_date: "2026-04-01T00:00:00Z"
    rollback_plan: "テストロールバック手順"
```

## 7. エラーハンドリングテスト

### 7.1 エラーシナリオ一覧

| シナリオ | テスト内容 | 期待動作 |
|---------|-----------|---------|
| DB接続断 | PostgreSQL停止中にAPIリクエスト | 503 Service Unavailable |
| 外部API障害 | GitHub API 500レスポンス | サーキットブレーカー発動、フォールバック |
| 認証失敗 | 無効トークンでのリクエスト | 401 Unauthorized |
| 認可不足 | 権限不足のロールでのリクエスト | 403 Forbidden |
| データ不存在 | 存在しないIDでの取得 | 404 Not Found |
| バリデーションエラー | 不正なリクエストボディ | 400 Bad Request |
| レート制限超過 | 短時間での大量リクエスト | 429 Too Many Requests |
| タイムアウト | 外部APIの応答遅延 | 504 Gateway Timeout |

### 7.2 サーキットブレーカーテスト

| 状態 | テスト内容 | 期待動作 |
|------|-----------|---------|
| Closed | 正常なリクエスト | 正常応答 |
| Open | 連続失敗後のリクエスト | 即座にフォールバック応答 |
| Half-Open | 一定時間経過後のリクエスト | 試行→成功でClosed、失敗でOpen |

## 8. パフォーマンス考慮

### 8.1 統合テスト実行時間の管理

| 施策 | 説明 |
|------|------|
| コンテナ再利用 | テストスイート間でコンテナを再起動しない |
| 並列実行 | テストファイル単位での並列実行 |
| 接続プール | DBコネクションプールの適切な設定 |
| データ最小化 | テストに必要最小限のデータのみ投入 |
| 選択実行 | 変更影響のあるテストのみ実行（影響分析） |

### 8.2 実行時間目標

| テスト種別 | 単一テスト | テストファイル | 全統合テスト |
|-----------|-----------|-------------|-------------|
| API統合テスト | 500ms以下 | 10秒以下 | 3分以下 |
| DB統合テスト | 200ms以下 | 5秒以下 | 2分以下 |
| 外部連携テスト | 300ms以下 | 5秒以下 | 2分以下 |
| 合計 | - | - | 7分以下 |

## 9. CI/CD統合

### 9.1 CI パイプラインでの位置

統合テストはCI/CDパイプラインの Stage 3 に配置される。

```
Stage 1: 静的解析    (並列)
Stage 2: ユニットテスト (並列)
Stage 3: 統合テスト   (並列、コンテナ依存)  <-- ここ
Stage 4: ビルド
Stage 5: E2Eテスト
Stage 6: 品質ゲート
```

### 9.2 統合テスト用CIジョブ設定

| 項目 | 設定値 |
|------|--------|
| サービスコンテナ | PostgreSQL 16、Redis 7 |
| タイムアウト | 10分 |
| リトライ | 失敗時1回リトライ |
| 並列度 | 4ワーカー |
| キャッシュ | Docker イメージキャッシュ |
| アーティファクト | テスト結果XML、ログファイル |

## 10. 監査・コンプライアンス対応

### 10.1 監査ログ統合テスト

ServiceMatrix は J-SOX 準拠の監査ログを要求するため、
以下の統合テストを必須とする。

| テスト項目 | 検証内容 |
|-----------|---------|
| ログ記録の完全性 | すべてのCRUD操作が監査ログに記録されること |
| ログの改竄防止 | 監査ログの変更が不可能であること |
| タイムスタンプ精度 | UTC タイムスタンプが正確に記録されること |
| 操作者記録 | 操作者情報が正確に記録されること |
| ログ検索 | 監査ログの検索・フィルタリングが正確に動作すること |

### 10.2 SoD（職務分離）統合テスト

| テスト項目 | 検証内容 |
|-----------|---------|
| 申請者自己承認禁止 | 変更要求の申請者が自身を承認できないこと |
| ロール分離 | 同一ユーザーが相反するロールを保持できないこと |
| 権限境界 | 各ロールが許可された操作のみ実行できること |
