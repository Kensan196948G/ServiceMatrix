# ユニットテストポリシー

ServiceMatrix Unit Test Policy

Version: 1.0
Status: Active
Classification: Internal Quality Document
Last Updated: 2026-03-02

---

## 1. 概要

本ドキュメントは、ServiceMatrix プロジェクトにおけるユニットテストの
方針・規約・実装ガイドラインを定義する。
全開発者はコード変更時に本ポリシーに従ったユニットテストを作成する義務を持つ。

---

## 2. テストフレームワークと技術スタック

### 2.1 採用技術

| 項目 | 技術 | バージョン | 用途 |
|------|------|-----------|------|
| テストランナー | Vitest | 1.x | テスト実行・アサーション |
| カバレッジ | c8 / Istanbul | - | コードカバレッジ測定 |
| モック | vi.mock / vi.fn | - | 依存関係のモック |
| APIモック | MSW (Mock Service Worker) | 2.x | HTTP リクエストモック |
| UIテスト | Testing Library | 14.x | React コンポーネントテスト |
| スナップショット | Vitest Snapshot | - | UI スナップショット比較 |

### 2.2 選定理由

| 技術 | 選定理由 |
|------|---------|
| Vitest | Vite ベースで高速、ESM ネイティブサポート、Jest 互換API |
| c8 | V8 ネイティブカバレッジで高精度かつ高速 |
| MSW | ネットワーク層でのインターセプトによりテストの独立性が高い |
| Testing Library | ユーザー視点のテストを促進、実装詳細に依存しない |

## 3. ファイル配置規約

### 3.1 ディレクトリ構造

```
src/
  services/
    incident/
      incident.service.ts
      __tests__/
        incident.service.test.ts
  utils/
    date-formatter.ts
    __tests__/
      date-formatter.test.ts
  components/
    dashboard/
      MetricCard.tsx
      __tests__/
        MetricCard.test.tsx
```

### 3.2 命名規約

| 対象 | 命名パターン | 例 |
|------|-------------|-----|
| テストファイル | `{対象ファイル名}.test.{ts,tsx}` | `incident.service.test.ts` |
| テストディレクトリ | `__tests__/` | `src/services/__tests__/` |
| テストユーティリティ | `__tests__/helpers/` | `__tests__/helpers/mock-data.ts` |
| フィクスチャ | `__tests__/fixtures/` | `__tests__/fixtures/incidents.json` |

### 3.3 テスト構造規約

```
describe('[対象モジュール名]', () => {
  describe('[メソッド名またはシナリオ]', () => {
    it('should [期待される動作] when [条件]', () => {
      // Arrange: テストデータ準備
      // Act: 対象メソッド実行
      // Assert: 結果検証
    });
  });
});
```

## 4. カバレッジ要件

### 4.1 カバレッジ閾値

| 対象ディレクトリ | 行カバレッジ | 分岐カバレッジ | 関数カバレッジ |
|-----------------|------------|--------------|--------------|
| services/ | 80% | 75% | 85% |
| utils/ | 90% | 85% | 95% |
| components/ | 70% | 65% | 80% |
| routes/ | 75% | 70% | 80% |
| middleware/ | 80% | 75% | 85% |
| models/ | 70% | 65% | 75% |

### 4.2 カバレッジ除外対象

| 除外対象 | 理由 |
|---------|------|
| 型定義ファイル（*.d.ts） | 実行コードを含まない |
| 設定ファイル（*.config.ts） | 宣言的な設定のみ |
| マイグレーション（migrations/） | DBスキーマ変更は統合テストで検証 |
| インデックスファイル（index.ts） | re-export のみ |
| テストファイル自体 | テストのテストは不要 |
| 生成コード | 自動生成されたコード |

### 4.3 カバレッジ強制

CI/CDパイプラインにおいて、カバレッジ閾値を下回るPRはマージをブロックする。
カバレッジレポートはPRコメントとして自動投稿される。

## 5. テスト品質基準

### 5.1 良いテストの特性

| 特性 | 説明 | アンチパターン |
|------|------|--------------|
| 独立性 | 他のテストに依存しない | テスト間の実行順序依存 |
| 決定性 | 同一条件で常に同じ結果 | 日時や乱数に依存する結果 |
| 高速性 | 個々のテストは数ms以内 | 外部サービス呼び出し |
| 明確性 | テスト名から目的がわかる | `test1`, `test2` のような命名 |
| 単一責任 | 1テスト1アサーション原則 | 複数の無関係な検証の混在 |
| 保守性 | 実装変更に強い | 内部実装詳細への依存 |

### 5.2 テスト必須ケース

すべてのパブリックメソッドに対し、最低限以下のテストケースを作成する。

| ケース分類 | 説明 | 例 |
|-----------|------|-----|
| 正常系 | 期待される入力での正常動作 | 有効なインシデントデータでの作成 |
| 境界値 | 入力の境界条件 | 空文字列、最大長、0、負数 |
| 異常系 | 不正入力でのエラーハンドリング | null、undefined、不正な型 |
| エッジケース | 特殊な状況 | 同時実行、タイムアウト |

### 5.3 禁止パターン

| パターン | 問題 | 代替手段 |
|---------|------|---------|
| 本番APIへの直接呼び出し | テスト環境の汚染 | MSWによるモック |
| `setTimeout` による待機 | テスト不安定化 | `vi.useFakeTimers()` |
| `Math.random()` の直接使用 | 非決定的テスト | シード付き乱数またはモック |
| グローバル状態の変更 | テスト間の干渉 | テスト毎のセットアップ/クリーンアップ |
| 実装詳細への依存 | リファクタリング脆弱性 | 公開インタフェースのテスト |
| 過度なモック | テストの意味喪失 | 必要最小限のモック |

## 6. モック戦略

### 6.1 モック対象判断基準

| 対象 | モック要否 | 理由 |
|------|-----------|------|
| 外部APIクライアント | Yes | ネットワーク依存排除 |
| データベース層 | Yes（Repository層） | I/O依存排除 |
| 日時取得 | Yes | 決定性確保 |
| ファイルシステム | Yes | 環境依存排除 |
| ロガー | Yes | 出力ノイズ排除 |
| 同一レイヤーのモジュール | No（原則） | 過度なモックの防止 |
| 値オブジェクト | No | 軽量で副作用なし |
| ユーティリティ関数 | No（原則） | 純粋関数は直接テスト可能 |

### 6.2 モックレベル

| レベル | 説明 | 使用場面 |
|--------|------|---------|
| Spy | 呼び出し追跡のみ | コールバック検証 |
| Stub | 固定値を返却 | 依存関係の出力制御 |
| Mock | 呼び出し検証 + 返却値制御 | 契約の検証 |
| Fake | 簡易実装 | インメモリDB等 |

### 6.3 GitHub API モック標準パターン

MSW を使用して GitHub API をモックする。

```
テスト対象: GitHubService.getIssue(owner, repo, number)

モック設定:
  エンドポイント: GET /repos/{owner}/{repo}/issues/{number}
  正常レスポンス: { id: 1, title: 'Test Issue', state: 'open' }
  エラーレスポンス: 404 Not Found

検証項目:
  (1) 正しいエンドポイントが呼ばれたこと
  (2) レスポンスが正しく変換されたこと
  (3) 404時に適切な例外がスローされたこと
```

## 7. テスト実行と CI 統合

### 7.1 ローカル実行コマンド

| コマンド | 用途 |
|---------|------|
| `npm run test` | 全ユニットテスト実行 |
| `npm run test:watch` | ウォッチモードで実行 |
| `npm run test:coverage` | カバレッジ付きで実行 |
| `npm run test -- --filter incident` | 特定テストのみ実行 |

### 7.2 CI パイプライン設定

| 項目 | 設定値 |
|------|--------|
| トリガー | PR作成・更新、mainブランチプッシュ |
| 並列実行 | テストファイル単位で並列化 |
| タイムアウト | 5分 |
| カバレッジ報告 | PRコメントに自動投稿 |
| カバレッジ閾値 | 設定値未満でPRブロック |
| キャッシュ | node_modules をキャッシュ |

### 7.3 テスト結果レポート

| レポート種別 | 形式 | 出力先 |
|-------------|------|--------|
| テスト結果 | JUnit XML | CI アーティファクト |
| カバレッジサマリ | テキスト | PRコメント |
| カバレッジ詳細 | HTML | CI アーティファクト |
| カバレッジ差分 | テキスト | PRコメント |

## 8. テストレビュー基準

PRレビュー時、テストに関して以下を確認する。

| チェック項目 | 詳細 |
|-------------|------|
| テスト存在確認 | 新規・変更コードにテストが存在するか |
| カバレッジ | 閾値を満たしているか |
| テスト品質 | 正常系・異常系・境界値がカバーされているか |
| 命名 | テスト名から目的がわかるか |
| 独立性 | テスト間の依存がないか |
| モック適切性 | 過度なモックがないか |
| アサーション | 適切なアサーションが使われているか |
| クリーンアップ | リソースが適切に解放されているか |

## 9. テスト負債管理

### 9.1 テスト負債の定義

| 負債種別 | 説明 | 対応方針 |
|---------|------|---------|
| 未テストコード | テストが存在しないコード | 新機能開発時に順次追加 |
| フレーキーテスト | 非決定的なテスト | 発見次第即座に修正 |
| 低品質テスト | 実装詳細に依存したテスト | リファクタリング時に改善 |
| 遅延テスト | 実行時間が長すぎるテスト | 最適化またはレイヤー移動 |

### 9.2 テスト負債の追跡

テスト負債は GitHub Issue でラベル `test-debt` を付与して管理する。
四半期ごとにテスト負債の棚卸しを実施し、優先度に基づき解消する。

## 10. AI Agent によるテスト支援

### 10.1 AI支援の範囲

| 支援内容 | 説明 | 自律レベル |
|---------|------|-----------|
| テストケース提案 | コード変更に対するテストケース案の提示 | L1（助言） |
| カバレッジ分析 | カバレッジギャップの特定と改善提案 | L1（助言） |
| フレーキーテスト検出 | テスト結果の分析によるフレーキーテスト識別 | L2（補助） |
| テストコード生成 | テストコードの自動生成（人間レビュー必須） | L2（補助） |

### 10.2 AI支援の制約

| 制約 | 理由 |
|------|------|
| 自動マージ禁止 | AIが生成したテストは必ず人間レビューを経る |
| セキュリティテスト制限 | セキュリティテストの自動生成は L1（助言）のみ |
| テスト削除禁止 | AIによるテストケースの削除は禁止 |
| カバレッジ閾値変更禁止 | カバレッジ閾値の変更はAdmin承認が必要 |
