# テスト戦略

ServiceMatrix Test Strategy

Version: 1.0
Status: Active
Classification: Internal Quality Document
Last Updated: 2026-03-02

---

## 1. 概要

本ドキュメントは、ServiceMatrix プロジェクトにおけるテスト戦略を定義する。
テストピラミッドに基づく多層テストアプローチにより、
品質・信頼性・セキュリティを体系的に保証する。

---

## 2. テスト原則

### 2.1 基本原則

| 原則 | 説明 |
|------|------|
| Shift Left | テストを開発ライフサイクルの早期段階に移動する |
| テストピラミッド | 下層ほど多く、上層ほど少なくテストを配置する |
| 自動化優先 | 繰り返し実行されるテストは自動化する |
| 継続的テスト | CI/CDパイプラインにテストを統合する |
| リスクベース | リスクの高い領域に重点的にテストを配置する |
| 独立性 | テスト環境はテスト対象から独立させる |

### 2.2 品質目標

| 指標 | 目標値 | 測定方法 |
|------|--------|----------|
| コードカバレッジ（行） | 80%以上 | Istanbul / c8 |
| コードカバレッジ（分岐） | 75%以上 | Istanbul / c8 |
| ユニットテスト成功率 | 100% | CI/CDパイプライン |
| 統合テスト成功率 | 100% | CI/CDパイプライン |
| E2Eテスト成功率 | 98%以上 | Playwright レポート |
| 重大バグの本番到達率 | 0件/リリース | インシデント記録 |
| テスト実行時間（ユニット） | 3分以内 | CI/CDパイプライン |
| テスト実行時間（全体） | 15分以内 | CI/CDパイプライン |

## 3. テストピラミッド

### 3.1 ピラミッド構造

```
                    /\
                   /  \
                  / E2E \          少数（重要フロー）
                 / Tests \         5-10%
                /----------\
               / Integration \     中程度（API・DB連携）
              /    Tests      \    20-30%
             /----------------\
            /    Unit Tests     \   大多数（ロジック検証）
           /                    \  60-70%
          /______________________\
```

### 3.2 各層の定義

| 層 | 対象 | 割合 | 実行速度 | 信頼性 |
|----|------|------|----------|--------|
| ユニットテスト | 関数・クラス・モジュール | 60-70% | 高速（ms単位） | 高い |
| 統合テスト | API・DB・外部連携 | 20-30% | 中速（秒単位） | 中程度 |
| E2Eテスト | ユーザーフロー全体 | 5-10% | 低速（分単位） | フレーキーリスクあり |

### 3.3 補助テスト

| テスト種別 | 対象 | 頻度 |
|-----------|------|------|
| セキュリティテスト | 脆弱性・認証・認可 | リリース毎 + 定期 |
| パフォーマンステスト | 応答時間・スループット | リリース毎 |
| アクセシビリティテスト | WCAG 2.1 AA準拠 | 四半期 |
| カオステスト | 障害耐性 | 月次（Staging） |
| 回帰テスト | 既存機能の維持 | リリース毎 |

## 4. テスト環境

### 4.1 環境定義

| 環境 | 用途 | データ | インフラ |
|------|------|--------|----------|
| Local | 開発者ローカルテスト | モックデータ | Docker Compose |
| CI | 自動テスト実行 | テストフィクスチャ | GitHub Actions |
| Development | 統合開発テスト | サンプルデータ | Kubernetes (dev) |
| Staging | リリース前検証 | 本番匿名化データ | Kubernetes (staging) |
| Production | 本番運用 | 本番データ | Kubernetes (prod) |

### 4.2 環境別テスト実行マトリクス

| テスト種別 | Local | CI | Development | Staging | Production |
|-----------|-------|----|-------------|---------|------------|
| ユニットテスト | Yes | Yes | - | - | - |
| 統合テスト | 部分的 | Yes | Yes | - | - |
| E2Eテスト | - | 部分的 | Yes | Yes | - |
| セキュリティテスト | - | SAST | SAST+DAST | Full | - |
| パフォーマンステスト | - | - | 基本 | Full | - |
| スモークテスト | - | - | Yes | Yes | Yes |
| カオステスト | - | - | - | Yes | - |

### 4.3 テストデータ管理

| 項目 | 方針 |
|------|------|
| テストデータ生成 | Faker.js によるランダムデータ生成 |
| フィクスチャ管理 | リポジトリ内に JSON/YAML で管理 |
| 本番データ利用 | Staging環境のみ、匿名化必須 |
| データクリーンアップ | テスト終了後に自動クリーンアップ |
| シード管理 | マイグレーションと同期したシードファイル |
| 機密データ | テストデータにシークレット情報を含めない |

## 5. CI/CDパイプラインにおけるテスト

### 5.1 パイプラインステージ

```
Pull Request 作成
  |
  v
[Stage 1] 静的解析（並列実行）
  +-- Lint（ESLint / Markdownlint）
  +-- 型検査（TypeScript tsc）
  +-- フォーマット検査（Prettier）
  +-- セキュリティ（SAST: CodeQL / Semgrep）
  |
  v
[Stage 2] ユニットテスト
  +-- フロントエンド（Vitest）
  +-- バックエンド（Vitest）
  +-- カバレッジ収集・レポート
  |
  v
[Stage 3] 統合テスト
  +-- API テスト（Supertest）
  +-- DB テスト（Testcontainers）
  +-- 外部サービスモックテスト
  |
  v
[Stage 4] ビルド・デプロイ
  +-- Docker イメージビルド
  +-- Development 環境デプロイ
  +-- ヘルスチェック
  |
  v
[Stage 5] E2E テスト（Development環境）
  +-- 主要ユーザーフロー
  +-- クリティカルパステスト
  |
  v
[Stage 6] 品質ゲート判定
  +-- カバレッジ閾値チェック
  +-- テスト成功率チェック
  +-- セキュリティ脆弱性チェック
  |
  v
PR レビュー → 承認 → Merge
```

### 5.2 テスト実行タイミング

| トリガー | 実行テスト | 最大実行時間 |
|---------|-----------|-------------|
| コミットプッシュ | Lint + 型検査 + ユニットテスト | 5分 |
| PR作成・更新 | Stage 1〜6 全ステージ | 15分 |
| mainブランチマージ | 全テスト + Stagingデプロイ + E2E | 30分 |
| リリースタグ | 全テスト + セキュリティ + パフォーマンス | 45分 |
| 定期実行（日次） | セキュリティスキャン + 依存関係チェック | 30分 |
| 定期実行（週次） | フルリグレッション + パフォーマンスベースライン | 60分 |

### 5.3 テスト失敗時の対応

| 失敗ステージ | 対応 | ブロック対象 |
|-------------|------|------------|
| 静的解析 | 開発者がローカルで修正 | PR マージ |
| ユニットテスト | 開発者がテスト修正 | PR マージ |
| 統合テスト | チームで原因調査 | PR マージ |
| E2Eテスト | フレーキーテスト判定後、必要に応じ修正 | PR マージ（フレーキー除外可） |
| セキュリティ | Critical/High は即時対応 | PR マージ + リリース |
| パフォーマンス | ベースライン比較で判定 | リリース |

## 6. テスト種別詳細

### 6.1 ユニットテスト

| 項目 | 仕様 |
|------|------|
| フレームワーク | Vitest |
| 対象 | ビジネスロジック、ユーティリティ関数、状態管理 |
| モック | vi.mock()、MSW（API モック） |
| カバレッジ | c8 / Istanbul |
| ファイル配置 | `__tests__/` ディレクトリまたは `.test.ts` 接尾辞 |
| 命名規則 | `describe('対象')` → `it('should 期待動作')` |

#### カバレッジ対象と除外

| 対象 | カバレッジ必須 |
|------|--------------|
| ビジネスロジック（services/） | Yes（80%以上） |
| API ハンドラ（routes/） | Yes（75%以上） |
| ユーティリティ（utils/） | Yes（90%以上） |
| 型定義（types/） | No |
| 設定ファイル（config/） | No |
| マイグレーション（migrations/） | No |

### 6.2 統合テスト

| 項目 | 仕様 |
|------|------|
| フレームワーク | Vitest + Supertest |
| データベース | Testcontainers（PostgreSQL） |
| 外部サービス | MSW（Mock Service Worker） |
| 対象 | API エンドポイント、DB操作、外部連携 |
| セットアップ | beforeAll でコンテナ起動、afterAll で破棄 |
| データ | テスト毎にトランザクションロールバック |

### 6.3 E2Eテスト

| 項目 | 仕様 |
|------|------|
| フレームワーク | Playwright |
| ブラウザ | Chromium、Firefox、WebKit |
| 対象 | クリティカルユーザーフロー |
| データ | API経由でテストデータ作成 |
| 認証 | テスト用 OAuth トークン |
| スクリーンショット | 失敗時に自動キャプチャ |
| リトライ | フレーキー対策として最大2回リトライ |

#### E2Eテスト対象フロー

| # | フロー | 優先度 |
|---|--------|--------|
| 1 | ログイン → ダッシュボード表示 | P1 |
| 2 | インシデント作成 → 割り当て → 解決 | P1 |
| 3 | 変更要求作成 → 承認 → 実施 | P1 |
| 4 | SLA 閾値超過 → アラート通知 | P1 |
| 5 | CMDB CI 登録 → 依存関係設定 | P2 |
| 6 | AI 提案表示 → 人間承認 → 自動実行 | P1 |
| 7 | 監査ログ検索 → エクスポート | P2 |
| 8 | ロール変更 → 権限反映確認 | P2 |
| 9 | リリース作成 → デプロイ → 検証 | P2 |
| 10 | レポート生成 → PDF出力 | P3 |

## 7. テスト自動化戦略

### 7.1 自動化対象の判断基準

| 基準 | 自動化推奨 | 手動推奨 |
|------|-----------|---------|
| 実行頻度 | 頻繁（毎PR） | 稀（年1回） |
| 安定性 | 安定した手順 | 探索的テスト |
| 再現性 | 高い | 低い |
| コスト | テスト作成コスト < 手動実行コスト×回数 | 逆の場合 |
| リスク | 高リスク領域 | 低リスク領域 |

### 7.2 自動化ロードマップ

| フェーズ | 期間 | 対象 | 目標カバレッジ |
|---------|------|------|--------------|
| Phase 1 | Month 1-2 | ユニットテスト基盤 | 60% |
| Phase 2 | Month 3-4 | 統合テスト + CI統合 | 70% |
| Phase 3 | Month 5-6 | E2E テスト + セキュリティ | 80% |
| Phase 4 | Month 7以降 | パフォーマンス + カオス | 80%維持 |

## 8. 非機能テスト

### 8.1 パフォーマンステスト

| 項目 | 基準 |
|------|------|
| API応答時間（P95） | 200ms以下 |
| ページロード時間（P95） | 3秒以下 |
| 同時接続ユーザー | 100ユーザー |
| スループット | 1000 req/sec |
| データベースクエリ | 50ms以下 |

### 8.2 セキュリティテスト

| テスト種別 | ツール | 頻度 |
|-----------|--------|------|
| SAST | CodeQL / Semgrep | 毎PR |
| DAST | OWASP ZAP | Staging デプロイ毎 |
| SCA | Dependabot / Snyk | 日次 |
| ペネトレーション | 外部ベンダー | 年次 |
| シークレットスキャン | GitGuardian / TruffleHog | 毎コミット |

### 8.3 可用性テスト

| テスト | 内容 | 頻度 |
|--------|------|------|
| ヘルスチェック | エンドポイント応答確認 | 常時（30秒間隔） |
| フェイルオーバー | レプリカ切り替え確認 | 月次 |
| バックアップ復元 | バックアップからの復元テスト | 四半期 |
| 災害復旧 | DR手順の実施テスト | 半年 |

## 9. テストレポーティング

### 9.1 レポート種別

| レポート | 内容 | 頻度 | 配信先 |
|---------|------|------|--------|
| CI テストレポート | テスト結果・カバレッジ | 毎PR | PR コメント |
| 品質ダッシュボード | カバレッジ推移・テスト傾向 | リアルタイム | WebUI |
| リリーステストレポート | リリース前テスト総合結果 | リリース毎 | 関係者全員 |
| セキュリティレポート | 脆弱性検出結果 | 週次 | セキュリティチーム |
| パフォーマンスレポート | ベースライン比較結果 | リリース毎 | 開発チーム |

### 9.2 メトリクス追跡

| メトリクス | 追跡方法 | アラート条件 |
|-----------|---------|-------------|
| カバレッジ率 | CI/CDパイプライン | 目標値を下回った場合 |
| テスト実行時間 | CI/CDパイプライン | 前回比20%以上増加 |
| フレーキーテスト率 | テスト結果分析 | 5%以上 |
| バグ検出率 | Issue トラッカー | テスト漏れバグ発生時 |
| テスト負債 | コードレビュー | 未テストコードの累積 |

## 10. 役割と責任

| 役割 | 責任 |
|------|------|
| 開発者 | ユニットテスト作成、統合テスト作成、テスト修正 |
| QAエンジニア | E2Eテスト設計、テスト戦略レビュー |
| セキュリティエンジニア | セキュリティテスト設計・実行 |
| DevOps | テストインフラ構築・維持、CI/CDパイプライン管理 |
| テックリード | テスト方針決定、品質ゲート基準設定 |
| AI Agent | テストケース提案、カバレッジ分析、フレーキーテスト検出 |
