## 1. 概要

本ドキュメントは、ServiceMatrix プロジェクトにおける未解決の課題・
保留中の判断事項・今後検討が必要な項目を管理する。

本リストから削除する。

## 2. オープン課題（優先度順）

### 2.1 アーキテクチャ・設計課題

#### OQ-001: マイクロサービス分割の粒度

| 項目 | 内容 |
|------|------|
| 優先度 | P1 |
| 状態 | 未解決 |
| 担当 | テックリード |
| 期限 | 2026-03-31 |
| 関連 Issue | - |

**課題内容:**
各 ITSM プロセス（インシデント・変更・問題・要求）を個別マイクロサービスとして分割するか、
モジュラーモノリスとして構築するかの判断が保留されている。

**検討ポイント:**
- 初期段階でのマイクロサービス化のオーバーヘッドは許容できるか？
- チームサイズとサービス数のバランスはどうするか？
- サービス間の通信方式（同期 REST vs 非同期イベント）は何を優先するか？

**影響範囲:**
- アーキテクチャ全体
- デプロイ戦略
- 開発チーム体制

#### OQ-002: リアルタイム通信のアーキテクチャ

| 項目 | 内容 |
|------|------|
| 優先度 | P1 |
| 状態 | 未解決 |
| 担当 | バックエンドリード |
| 期限 | 2026-04-15 |
| 関連 Issue | - |

**課題内容:**
ダッシュボードのリアルタイム更新とインシデント通知の実現方式について決定が必要。

**検討ポイント:**

**影響範囲:**
- バックエンドアーキテクチャ
- インフラ設計
- SLA（リアルタイム通知の遅延基準）

### 2.2 AI ガバナンス課題

#### OQ-003: AI 自律度向上のマイルストーン設定

| 項目 | 内容 |
|------|------|
| 優先度 | P2 |
| 状態 | 未解決 |
| 担当 | CTO + CISO |
| 期限 | 2026-06-30 |
| 関連 Issue | - |

**課題内容:**
現在の L1（助言のみ）から L2（半自動）への移行タイミングと条件の設定が必要。

**検討ポイント:**
- L2 への移行を承認するための定量的基準は何か？
  （例: AI 提案の人間承認率が X%以上を N ヶ月継続 等）
- L2 で許可する自律操作の範囲（スケール調整のみ？設定変更も含む？）
- AI 判断の説明可能性をどのように評価・測定するか？

**影響範囲:**
- AI ガバナンスポリシー
- J-SOX IT 全般統制の適用範囲
- 運用体制（オンコール要員数等）

### 2.3 セキュリティ・コンプライアンス課題

#### OQ-005: 個人情報（PII）の取り扱い方針

| 項目 | 内容 |
|------|------|
| 優先度 | P1 |
| 状態 | 未解決 |
| 担当 | CISO + 法務 |
| 期限 | 2026-04-30 |
| 関連 Issue | - |

**課題内容:**
インシデント管理においてユーザー情報（メールアドレス、氏名等）を扱う場合の
個人情報保護法（APPI）対応方針が未決定。

**検討ポイント:**
- 監査ログに個人情報を含める場合の保存・アクセス制限
- データ主体の削除要求（忘れられる権利）への対応方法
- 開発・テスト環境での個人情報の取り扱い（匿名化の実装方法）
- 個人情報をログに出力しないためのコーディング規約

**影響範囲:**
- データモデル
- 監査証跡設計
- テストデータ管理

### 2.4 運用・インフラ課題

#### OQ-007: データ保持・アーカイブ戦略

| 項目 | 内容 |
|------|------|
| 優先度 | P2 |
| 状態 | 未解決 |
| 担当 | DevOps リード |
| 期限 | 2026-05-31 |
| 関連 Issue | - |

**課題内容:**
インシデント・変更・監査ログ等の長期保存戦略が未定義。
J-SOX 要件（7 年保存）に対応しながらデータベースの肥大化を防ぐ方法が必要。

**検討ポイント:**

- アーカイブデータの検索可能性の要件（監査時の参照頻度）
- PostgreSQL パーティショニングの採用可否

**影響範囲:**
- データベース設計
- インフラコスト
- 監査証跡収集手順

## 4. 課題管理規則

### 4.1 課題の追加方法

新たなオープン課題は以下の手順で追加する。

1. GitHub Issue を作成し、ラベル `open-question` を付与
2. 本ドキュメントに課題を記載（PR 経由）
3. 担当者と期限を設定
4. 四半期レビューで優先度を見直す

### 4.2 課題のクローズ方法

課題が解決された場合の手順。

1. 決定内容を DECISION_LOG.md に ADR として記録
2. 解決済み課題として本ドキュメントの参考テーブルに移動
3. 関連する設計ドキュメントを更新
4. GitHub Issue をクローズ
