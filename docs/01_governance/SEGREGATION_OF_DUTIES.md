Status: Active

## 1. 文書の目的

本文書は、ServiceMatrix における職務分掌（SoD: Segregation of Duties）の
原則・禁止役割組み合わせ・GitHub アクセス制御・AI 統治における SoD・
例外処理と補完的コントロールを定義する。

職務分掌は以下の目的で実施する：

- 不正行為・不正変更の防止
- エラーの相互検出
- J-SOX における IT 全般統制（ITGC）の要件充足
- ISO/IEC 20000 アクセス管理要件への適合
- AI 統治における人間の最終決定権の確保

---

## 2. 職務分掌の基本原則

### 2.1 SoD 四原則

```
原則 1: 申請・実施・承認の分離
  ┌──────────────────────────────────────────────┐
  │  一人の人物が「申請」「実施」「承認」を同時に  │
  │  行うことを禁止する。                          │
  │                                               │
  │  申請者 ≠ 承認者 ≠ 監査者                    │
  └──────────────────────────────────────────────┘

原則 2: 変更・レビュー・承認の分離
  ┌──────────────────────────────────────────────┐
  │  コードまたは設定を「変更した者」が           │
  │  その変更を「承認・マージ」することを禁止。    │
  │                                               │
  │  作成者 ≠ 承認者                             │
  └──────────────────────────────────────────────┘

原則 3: 実施・検証の分離
  ┌──────────────────────────────────────────────┐
  │  変更を「実施した者」が「検証（PIR）」を      │
  │  単独で完結することを禁止。                    │
  │                                               │
  │  実施者 ≠ 単独検証者                         │
  └──────────────────────────────────────────────┘

原則 4: アクセス権付与・使用の分離
  ┌──────────────────────────────────────────────┐
  │  アクセス権を「付与する者」と「使用する者」は  │
  │  同一であってはならない。                      │
  │                                               │
  │  権限付与者 ≠ 権限使用者（原則）             │
  └──────────────────────────────────────────────┘
```

### 2.2 SoD のスコープ

SoD は以下のプロセス領域に適用される：

| プロセス領域 | SoD 適用範囲 |
|------------|------------|
| 変更管理 | RFC 起票 / リスク評価 / 承認 / 実施 / 検証 |
| インシデント管理 | 起票 / 対応 / 解決 / クローズ / RCA |
| アクセス管理 | 申請 / 承認 / 付与 / レビュー / 剥奪 |
| AI 統治 | 提案 / リスク評価 / 承認 / 実施 / 監視 |
| 監査 | 被監査 / 監査実施 / 結果承認 |
| リリース管理 | 開発 / テスト / 本番デプロイ承認 |

## 3. 禁止役割組み合わせ（Forbidden Role Combinations）

### 3.1 絶対禁止（Critical SoD Violations）

以下の役割組み合わせは、いかなる状況においても同一人物が担うことを禁止する。
例外処理の対象外とする。

| # | 役割 A | 役割 B | 禁止理由 |
|---|--------|--------|---------|
| F1 | RFC 起票者 | 同 RFC の承認者 | 自己承認防止 |
| F2 | コード作成者 | 同 PR の単独マージ承認者 | コードレビュー回避防止 |
| F3 | システム管理者（SysA） | 内部監査人 | 監査独立性確保 |
| F4 | アクセス権申請者 | アクセス権付与者（自己付与） | 権限昇格防止 |
| F5 | AI エージェント | 変更の最終承認者 | AI 自己承認禁止 |
| F6 | 変更実施者 | 同変更の CAB 承認者（利益相反） | 独立審査確保 |
| F7 | インシデント対応者 | 同インシデントの RCA 単独実施者 | 客観性確保 |

### 3.2 条件付き禁止（Contextual SoD Controls）

以下の組み合わせは、原則禁止とするが、補完的コントロールが存在する場合に限り例外を認める。

| # | 役割 A | 役割 B | 補完的コントロール要件 |
|---|--------|--------|---------------------|
| C1 | Process Owner | 同プロセスの変更承認者 | 上位レベルの独立レビュー必須 |
| C2 | Change Manager | Emergency Change 実施者 | SM の事前承認 + 事後 CAB レビュー必須 |
| C3 | Security Officer | セキュリティ変更の実施者 | CM 承認 + 独立セキュリティレビュー必須 |
| C4 | AI Governance Officer | AI 設定の変更申請者 | SM + EGB 承認必須 |
| C5 | Service Manager | CAB メンバー | 自己関連変更の審査から回避必須（conflict of interest 宣言） |

## 4. GitHub アクセス制御との統合

### 4.1 GitHub ロールと SoD の対応

GitHub のアクセスロールは、SoD 原則に基づいて割り当てる。

```
GitHub ロール × ServiceMatrix ロール マッピング:

  Owner（最高権限）:
    → EGB メンバーのみ
    → リポジトリ設定・ブランチ保護ルール変更が可能
    → SoD: Owner は自身の PR を直接マージ禁止

  Admin:
    → Service Manager、Security Officer
    → CODEOWNERS・Secrets 管理が可能
    → SoD: Admin が申請した変更は別の Admin または Owner が承認

  Maintainer:
    → Change Manager、Technical Lead
    → PR マージ、ラベル管理が可能
    → SoD: 自身が作成した PR のマージ禁止

  Write:
    → Developer、Operations、Process Owner
    → ブランチ作成・PR 作成が可能
    → SoD: 自身の PR の単独承認禁止

  Read:
    → End User、外部レビュアー
    → 読み取りのみ
    → SoD: 制約なし（書き込み権限なし）
```

### 4.2 保護ブランチ（Protected Branch）と SoD

```
main ブランチ保護ルールの SoD 実装:

  ┌─────────────────────────────────────────────────────┐
  │  main ブランチ                                       │
  │                                                     │
  │  保護設定（SoD 実装）:                               │
  │  1. PR レビュー必須（最低2名）                        │
  │     → 作成者を除く2名の承認が必要                    │
  │     → 作成者が承認に含まれる場合は無効               │
  │                                                     │
  │  2. Dismiss stale reviews: ON                       │
  │     → 新しいコミット後は再承認が必要                  │
  │     → 既存承認の再利用禁止                           │
  │                                                     │
  │  3. Require review from CODEOWNERS: ON              │
  │     → 影響ファイルの CODEOWNERS による承認必須         │
  │     → CODEOWNERS 自身の変更は別メンバーが承認          │
  │                                                     │
  │  4. Require signed commits: ON                      │
  │     → GPG 署名必須（実施者の証跡）                    │
  │                                                     │
  │  5. No self-merge: 設定により強制                    │
  │     → PR 作成者による自己マージ禁止                   │
  └─────────────────────────────────────────────────────┘
```

### 4.3 CODEOWNERS による SoD 実装

CODEOWNERS ファイルは SoD の技術的強制メカニズムとして機能する。

```
# ファイル: .github/CODEOWNERS
# SoD 実装のための CODEOWNERS 設定

# 変更管理文書: CM と SM の分離
/docs/01_governance/ @service-manager @change-manager
# SM と CM は互いの変更を承認することで SoD を実現

# セキュリティ文書: SEC と CM で相互チェック
/docs/06_security_compliance/ @security-officer @change-manager
# SEC が作成した変更は CM が承認、CM が作成した変更は SEC が承認

# AI 統治設定: AIG と SM による分離
/docs/04_agents_ai/ @ai-governance @service-manager
/.claude/ @ai-governance @service-manager
# どちらか一方だけでは承認できない

# GitHub 設定: SEC と TL による相互チェック
/.github/workflows/ @security-officer @technical-lead
/.github/CODEOWNERS @service-manager @change-manager
# CODEOWNERS 自体の変更は SM と CM が承認

# インフラ: TL と SEC による相互チェック
/infrastructure/ @technical-lead @security-officer
```

### 4.4 GitHub Actions による SoD 自動チェック

CI パイプラインで SoD 違反を自動検出する。

```yaml
# .github/workflows/sod-check.yml の概要

name: SoD Check
on: [pull_request]

jobs:
  check-sod:
    steps:
      # 1. PR 作成者が承認者でないことを確認
      - name: Check self-approval
        run: |
          PR_AUTHOR="${{ github.event.pull_request.user.login }}"
          APPROVERS=$(gh pr view ${{ github.event.number }} --json reviews)
          # PR 作成者が承認者に含まれる場合は失敗

      # 2. 高リスク変更の複数承認を確認
      - name: Check required approvals
        run: |
          RISK_LABEL=$(gh pr view ${{ github.event.number }} --json labels)
          # risk:high または risk:critical の場合、3名以上の承認を確認

      # 3. CODEOWNERS ルールの確認
      - name: Verify CODEOWNERS compliance
        run: |
          # 変更ファイルの CODEOWNERS が全員承認済みか確認

      # 4. AI 変更の人間承認確認
      - name: Check AI change approval
        run: |
          # ai-generated ラベルがある場合、人間の承認を確認
```

## 5. AI 統治における職務分掌

### 5.1 AI エージェントの SoD 制約

ClaudeCode（AI エージェント）に適用される職務分掌は、通常の人間役割に加えて
以下の特別な制約を持つ。

```
AI エージェントの SoD 原則:

  ┌──────────────────────────────────────────────────────┐
  │  AI は「提案」「評価」「補助」の役割のみ担える        │
  │  AI は「最終承認」「権限行使」の役割を担えない        │
  └──────────────────────────────────────────────────────┘

  具体的な制約:

  ✅ AI が担える役割:
    - 変更の草案作成
    - リスクスコアの算出
    - コードレビューコメントの作成
    - テスト結果の分析
    - PIR 草案の作成
    - 類似インシデントの検索

  ❌ AI が担えない役割:
    - 変更の最終承認（マージ）
    - アクセス権の付与
    - SoD 制約のオーバーライド
    - AI 自律性レベルの変更
    - 自己の変更の承認
    - 監査結果の確定
```

### 5.2 AI 変更提案フローにおける SoD

```
AI 変更提案の SoD フロー:

  AI（ClaudeCode）
    │
    ├─→ 変更提案を Issue/PR として作成（提案者ロール）
    │
  人間（CM / TL / CAB）
    │
    ├─→ AI 提案を評価・承認または却下（承認者ロール）
    │   ※ AI が提案した変更を AI が承認することは禁止
    │
  AI または人間
    │
    ├─→ 承認後の実装作業（実施者ロール）
    │   ※ AI は実装補助可（低リスクのみ）
    │
  人間（独立した担当者）
    │
    └─→ 変更の検証・PIR（検証者ロール）
        ※ AI は検証補助可（草案作成等）
        ※ 検証の最終確認は人間が実施
```

### 5.3 AI 自律性レベルの SoD 制約

| AI 自律性レベル | 変更提案 | リスク評価 | 実施 | 承認 | 検証 |
|----------------|---------|----------|------|------|------|
| L0（完全手動） | 禁止 | 禁止 | 禁止 | 禁止 | 禁止 |
| L1（補助） | ○ | ○ | 禁止 | 禁止 | 補助のみ |
| L2（推奨） | ○ | ○ | 禁止 | 禁止 | 補助のみ |
| L3（一部自動） | ○ | ○ | L0 変更のみ | 禁止 | 補助のみ |
| L4（自律）禁止 | - | - | - | - | - |

※ 「補助のみ」= 草案作成・情報収集は可、最終確認は人間が実施

## 6. 例外処理と補完的コントロール

### 6.1 例外処理の原則

SoD 制約の例外は、以下の条件をすべて満たす場合に限り認められる。

```
SoD 例外の許可条件:

  必須条件（全て満たすこと）:
  1. 業務上の合理的な理由が存在する
  2. リスク評価が実施されている
  3. 補完的コントロールが実装されている
  4. EGB または SM の承認を得ている
  5. 例外内容が文書化・記録されている
  6. 例外期間が明示されている（原則3ヶ月以内）
  7. 定期的な見直しがスケジュールされている

  絶対禁止（例外なし）:
  - F1〜F7 の絶対禁止組み合わせ（Section 3.1）は例外不可
  - AI による最終承認は例外不可
  - main ブランチへの直接プッシュは例外不可
```

### 6.2 補完的コントロール（Compensating Controls）

SoD 制約の条件付き例外を認める際に、必ず実装する補完的コントロール。

| SoD 制約 | 補完的コントロール | 監視方法 |
|---------|----------------|---------|
| C1: PO が同プロセス変更承認 | 上位レベルの独立レビュー + 事後監査 | 月次 SM レビュー |
| C2: CM が Emergency Change 実施 | SM 事前承認 + 事後 CAB レビュー + 全記録保存 | CAB 事後レビュー |
| C3: SEC がセキュリティ変更実施 | CM 承認 + 独立 SEC レビュー（外部または代理） | 四半期監査 |
| C4: AIG が AI 設定申請 | SM + EGB 承認 + 全変更の詳細ログ | 月次 AIG レビュー |
| C5: SM が CAB 審査参加 | conflict of interest 宣言 + 審査除外 | CAB 議長確認 |

### 6.3 例外申請フロー

```
SoD 例外申請フロー:

  1. 例外申請 Issue 作成
     ├── 対象 SoD 制約の番号
     ├── 業務上の理由
     ├── リスク評価
     ├── 提案する補完的コントロール
     └── 希望する例外期間

  2. レビュー
     ├── SM による初期評価
     ├── Sec による リスク評価
     └── 必要に応じて EGB レビュー

  3. 承認
     ├── 低リスク例外: SM 承認
     ├── 中リスク例外: SM + Sec 承認
     └── 高リスク例外: EGB 承認

  4. 実装
     ├── 補完的コントロールの実装
     ├── 例外レジスタへの登録
     └── 期限と見直し日程の設定

  5. 定期見直し
     └── 期限到来時: 延長または終了の判断
```

### 6.4 例外レジスタ

SoD 例外はすべて例外レジスタに記録・管理する。

| 項目 | 内容 |
|------|------|
| 例外 ID | SoD-EXC-YYYYMM-NNN 形式 |
| 対象制約 | 対応する SoD 制約番号 |
| 申請者 | 例外申請者 |
| 承認者 | 例外承認者 |
| 業務理由 | 例外を必要とする理由 |
| 補完的コントロール | 実装したコントロール内容 |
| 有効期間 | 開始日〜終了日 |
| 見直し日 | 次回レビュー日 |
| 現在ステータス | 有効 / 期限切れ / 終了 |
| 記録 Issue | GitHub Issue URL |

## 7. SoD の監視と違反対応

### 7.1 SoD 監視の仕組み

```
SoD 監視アーキテクチャ:

  自動監視（リアルタイム）:
  ├── GitHub Actions: PR 作成時に SoD チェック
  ├── Branch Protection: 自己マージの技術的ブロック
  └── CODEOWNERS: 必要な承認者の強制

  定期監視（日次・週次・月次）:
  ├── GitHub Audit Log の定期レビュー
  ├── 権限レジスタと実態の照合
  └── SoD 違反レポートの作成

  人間レビュー（月次・四半期）:
  ├── SM による例外レジスタのレビュー
  ├── Sec による権限マトリクスのレビュー
  └── 監査人による独立 SoD 評価
```

### 7.2 SoD 違反の検出方法

| 検出方法 | 対象 | 担当 | 頻度 |
|---------|------|------|------|
| CI/CD 自動チェック | 自己マージ・承認不足 | 自動 | 全 PR |
| GitHub Audit Log 分析 | 権限変更・直接プッシュ | Sec | 日次 |
| PR レビュー分析 | 承認者の SoD 適合性 | CM | 週次 |
| 権限レジスタ照合 | 権限の実態との乖離 | SM | 月次 |
| 独立監査 | 全 SoD 制約 | 監査人 | 四半期 |

### 7.3 SoD 違反の分類と対応

| 違反レベル | 定義 | 対応アクション | 対応期限 |
|-----------|------|-------------|---------|
| Critical | F1〜F7 の絶対禁止違反 | 即座に変更を無効化 + EGB 報告 | 2時間以内 |
| High | 補完的コントロールなしの条件付き禁止違反 | SM に即時通知 + CAB レビュー | 24時間以内 |
| Medium | 軽微な SoD 制約違反 | CM に通知 + 是正計画 | 5営業日以内 |
| Low | プロセス上の軽微な逸脱 | 記録 + 次回レビューで対応 | 30日以内 |

### 7.4 SoD 違反対応フロー

```
SoD 違反検出時の対応フロー:

  検出
    │
    ├─→ Critical: 即座に変更を無効化（PR クローズ / マージ取り消し要求）
    │             EGB + SM + Sec に即時通知
    │             インシデントとして管理
    │
    ├─→ High: SM に即時通知
    │         影響を受けた変更の一時停止
    │         CAB による緊急レビュー
    │
    ├─→ Medium: CM に通知
    │           是正計画の作成
    │           原因分析の実施
    │
    └─→ Low: 記録のみ
              次回定期レビューで評価

  共通アクション（全レベル）:
    - GitHub Issue として記録
    - 根本原因分析（RCA）
    - 再発防止策の策定
    - SoD フレームワークへのフィードバック
```

## 8. SoD KPI と報告

### 8.1 SoD KPI

| KPI | 目標値 | 測定頻度 | 報告先 |
|-----|--------|---------|--------|
| Critical SoD 違反件数 | 0件 | 月次 | EGB |
| High SoD 違反件数 | 0件 | 月次 | SM + CAB |
| Medium SoD 違反件数 | ≤ 2件/月 | 月次 | CM |
| SoD 例外数（有効中） | ≤ 5件 | 月次 | SM |
| SoD 例外の期限超過率 | 0% | 月次 | SM |
| AI SoD 違反件数 | 0件 | 月次 | AIG + SM |
| SoD CI チェック通過率 | ≥ 99% | 週次 | CM |

### 8.2 SoD 報告サイクル

```
月次 SoD レポート（SM 作成 → EGB 報告）:
  - 違反件数サマリー
  - 例外レジスタ状況
  - AI SoD コンプライアンス状況
  - 改善事項

四半期 SoD 評価（Sec 主導）:
  - 権限マトリクスの更新確認
  - CODEOWNERS の最新性確認
  - SoD チェックの自動化カバレッジ評価
  - 次四半期の改善計画

年次 SoD 監査（外部監査人）:
  - J-SOX ITGC 観点の評価
  - ISO 20000 アクセス管理要件の評価
  - 例外レジスタの妥当性評価
  - 次年度推奨事項
```

## 9. 役割別 SoD チェックリスト

### 9.1 Change Manager（CM）チェックリスト

```
CM が変更を承認する際の SoD チェック:
  □ 自分が申請した変更を承認していないか
  □ 承認者として SoD 制約に違反していないか
  □ CAB が必要な変更で CAB 承認を省略していないか
  □ Emergency Change の場合、SM 承認を取得しているか
  □ AI 提案変更に対して人間承認が適切か
```

### 9.2 Security Officer（Sec）チェックリスト

```
SEC がセキュリティ変更をレビューする際の SoD チェック:
  □ 自分が実施した変更を自己承認していないか
  □ CODEOWNERS の SEC 要件を満たしているか
  □ セキュリティ設定変更に CM 承認が存在するか
  □ AI によるセキュリティ設定変更に人間承認があるか
```

### 9.3 AI Governance Officer（AIG）チェックリスト

```
AIG が AI 関連変更をレビューする際の SoD チェック:
  □ AI が提案した変更に対して AI が承認していないか
  □ AI 自律性レベル変更に EGB 承認があるか
  □ AI 設定変更に SM 承認があるか
  □ AI の変更実施ログが記録されているか
  □ AI による権限昇格が発生していないか
```
