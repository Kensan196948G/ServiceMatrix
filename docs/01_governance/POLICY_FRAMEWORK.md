# ServiceMatrix ポリシーフレームワーク（Policy Framework）

Version: 1.0
Status: Active
Owner: Service Governance Authority
Classification: Governance Design Document
Last Updated: 2026-03-02

---

## 1. 文書の目的

本文書は、ServiceMatrix における統治ポリシーの体系・構造・管理方法を定義する。
個々のポリシーの上位構造を提供し、ポリシー間の整合性と
ポリシーライフサイクルの管理を規定する。

---

## 2. ポリシー体系の全体構造

ServiceMatrix のポリシーは、以下の5層で構成される。

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 1: 憲章層（Charter Layer）                            │
│  ServiceMatrix の根本原則・使命・ビジョン・不可侵原則          │
│  変更権限: Executive Governance Board のみ                   │
│  文書例: SERVICEMATRIX_CHARTER.md, VISION_AND_MISSION.md    │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: ポリシー層（Policy Layer）                         │
│  統治の具体的な方針・目標・制約の定義                          │
│  変更権限: Executive Governance Board の承認                 │
│  文書例: GOVERNANCE_MODEL.md, AI_GOVERNANCE_POLICY.md       │
├─────────────────────────────────────────────────────────────┤
│  Layer 3: 基準層（Standard Layer）                           │
│  ポリシーを実現するための測定可能な基準・仕様                   │
│  変更権限: Management Layer の承認                           │
│  文書例: SLA_DEFINITION.md, RACI_MATRIX.md                  │
├─────────────────────────────────────────────────────────────┤
│  Layer 4: 手順層（Procedure Layer）                          │
│  基準を実現するための具体的なステップ・作業手順                  │
│  変更権限: プロセスオーナーの承認                             │
│  文書例: INCIDENT_MANAGEMENT_MODEL.md, 運用手順書            │
├─────────────────────────────────────────────────────────────┤
│  Layer 5: ガイドライン層（Guideline Layer）                   │
│  推奨する実施方法・ベストプラクティス（強制ではない）             │
│  変更権限: 担当チームリードの判断                             │
│  文書例: コーディング規約, コミットメッセージガイド              │
└─────────────────────────────────────────────────────────────┘
```

**階層の原則**:
- 下位層は上位層に矛盾してはならない
- 上位層が定めた制約の範囲内で下位層を設計する
- 不可侵原則（Non-Negotiables）は Layer 1 に位置し、例外を認めない

### 2.1 Layer 1: 憲章層（Charter Layer）

**目的**: ServiceMatrix の根本的な存在意義と統治原則を定義する。いかなる状況でも変更困難な「憲法」に相当する。

**特徴**:
- 最も安定性が高く、変更頻度が最も低い
- 不可侵原則（Non-Negotiables）を含む
- 変更には Executive Governance Board の全会一致が必要

**対象文書**: SERVICEMATRIX_CHARTER.md, VISION_AND_MISSION.md, STRATEGIC_OBJECTIVES.md

### 2.2 Layer 2: ポリシー層（Policy Layer）

**目的**: 統治の具体的な方針・目標・制約を定義する。何を達成すべきか（"what"）を規定する。

**特徴**:
- 年次〜2年ごとに見直し
- 各プロセス・システムに適用される方針を記述
- 例外処理の基準を定める

**対象文書**: GOVERNANCE_MODEL.md, POLICY_FRAMEWORK.md, AI_GOVERNANCE_POLICY.md, SEGREGATION_OF_DUTIES.md, CHANGE_AUTHORITY_STRUCTURE.md

### 2.3 Layer 3: 基準層（Standard Layer）

**目的**: ポリシーを実現するための測定可能な基準・仕様を定義する。どのレベルで達成すべきか（"how much"）を規定する。

**特徴**:
- 定量的な指標・閾値を含む
- 半年〜年次で見直し
- KPI・SLA 目標値などが含まれる

**対象文書**: SLA_DEFINITION.md, OLA_DEFINITION.md, RACI_MATRIX.md, STATE_TRANSITION_MODEL.md, APPROVAL_CONTROL_MODEL.md

### 2.4 Layer 4: 手順層（Procedure Layer）

**目的**: 基準を実現するための具体的なステップ・作業手順を定義する。どのように達成するか（"how"）を規定する。

**特徴**:
- 実務担当者が直接参照する
- 変更頻度が最も高い層
- 役割・入力・出力・成功基準を明記

**対象文書**: INCIDENT_MANAGEMENT_MODEL.md, CHANGE_MANAGEMENT_MODEL.md, PROBLEM_MANAGEMENT_MODEL.md, REQUEST_MANAGEMENT_MODEL.md

### 2.5 Layer 5: ガイドライン層（Guideline Layer）

**目的**: 推奨する実施方法・ベストプラクティスを提供する。遵守は任意だが、準拠することが望ましい。

**特徴**:
- 強制力はないが推奨される
- チーム・状況に応じてカスタマイズ可能
- 随時更新可能

**対象文書**: コーディング規約, コミットメッセージガイドライン, ブランチ命名規則

## 5. ポリシーライフサイクル管理

### 5.1 ライフサイクルの概要

```
Draft → Review → Approved → Active → Under Review → Revised/Retired
```

### 5.2 各フェーズの詳細

| フェーズ | 説明 | 責任者 |
|---------|------|--------|
| Draft | 草案作成 | ポリシーオーナー |
| Review | ステークホルダーレビュー | レビュー対象者 |
| Approved | 承認権限者による承認 | 承認権限者 |
| Active | 施行中 | ポリシーオーナー |
| Under Review | 定期レビュー中 | ポリシーオーナー |
| Revised | 改定版が Active に | ポリシーオーナー |
| Retired | 廃止 | 承認権限者 |

### 5.3 ポリシー作成手順

1. ポリシーの必要性を特定する
2. ポリシーオーナーを任命する
3. 影響範囲を分析する
4. 草案を作成する（標準構成に従う）
5. ステークホルダーレビューを実施する
6. フィードバックを反映する
7. 承認を取得する（ポリシー層に応じた承認権限）
8. 施行開始を周知する
9. 技術的強制を実装する（該当する場合）
10. Git にコミットし、バージョン管理する

### 5.4 ポリシー改定手順

1. 改定の必要性を特定する（定期レビューまたは随時）
2. Change Issue を起票する
3. 改定草案を作成する
4. 影響分析を実施する
5. ステークホルダーレビューを実施する
6. 承認を取得する
7. PR を通じて改定を反映する
8. 技術的強制の更新を実施する（該当する場合）
9. 周知・研修を実施する

### 5.5 定期レビュースケジュール

| ポリシー層 | レビュー頻度 | レビュー責任者 |
|-----------|-------------|---------------|
| Layer 1: 憲章・原則 | 年1回 | Executive Governance Board |
| Layer 2: 統治ポリシー | 半年ごと | Service Manager + EGB |
| Layer 3: プロセスポリシー | 四半期ごと | プロセスオーナー |
| Layer 4: 技術ポリシー | 四半期ごと | Technical Lead |

## 6. ポリシーの技術的強制

### 6.1 強制メカニズム

ポリシーは文書として存在するだけでなく、技術的に強制される。

| 強制手段 | 適用対象 | 例 |
|---------|---------|-----|
| GitHub Branch Protection | ブランチ保護ポリシー | main への直接 push 禁止 |
| GitHub CODEOWNERS | コードレビューポリシー | 特定パスのレビュー必須化 |
| GitHub Actions | CI/CD ポリシー | テスト通過の必須化 |
| PR Template | 変更管理ポリシー | 必須情報の入力強制 |
| Label Required | プロセスポリシー | ラベル付与の強制 |
| AI Policy Engine | AI 統治ポリシー | AI 行動の制約と記録 |

### 6.2 技術的強制の原則

1. 文書に定めたポリシーは、可能な限り技術的に強制する
2. 技術的強制が困難な場合は、自動監視と検知で補完する
3. 技術的強制と文書の整合性を常に維持する
4. 技術的強制の変更は、ポリシー改定と同時に行う

## 7. ポリシー例外管理

### 7.1 例外の定義

ポリシー例外とは、正当な理由に基づき、一時的または限定的に
ポリシーの適用を免除すること。

### 7.2 例外申請手順

1. 例外申請を Issue として起票する
2. 以下を記載する
   - 対象ポリシー
   - 例外の理由
   - 例外の範囲（期間・対象）
   - リスク評価
   - 代替制御策
3. ポリシーオーナーがレビューする
4. 承認権限者が判断する
   - Layer 1-2: Executive Governance Board
   - Layer 3: Management Layer
   - Layer 4: Technical Lead + Management
5. 承認された場合、例外レジスタに登録する
6. 例外期限の到来時にレビューする

### 7.3 例外レジスタ

すべての承認された例外は、以下の情報とともに管理される。

| 項目 | 内容 |
|------|------|
| 例外ID | 一意の識別子 |
| 対象ポリシー | 例外が適用されるポリシー |
| 申請者 | 例外を申請した者 |
| 承認者 | 例外を承認した者 |
| 理由 | 例外の正当な理由 |
| 範囲 | 例外の適用範囲 |
| 期間 | 例外の有効期間 |
| 代替制御 | ポリシー免除に代わる制御策 |
| リスク評価 | 例外によるリスク |
| レビュー日 | 次回レビュー日 |

### 7.4 例外の原則

- 不可侵原則（Non-Negotiables）に対する例外は認めない
- すべての例外には期限を設定する
- 例外は恒久化しない（恒久的な場合はポリシー改定を行う）
- 例外に対しても監査証跡を維持する

## 8. ポリシー間の整合性管理

### 8.1 整合性の原則

- 下位層のポリシーは上位層のポリシーに矛盾してはならない
- 同一層のポリシー間で矛盾が生じた場合は、ポリシーオーナー間で協議し解決する
- 矛盾が解決できない場合は、上位層の承認権限者が判断する

### 8.2 整合性チェック

ポリシーの作成・改定時には、以下の整合性チェックを実施する。

| チェック項目 | 確認内容 |
|-------------|---------|
| 上位ポリシーとの整合 | 上位層のポリシーと矛盾しないか |
| 同層ポリシーとの整合 | 同一層の他ポリシーと矛盾しないか |
| RACI との整合 | 役割と責任が RACI マトリクスと一致するか |
| 技術的強制との整合 | 技術的実装がポリシーの意図を正確に反映しているか |
| 用語の整合 | TERMINOLOGY_DEFINITIONS.md の定義と一致しているか |

## 9. ポリシー遵守のモニタリング

### 9.1 モニタリング手段

| 手段 | 頻度 | 対象 |
|------|------|------|
| 自動チェック（GitHub Actions） | 常時 | 技術ポリシー |
| 日次レポート | 日次 | プロセスポリシー |
| 週次レビュー | 週次 | 全ポリシー |
| 内部監査 | 四半期 | 全ポリシー |

### 9.2 遵守率の測定

ポリシー遵守率は以下の計算式で算出する。

```
遵守率 = (準拠チェック合格件数 / 全チェック件数) × 100%
```

目標遵守率:
- Layer 1-2: 100%（違反ゼロ）
- Layer 3: 95%以上
- Layer 4: 90%以上
