# ServiceMatrix 変更権限構造（Change Authority Structure）

Version: 1.0
Status: Active
Owner: Service Governance Authority
Classification: Governance Design Document
Last Updated: 2026-03-02

---

## 1. 文書の目的

本文書は、ServiceMatrix における変更権限の構造を定義する。
誰が、どのような変更を、どのレベルで承認する権限を持つかを
明確に規定し、変更管理の統治を確保する。

本文書は APPROVAL_CONTROL_MODEL.md と連携し、
承認フローの権限側面を詳細化するものである。

---

## 2. 変更権限の基本原則

### 2.1 権限設計の原則

1. **最小権限の原則**: 変更に必要な最小限の承認権限のみを付与する
2. **職務分掌の原則**: 申請・実施・承認・監査の権限を分離する
3. **リスク比例の原則**: リスクに比例した承認レベルを適用する
4. **明示的権限の原則**: すべての権限は明示的に付与され、暗黙の権限は存在しない
5. **委任の制限**: 権限の委任は事前登録制とし、再委任は禁止する

### 2.2 権限の範囲

変更権限は以下の行為に対して定義される。

| 権限カテゴリ | 内容 |
|------------|------|
| 起票権限 | RFC（変更要求）を起票する権限 |
| 評価権限 | 変更のリスクと影響を評価する権限 |
| 承認権限 | 変更の実施を承認する権限 |
| 実施権限 | 承認された変更を実施する権限 |
| ロールバック権限 | 変更をロールバックする権限 |
| クローズ権限 | 変更をクローズする権限 |

## 3. 変更権限者の階層構造

### 3.1 権限階層

```
┌──────────────────────────────────────────┐
│  Level 4: Executive Governance Board     │
│  最高承認権限。極高リスク変更の承認        │
├──────────────────────────────────────────┤
│  Level 3: Change Advisory Board (CAB)     │
│  高リスク変更の審査・承認                  │
├──────────────────────────────────────────┤
│  Level 2: Change Manager + Service Manager│
│  中リスク変更の承認、変更プロセスの管理     │
├──────────────────────────────────────────┤
│  Level 1: Process Owners / Technical Lead │
│  低リスク変更の承認、技術レビュー          │
├──────────────────────────────────────────┤
│  Level 0: Standard Change（事前承認）      │
│  定型変更の自動承認（CI通過による）         │
└──────────────────────────────────────────┘
```

### 3.2 各レベルの詳細

#### Level 0: Standard Change 自動承認

**権限者**: システム（CI パイプライン）

**承認条件**:
- Standard Change カタログに登録済み
- CI パイプラインが全チェック通過
- CODEOWNERS ルールに基づくレビュー完了（設定による）

**対象変更**:
- 定期パッチ適用
- ドキュメント更新
- 設定値の軽微な変更（事前承認済みリスト内）
- テストの追加・修正

#### Level 1: Process Owner / Technical Lead 承認

**権限者**:
- Technical Lead（TL）
- Process Owner（該当プロセスのオーナー）

**承認条件**:
- リスクスコア 0-20（低リスク）
- 影響範囲が限定的
- ロールバック計画が明確

**対象変更**:
- 単一サービスの機能修正
- 設定変更（影響範囲が限定的）
- 低リスクなバグ修正

**GitHub 連携**:
- PR に `risk:low` ラベル
- CODEOWNERS による TL のレビュー承認

#### Level 2: Change Manager / Service Manager 承認

**権限者**:
- Change Manager（CM）
- Service Manager（SM）

**承認条件**:
- リスクスコア 21-50（中リスク）
- 複数サービスへの影響がある場合
- SLA への影響が想定される場合

**対象変更**:
- 複数サービスに影響する機能追加
- インフラ設定の変更
- セキュリティ設定の変更

**GitHub 連携**:
- PR に `risk:medium` ラベル
- CM + TL によるレビュー承認
- セキュリティ関連は SEC のレビューも必要

#### Level 3: Change Advisory Board (CAB) 承認

**権限者**: CAB（委員会）

**承認条件**:
- リスクスコア 51-80（高リスク）
- 広範なサービス影響
- 重大な技術変更

**対象変更**:
- アーキテクチャ変更
- データベーススキーマ変更
- 本番環境のインフラ大規模変更
- セキュリティ統制の変更

**GitHub 連携**:
- PR に `risk:high` ラベル + `cab:required` ラベル
- CAB レビューの議事録を Issue コメントに記録
- CAB 承認後に `cab:approved` ラベルを付与

#### Level 4: Executive Governance Board (EGB) 承認

**権限者**: EGB（最高統治機関）

**承認条件**:
- リスクスコア 81-100（極高リスク）
- 統治ポリシーへの影響
- 組織横断的な影響

**対象変更**:
- 統治フレームワーク自体の変更
- SLA 目標の変更
- AI 自律レベルの変更
- 組織的なプロセス変更

**GitHub 連携**:
- PR に `risk:critical` ラベル
- CAB 承認 + EGB 承認の両方が必要
- EGB 承認は PR コメントまたは会議議事録として記録

## 4. 変更規模分類（L1 Minor / L2 Normal / L3 Major / Emergency）

ServiceMatrix では変更のリスクレベルに加え、**変更規模**による分類を併用する。
変更規模は承認フローの選択基準として機能する。

### 4.1 L1 Minor Change（軽微変更）

```
定義: 影響範囲が限定的で、ロールバックが容易な軽微な変更

条件（全て満たすこと）:
  - リスクスコア: 0-20
  - 影響ユーザー数: 100名未満
  - 影響サービス数: 1サービス
  - ダウンタイム: なし（または計画的な5分未満）
  - ロールバック: 即座に実施可能

典型的な例:
  - ドキュメントの誤字修正・更新
  - 非機能的なコードコメントの変更
  - 設定ファイルの閾値の微調整（事前承認済み範囲内）
  - テストコードの追加・修正

承認権限: Level 1（CM 単独承認）
GitHub ラベル: risk:low, change-size:minor
PR 承認数: 最低 1名
```

### 4.2 L2 Normal Change（通常変更）

```
定義: 標準的な変更プロセスを経て実施される、中程度のリスクを持つ変更

条件（リスクスコア 21-50 または以下の条件のいずれか）:
  - 影響ユーザー数: 100-1,000名
  - 影響サービス数: 2-5サービス
  - ダウンタイム: 計画的な30分未満
  - 外部 API や依存関係の変更を含む

典型的な例:
  - 新機能の追加（単一サービス）
  - バグ修正（影響範囲が中程度）
  - API エンドポイントの追加・変更
  - 設定変更（影響範囲が複数コンポーネント）
  - 依存ライブラリのバージョンアップ

承認権限: Level 2（CM + TL 承認）
GitHub ラベル: risk:medium, change-size:normal
PR 承認数: 最低 2名
```

### 4.3 L3 Major Change（重大変更）

```
定義: 広範な影響を持ち、CAB による審査が必要な重大な変更

条件（リスクスコア 51-100 または以下の条件のいずれか）:
  - 影響ユーザー数: 1,000名以上
  - 影響サービス数: 6サービス以上（または全サービス）
  - ダウンタイム: 30分以上または無計画ダウンタイムリスクあり
  - データベーススキーマの変更
  - セキュリティアーキテクチャの変更
  - 技術スタックの変更

典型的な例:
  - アーキテクチャ設計の変更
  - データベーススキーマの大規模変更
  - 認証・認可システムの変更
  - 本番インフラの大規模変更
  - AI 自律性レベルの変更

承認権限: Level 3-4（CAB 承認 または CAB + EGB 承認）
GitHub ラベル: risk:high または risk:critical, change-size:major
PR 承認数: 最低 3名（CAB メンバー含む）
```

### 4.4 Emergency Change（緊急変更）

```
定義: サービス継続性を脅かす緊急事態への即時対応変更

発動条件（いずれかを満たすこと）:
  - P1 インシデント発生中（サービス中断）
  - セキュリティ脆弱性の緊急対応
  - SLA 重大違反が発生中または切迫している

制約:
  - 通常の変更フローを簡略化して実施
  - 事後 CAB レビュー（72時間以内）が必須
  - 簡略化の記録と理由が必須

承認権限: Emergency Level（SM 単独承認）
GitHub ラベル: type:emergency-change
PR 承認数: 最低 1名（SM）、事後 CAB レビュー必須
```

### 4.5 変更規模 × 変更タイプ対応表

| 変更規模 | Standard | Normal | Emergency |
|---------|---------|--------|-----------|
| L1 Minor | ○（典型） | ○ | - |
| L2 Normal | △（大規模な Standard） | ○（典型） | - |
| L3 Major | - | ○ | △（大規模緊急） |
| Emergency | - | - | ○（典型） |

## 5. GitHub ブランチ保護統合（Branch Protection Integration）

### 5.1 変更規模別ブランチ保護マッピング

変更規模と承認権限レベルを GitHub ブランチ保護ルールで技術的に強制する。

```
ブランチ戦略と保護レベルの対応:

  main ブランチ（L3 Major / Emergency の最終マージ先）:
  ┌─────────────────────────────────────────────────────┐
  │  Required approvals: 2                              │
  │  Dismiss stale reviews on push: ON                  │
  │  Require review from CODEOWNERS: ON                 │
  │  Required status checks:                            │
  │    - ci/lint                                        │
  │    - ci/test                                        │
  │    - ci/security-scan                               │
  │  Require branches to be up to date: ON              │
  │  Require signed commits: ON                         │
  │  Require linear history: ON                         │
  │  Block force pushes: ON                             │
  │  Restrict who can push: Admins only                 │
  └─────────────────────────────────────────────────────┘

  release/* ブランチ（L2 Normal の統合先）:
  ┌─────────────────────────────────────────────────────┐
  │  Required approvals: 2                              │
  │  Require review from CODEOWNERS: ON                 │
  │  Required status checks: ci/lint, ci/test           │
  │  Block force pushes: ON                             │
  └─────────────────────────────────────────────────────┘

  feature/* ブランチ（L1 Minor / L2 Normal の開発先）:
  ┌─────────────────────────────────────────────────────┐
  │  Required approvals: 1                              │
  │  Required status checks: ci/lint                    │
  └─────────────────────────────────────────────────────┘

  hotfix/* ブランチ（Emergency Change 用）:
  ┌─────────────────────────────────────────────────────┐
  │  Required approvals: 1（SM 承認で可）               │
  │  Required status checks: ci/lint, ci/test（努力目標） │
  │  type:emergency-change ラベル必須                   │
  │  事後 CAB レビュー Issue 自動作成                   │
  └─────────────────────────────────────────────────────┘
```

### 5.2 変更規模と必要承認数マッピング

| 変更規模 | ブランチ | 必要承認数 | CODEOWNERS | CI 必須 |
|---------|---------|-----------|-----------|--------|
| L1 Minor | feature/* | 1（CM） | 任意 | lint |
| L2 Normal | feature/* → release/* | 2（CM + TL） | 必須 | lint + test |
| L3 Major | feature/* → main | 3以上（CAB含む） | 必須 | 全 CI |
| Emergency | hotfix/* → main | 1（SM）+ 事後 CAB | 任意 | 可能な限り |

### 5.3 GitHub ラベルによる変更規模の強制

GitHub Actions が PR ラベルを検証し、承認数の過不足を自動チェックする。

```yaml
# .github/workflows/change-authority-check.yml の概要

name: Change Authority Check
on:
  pull_request_review:
    types: [submitted]

jobs:
  verify-authority:
    steps:
      - name: Check change-size label
        run: |
          LABELS=$(gh pr view $PR_NUMBER --json labels --jq '.labels[].name')
          # change-size:major かつ CAB 承認なし → 失敗
          # change-size:normal かつ承認数 < 2 → 失敗
          # type:emergency-change かつ SM 承認なし → 失敗

      - name: Verify AI change approval
        run: |
          # ai-generated ラベルがある場合、人間承認を確認
          # AI が承認者でないことを確認
```

## 6. 委任規則（Delegation Rules）

### 6.1 通常時の委任規則

権限者が不在・休暇等の場合に、承認権限を委任できる。

```
委任の基本規則:
  1. 委任は書面（GitHub Issue）で事前に登録する
  2. 委任先は元の権限者と同等以上の資格を持つ者に限る
  3. 委任期間の上限は 7日間（SM 承認で延長可）
  4. 再委任（委任先から別の者への委任）は禁止
  5. 委任元は委任期間中も結果責任を負う
  6. 自己関連変更の承認への委任は禁止（SoD 原則）
```

| 権限者 | 委任可能な代理 | 委任承認者 | 最長期間 |
|--------|-------------|-----------|---------|
| CM | 事前登録済み CM 代理 | SM | 7日 |
| TL | 指定シニアエンジニア | CM | 7日 |
| SM | 事前登録済み SM 代理 | EGB | 7日 |
| CAB 議長（CM） | CAB 副議長（SM） | EGB | 14日 |
| CAB メンバー | 指定代理 CAB メンバー | CAB 議長 | 7日 |
| EGB メンバー | 委任不可 | - | - |
| SEC | 指定代理セキュリティ担当 | SM | 7日 |
| AIG | 指定代理 AI 担当 | SM + EGB | 7日 |

### 6.2 緊急時の委任規則

緊急変更（Emergency Change）発動時に、通常の権限者が不在の場合の特例。

```
緊急時委任フロー（連絡不能な場合の代行順序）:

  通常 Emergency 承認者: SM
    │ 不在・連絡不能
    ↓
  第1代行: CM（SM 権限を緊急代行）
    │ CM も不在・連絡不能
    ↓
  第2代行: TL（CM 権限の一部を緊急代行、コード変更のみ）
    │ TL も不在・連絡不能
    ↓
  第3代行: EGB に直接エスカレーション
    （EGB メンバー1名で Emergency 承認可）

  緊急代行時の記録要件:
  - 代行実施の理由を Issue に記録
  - 代行者名と代行時刻を記録
  - 元の権限者への事後報告（復帰後 2時間以内）
  - 通常体制回復後の事後承認取得（24時間以内）
```

### 6.3 委任レジスタ

有効な委任はすべて委任レジスタで管理する。

| 項目 | 内容 |
|------|------|
| 委任 ID | DEL-YYYYMM-NNN 形式 |
| 委任元 | 委任する権限者 |
| 委任先 | 代理承認者 |
| 権限範囲 | 委任する承認権限の範囲 |
| 有効期間 | 開始日〜終了日 |
| 委任承認者 | 委任を承認した者 |
| GitHub Issue | 委任記録の Issue URL |
| ステータス | 有効 / 期限切れ / 終了 |

## 7. CAB（Change Advisory Board）の詳細

### 4.1 CAB の構成

| 役割 | 人数 | 責任 |
|------|------|------|
| 議長（CM） | 1名 | 会議の運営、議事進行、最終取りまとめ |
| 技術代表（TL） | 1-2名 | 技術的リスクの評価 |
| サービス代表（SM） | 1名 | サービス影響の評価 |
| セキュリティ代表（SEC） | 1名 | セキュリティリスクの評価 |
| SME（臨時） | 必要に応じ | 特定領域の専門的評価 |

### 4.2 CAB の運営

**定例 CAB**:
- 頻度: 週1回（毎週水曜日）
- 所要時間: 最大60分
- 議題: 週内の Normal Change（高リスク）の審査

**臨時 CAB**:
- 召集条件: 極高リスク変更、緊急変更の事後レビュー
- 召集権限: CM または SM
- 開催期限: 召集から24時間以内

**ECAB（Emergency CAB）**:
- 召集条件: Emergency Change の事後レビュー
- 構成: 最低3名（CM + TL + SM）
- 開催期限: 緊急変更実施後72時間以内

### 4.3 CAB の意思決定

| 判断 | 条件 | 結果 |
|------|------|------|
| 承認 | 出席メンバーの過半数が承認 | 変更を Approved 状態に遷移 |
| 条件付き承認 | 追加条件の履行を前提に承認 | 条件充足後に Approved に遷移 |
| 差し戻し | 追加情報が必要 | 変更を Review 状態に戻す |
| 却下 | リスクが許容不可 | 変更を Rejected 状態に遷移 |

### 4.4 CAB 議事録

すべての CAB 会議は、以下の議事録を残す。

| 項目 | 内容 |
|------|------|
| 日時 | 開催日時 |
| 参加者 | 出席メンバー一覧 |
| 審査変更一覧 | 審査した変更 ID と概要 |
| 各変更の判断 | 承認/却下/差し戻しとその理由 |
| リスク評価サマリー | AI スコアと CAB の追加評価 |
| 条件・アクション | 条件付き承認の場合の条件、次回アクション |

議事録は GitHub Issue にコメントとして記録し、監査証跡とする。

## 8. 緊急変更の権限構造

### 8.1 緊急変更の発動権限

| 役割 | 権限 |
|------|------|
| Incident Manager | 緊急変更の発動を要請できる |
| Change Manager | 緊急変更の発動を承認できる |
| Service Manager | CM 不在時に緊急変更を承認できる |

### 8.2 緊急変更の実施権限

| 実施内容 | 権限者 | 条件 |
|---------|--------|------|
| コード変更 | TL / DEV | CM または SM の口頭承認 |
| 設定変更 | OPS / TL | CM または SM の口頭承認 |
| ロールバック | OPS / TL / IM | IM の判断で即時実行可 |
| インフラ変更 | OPS | CM + SM の口頭承認 |

### 8.3 緊急変更の事後処理

1. 緊急変更レポートの作成（24時間以内）
2. ECAB レビューの実施（72時間以内）
3. 恒久対策の計画（必要な場合）
4. 統治記録の完成

## 9. AI エージェントの変更権限

### 9.1 AI が行える変更

| 変更内容 | 権限 | 条件 |
|---------|------|------|
| 分析・レポート作成 | 自動許可 | 記録必須 |
| PR の作成 | 自動許可 | 承認は人間が行う |
| 低リスク自動修復 | 条件付き | ポリシー範囲内 + 記録必須 |
| CI 失敗の自動修正提案 | 自動許可 | 提案のみ |
| Standard Change の実施 | 条件付き | CI 通過 + ポリシー範囲内 |

### 9.2 AI が行えない変更

| 変更内容 | 理由 |
|---------|------|
| Normal / Emergency Change の承認 | 承認は人間のみ |
| 権限設定の変更 | セキュリティ統制 |
| 統治ポリシーの変更 | 統治の主体は人間 |
| 自身の自律レベルの変更 | 自己権限昇格の禁止 |
| main ブランチへの直接変更 | ブランチ保護ルール |
| CODEOWNERS の変更 | 承認構造の変更は人間のみ |

### 9.3 AI 変更の監視

AI が行ったすべての変更は、以下の監視対象となる。

| 監視項目 | 方法 | 頻度 |
|---------|------|------|
| 変更内容の妥当性 | 人間レビュー | 全件 |
| ポリシー準拠 | 自動チェック | 全件 |
| 権限範囲の逸脱 | 自動検知 | 全件 |
| 影響範囲の正確性 | 事後レビュー | 週次 |

## 10. 権限の付与と剥奪

### 10.1 権限付与の手順

1. 権限付与の必要性を確認する
2. 対象者の適格性を確認する
3. SoD 制約との整合性を確認する
4. 権限付与申請を Issue で起票する
5. 承認権限者の承認を取得する
6. GitHub 権限設定に反映する
7. 権限レジスタを更新する

### 10.2 権限剥奪の手順

1. 権限剥奪の理由を記録する
2. 権限剥奪を Issue で起票する
3. GitHub 権限設定から削除する
4. 権限レジスタを更新する
5. 対象者に通知する

### 10.3 定期レビュー

| レビュー対象 | 頻度 | 責任者 |
|-------------|------|--------|
| 個人の権限 | 四半期 | SM |
| CAB メンバー構成 | 半年 | CM |
| EGB メンバー構成 | 年次 | EGB 議長 |
| AI 権限設定 | 四半期 | SM + SEC |
| CODEOWNERS | 四半期 | TL |
