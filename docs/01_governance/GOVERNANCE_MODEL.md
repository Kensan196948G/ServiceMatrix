Version: 1.0
Status: Active
Owner: Service Governance Authority
Classification: Governance Design Document
Last Updated: 2026-03-02

## 1. 文書の目的

本文書は、ServiceMatrix における統治モデルの全体設計を定義する。
統治の階層構造、意思決定フレームワーク、承認権限マトリクス、
統治指標（KGI/KPI）、違反対応手順を体系的に記述する。

本文書は SERVICEMATRIX_CHARTER.md に基づき、
統治の具体的な実装設計を提供する。

---

## 2. 統治の定義と範囲

### 2.1 統治の定義

ServiceMatrix における統治とは、
IT サービスの運用・変更・改善に関するすべての活動を、
事前に定義された原則・ポリシー・プロセスに基づいて
制御・監視・改善する体系的な仕組みである。

### 2.2 統治の範囲

| 統治対象 | 範囲 |
|----------|------|
| プロセス | Incident / Change / Problem / Request / Release |
| 技術基盤 | GitHub / CI/CD / インフラ |
| AI エージェント | Agent Teams / SubAgent の行動・権限 |
| コンプライアンス | SLA / 監査 / 法規制 / 社内ポリシー |
| ドキュメント | 統治文書体系の維持と改定 |

## 3. 統治階層構造

ServiceMatrix の統治は、3層の階層構造で運営される。

### 3.1 階層構造の全体像

```
┌─────────────────────────────────────────┐
│         Executive Layer（経営層）          │
│   戦略的方向性・ポリシー承認・最終決裁     │
├─────────────────────────────────────────┤
│        Management Layer（管理層）          │
│   プロセス運営・KPI 管理・統治監視         │
├─────────────────────────────────────────┤
│       Operational Layer（運用層）          │
│   実務遂行・インシデント対応・変更実施     │
└─────────────────────────────────────────┘
```

### 3.2 Executive Layer（経営層）

**役割**: 統治の戦略的方向性を決定し、最終的な説明責任を負う。

| 権限 | 説明 |
|------|------|
| ポリシー承認 | 統治ポリシーの最終承認 |
| 戦略決定 | 統治の戦略的方向性の決定 |
| 予算承認 | 統治関連投資の承認 |
| 例外承認 | ポリシー逸脱の例外承認 |
| 是正指示 | 重大な統治違反に対する是正指示 |

**構成**: Executive Governance Board（EGB）

**会議体**:
- 統治ステアリング委員会（四半期ごと）
- 緊急統治会議（重大事案発生時）

### 3.3 Management Layer（管理層）

**役割**: プロセスの設計・運営・改善を管理し、KPI の達成を推進する。

| 権限 | 説明 |
|------|------|
| プロセス設計 | 運用プロセスの設計と改定 |
| KPI 管理 | 統治指標の設定と監視 |
| 変更承認 | Normal Change の承認 |
| リソース配分 | 運用リソースの配分決定 |
| エスカレーション判断 | 経営層へのエスカレーション |

**構成**:
- Service Manager（SM）
- Change Manager（CM）
- Incident Manager（IM）
- Problem Manager（PM）
- Release Manager（RM）
- Security Officer（SEC）

**会議体**:
- 運用レビュー会議（週次）
- CAB（変更諮問委員会、週次または随時）
- サービス改善会議（月次）

### 3.4 Operational Layer（運用層）

**役割**: 日常の運用業務を遂行し、統治ポリシーに準拠した作業を実施する。

| 権限 | 説明 |
|------|------|
| インシデント対応 | 定義されたプロセスに基づく対応 |
| Standard Change 実施 | 事前承認済み変更の実施 |
| サービス要求処理 | カタログに基づくサービス提供 |
| 日常監視 | システム・SLA の日常監視 |

**構成**:
- Service Desk（SD）
- Technical Lead（TL）
- Operations Team（OPS）
- Development Team（DEV）
- AI Agent（ClaudeCode Agent Teams / SubAgent）

## 4. 意思決定フレームワーク

### 4.1 意思決定の原則

ServiceMatrix における意思決定は、以下の原則に従う。

1. **権限に基づく判断**: 定義された権限範囲内で判断する
2. **記録の義務**: すべての判断を記録する
3. **エビデンスベース**: データと証拠に基づいて判断する
4. **エスカレーション義務**: 権限を超える場合は上位層にエスカレーションする
5. **SoD の遵守**: 職務分掌の制約を遵守する

### 4.2 意思決定マトリクス

| 判断カテゴリ | Operational | Management | Executive |
|-------------|-------------|------------|-----------|
| インシデント対応（P3-P5） | 決定 | 報告受領 | - |
| インシデント対応（P1-P2） | 実行 | 決定 | 報告受領 |
| Standard Change | 実行 | 報告受領 | - |
| Normal Change | 提案 | 決定（CAB） | 報告受領 |
| Emergency Change | 実行 | 決定・事後承認 | 報告受領 |
| 高リスク変更 | 提案 | 推薦 | 決定 |
| ポリシー変更 | 提案 | レビュー | 決定 |
| SLA 目標変更 | - | 提案 | 決定 |
| AI 自律レベル変更 | - | 提案 | 決定 |
| 統治例外承認 | - | 申請 | 決定 |

### 4.3 エスカレーションフレームワーク

| エスカレーション条件 | エスカレーション先 | 期限 |
|---------------------|-------------------|------|
| P1 インシデント未解決（1時間超） | Management Layer | 即時 |
| SLA 逸脱リスク（残30%） | Management Layer | 自動 |
| SLA 違反確定 | Executive Layer | 1時間以内 |
| 変更失敗（本番影響あり） | Management Layer | 即時 |
| セキュリティインシデント | SEC + Management | 即時 |
| AI 異常行動検知 | Management Layer | 即時 |
| 統治ポリシー違反 | Management Layer | 24時間以内 |
| 重大統治違反 | Executive Layer | 即時 |

## 5. 承認権限マトリクス

### 5.1 変更承認権限

| 変更タイプ | リスクレベル | 承認権限 | 承認方法 |
|-----------|-------------|---------|---------|
| Standard Change | 低 | 事前承認済み | 自動（PR CI 通過） |
| Normal Change（低リスク） | 低-中 | Change Manager | PR レビュー承認 |
| Normal Change（中リスク） | 中 | CAB | CAB レビュー + PR 承認 |
| Normal Change（高リスク） | 高 | CAB + SM | CAB + SM 承認 |
| Emergency Change | 可変 | IM + SM（事後 CAB） | 緊急承認フロー |
| ポリシー変更 | - | EGB | 正式承認プロセス |

### 5.2 運用承認権限

| 対象 | 承認権限 | 条件 |
|------|---------|------|
| インシデント クローズ | IM | 復旧確認済み |
| 問題 クローズ | PM | 根本原因解決済み |
| サービス要求承認 | SRO | カタログ準拠 |
| 本番デプロイ | RM + CM | CI 通過 + レビュー完了 |
| ロールバック実行 | IM or CM | サービス影響確認 |

### 5.3 AI 権限承認

| AI 行動 | 承認要件 |
|---------|---------|
| 分析・レポート生成 | 自動（記録のみ） |
| 低リスク自動修復 | ポリシー範囲内で自動（事後レビュー） |
| PR 作成 | 自動（レビュー承認は人間が行う） |
| 高リスク操作 | 人間による事前承認 |
| Agent Teams 構成変更 | SM 承認 |

## 7. 統治ダッシュボード

### 7.1 ダッシュボードの構成

統治状態をリアルタイムに可視化するダッシュボードを運用する。

| セクション | 表示内容 |
|-----------|---------|
| サービス健全性 | SLA 達成状況、アクティブインシデント数 |
| 変更管理状況 | 承認待ち変更数、進行中変更数、変更成功率 |
| コンプライアンス | ポリシー準拠率、SoD 違反アラート |
| AI 統治 | AI 自律レベル状態、判断ログ件数 |
| KPI トレンド | 主要 KPI の時系列推移 |
| アラート | 未対応エスカレーション、SLA 逸脱リスク |

### 7.2 レポートスケジュール

| レポート | 頻度 | 対象 | 内容 |
|---------|------|------|------|
| 日次統治サマリー | 日次 | Management | インシデント・変更・SLA |
| 週次運用レビュー | 週次 | Management | KPI 進捗・課題・対策 |
| 月次統治レポート | 月次 | Executive + Management | 全 KPI・トレンド・改善 |
| 四半期統治アセスメント | 四半期 | Executive | 成熟度・戦略進捗 |

## 8. 違反対応手順

### 8.1 違反の分類

| 分類 | 説明 | 例 |
|------|------|-----|
| Level 1: 軽微 | 手続き上の不備 | 記録の不備、軽微な手順逸脱 |
| Level 2: 中程度 | プロセス違反 | 承認なき変更実施、SLA 逸脱 |
| Level 3: 重大 | 統治原則違反 | SoD 違反、監査証跡の改ざん |
| Level 4: 致命的 | 不可侵原則違反 | セキュリティ侵害、権限の不正昇格 |

### 8.2 対応フロー

#### Level 1: 軽微な違反

1. 検知（自動または手動）
2. 記録（違反ログに記録）
3. 通知（プロセスオーナーに通知）
4. 是正（プロセスオーナーが対応）
5. 完了確認
6. 記録クローズ

**対応期限**: 5営業日以内
**エスカレーション**: 不要

#### Level 2: 中程度の違反

1. 検知（自動または手動）
2. 記録（違反ログに記録）
3. アラート（Management Layer に即時通知）
4. 影響分析（AI による影響範囲の推定）
5. 暫定対策（即時対応が必要な場合）
6. 根本原因分析
7. 是正計画策定
8. 是正実施
9. 効果確認
10. 記録クローズ

**対応期限**: 3営業日以内
**エスカレーション**: Management Layer

#### Level 3: 重大な違反

1. 検知（自動検知が望ましい）
2. 即時記録
3. 即時アラート（Management Layer + Executive Layer）
4. 緊急対策（被害拡大防止）
5. 詳細影響分析
6. 根本原因分析
7. 是正計画策定（Executive 承認要）
8. 是正実施
9. 効果確認
10. 再発防止策の策定
11. 事後レビュー
12. 記録クローズ

**対応期限**: 即時着手、24時間以内に暫定対策
**エスカレーション**: Executive Layer

#### Level 4: 致命的な違反

1. 即時検知・記録
2. 即時アラート（全層）
3. 緊急停止措置（必要に応じ）
4. 緊急統治会議の招集
5. 被害範囲の確定
6. 緊急是正措置
7. 詳細調査
8. 根本原因分析
9. 包括的是正計画
10. 是正実施
11. 独立検証
12. 再発防止策の策定・承認
13. 統治フレームワークの見直し
14. 事後レビュー
15. 記録クローズ

**対応期限**: 即時
**エスカレーション**: Executive Layer + 必要に応じ外部

### 8.3 是正措置の追跡

すべての是正措置は以下の情報とともに追跡される。

- 違反ID
- 違反分類
- 検知日時
- 影響範囲
- 根本原因
- 是正措置の内容
- 担当者
- 期限
- 進捗状況
- 完了確認者
- 完了日時

## 9. 統治モデルの進化

### 9.1 成熟度モデル

ServiceMatrix の統治成熟度を以下のレベルで評価する。

| レベル | 名称 | 特徴 |
|--------|------|------|
| 1 | 初期 | アドホックな統治、属人的 |
| 2 | 管理 | プロセスが定義され、部分的に実施 |
| 3 | 定義 | 全プロセスが標準化され、組織的に実施 |
| 4 | 定量管理 | 定量的に測定・管理されている |
| 5 | 最適化 | 継続的に改善されている |

### 9.2 目標成熟度

| フェーズ | 期限 | 目標レベル |
|---------|------|-----------|
| 基盤構築 | 2026 Q2 | レベル2 |
| プロセス実装 | 2026 Q4 | レベル3 |
| 統治成熟 | 2027 Q2 | レベル3（安定） |
| 最適化 | 2028 | レベル4 |

### 9.3 改定手順

本文書の改定は以下の手順に従う。

1. 改定提案（Issue 起票）
2. 影響分析
3. Management Layer レビュー
4. Executive Layer 承認
5. 文書更新（PR 経由）
6. 関連文書の整合性確認
7. 周知・研修
