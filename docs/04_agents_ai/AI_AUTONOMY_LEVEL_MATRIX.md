# AI自律度レベルマトリックス（AI Autonomy Level Matrix）

Version: 2.0
Status: Active
Classification: Internal / Governance
Owner: Service Governance Authority
Compliance: ITIL 4 / ISO/IEC 20000 / J-SOX
Last Updated: 2026-03-02

---

## 1. 目的

本文書は、ServiceMatrix における AI エージェントの自律度レベルを4段階で定義し、
タスク種別ごとの適用マトリックス、レベル変更手順、および緊急時の対応を規定する。

AI の自律度は統治上の重要な制御変数である。
適切な自律度設定により、効率性と統制のバランスを維持する。

---

## 2. 自律度4段階の詳細定義

### 2.1 L0: 完全人間制御（Full Human Control）

| 項目 | 内容 |
|------|------|
| 定義 | AIは情報収集・提示のみ行い、判断・実行はすべて人間が行う |
| AIの役割 | 情報提供者・アナリスト |
| 人間の役割 | すべての判断・実行・記録 |
| 承認要件 | 全操作に対して事前承認必須 |
| 適用場面 | セキュリティ設定変更・データスキーマ変更・インシデント対応 |
| リスク許容度 | ゼロリスク原則 |

AIが実施できる操作（L0）:
- ファイルの読み取り・内容分析
- コードレビューコメントの生成
- 調査結果・分析結果の提示
- 修復オプションの説明（実行なし）

### 2.2 L1: 推奨のみ（Recommendation Only）

| 項目 | 内容 |
|------|------|
| 定義 | AIは具体的な修復手順・コード差分を提示する。実行は人間が手動で行う |
| AIの役割 | アドバイザー・手順書生成者 |
| 人間の役割 | 提案内容の評価・手動実行・記録 |
| 承認要件 | 実行前に人間確認必須 |
| 適用場面 | 設計変更提案・新機能実装・複雑なバグ修正 |
| リスク許容度 | 低リスクまで |

AIが実施できる操作（L1）:
- L0の全操作
- 具体的なコード差分の生成・提示
- GitHub Issueへのコメント投稿
- PR説明文の草稿作成
- 修復手順書の生成

### 2.3 L2: 低リスク自動実行（Low-Risk Automation）

| 項目 | 内容 |
|------|------|
| 定義 | リスクスコアLow以下（<0.4）の修復を自動実行する。事後にログ・通知で報告する |
| AIの役割 | 実行者（制限範囲内） |
| 人間の役割 | 例外・中リスク以上への対応・定期レビュー |
| 承認要件 | 低リスクは不要（事後通知）。中リスク以上は事前承認 |
| 適用場面 | Lintエラー修正・フォーマット・Markdownlint・パッチ更新 |
| リスク許容度 | Low（スコア0.4未満）まで |

AIが実施できる操作（L2）:
- L1の全操作
- Lintエラーの自動修正（featureブランチ上）
- フォーマット不整合の自動修正
- 未使用インポートの自動除去
- Markdownlintエラーの自動修正
- CI再実行のトリガー
- 修復コミットの作成（featureブランチのみ）

### 2.4 L3: 自動実行＋報告（Autonomous Execution with Reporting）

| 項目 | 内容 |
|------|------|
| 定義 | 設定された範囲内で修復を完全自動実行し、結果を包括的に報告する |
| AIの役割 | 自律実行者・報告者 |
| 人間の役割 | 例外対応・設定管理・定期レビュー |
| 承認要件 | 通常操作は不要。設計変更・セキュリティ変更は引き続き禁止 |
| 適用場面 | CI自動修復ループ・定型メンテナンス・テスト実行 |
| リスク許容度 | Low（スコア0.4未満）まで（範囲は L2 と同様） |

AIが実施できる操作（L3）:
- L2の全操作
- PRの自動作成（featureブランチ → mainへの提案）
- GitHub Issueの自動作成（修復提案・エスカレーション）
- 修復レポートの自動生成・投稿
- 定期メンテナンスタスクの自動実行

## 4. 自律度変更の承認プロセス

### 4.1 変更フロー

```
申請者がChange Issueを作成
    │
    ├─ 必須記載事項
    │   ├─ 変更対象のタスク種別
    │   ├─ 現在の自律度レベル
    │   ├─ 要求する自律度レベル
    │   ├─ 変更理由・必要性
    │   ├─ リスク評価
    │   ├─ 適用期間（恒久 or 期間限定）
    │   └─ ロールバック計画
    │
    ▼
Service Governance Authority によるレビュー
    │
    ├─ 承認基準の確認
    │   ├─ リスク評価の妥当性
    │   ├─ J-SOX・ITIL整合性
    │   └─ 前例・影響範囲
    │
    ▼
決定（承認 / 条件付き承認 / 否認）
    │
    ├─ 承認 ──→ AI_AUTONOMY_LEVEL_MATRIX.md を更新 → 変更記録
    └─ 否認 ──→ 否認理由をIssueにコメント → クローズ
```

### 4.2 変更の承認基準

| 変更方向 | 承認要件 |
|---------|---------|
| L0 → L1 | Service Governance Authority 承認 |
| L1 → L2 | Service Governance Authority 承認 + リスク評価書 |
| L2 → L3 | Service Governance Authority 承認 + リスク評価書 + パイロット期間 |
| L2 → L1 | Service Governance Authority 通知（セキュリティ強化として推奨） |
| L1 → L0 | Service Governance Authority 通知 |
| セキュリティ関連変更 | 変更不可（常にL0固定） |

## 5. 緊急時の自律度一時拡張手順

### 5.1 一時拡張が認められる条件

以下のすべてを同時に満たす場合のみ、自律度の一時拡張を認める。

1. 本番環境または主要サービスに影響するインシデントが発生している
2. 通常の変更管理プロセスに必要な時間が確保できない
3. 一時拡張によるリスクよりインシデント継続リスクが明らかに大きい
4. 拡張権限を付与する担当者（承認者）が明示できる

### 5.2 緊急一時拡張の手順

```
Step 1: 緊急Change Issueの作成
  必須記載:
  ├─ インシデント内容・影響範囲
  ├─ 一時拡張する自律度レベル（例: L1 → L2）
  ├─ 拡張期間（最大: インシデント解決まで、最長4時間）
  ├─ 承認者氏名
  └─ 緊急承認タイムスタンプ

Step 2: 承認者による即時承認（口頭 + Issue記録）

Step 3: 一時拡張の実施
  ├─ 拡張範囲の厳密な制限（インシデント対応タスクのみ）
  └─ 実施内容をリアルタイムでIssueにコメント記録

Step 4: インシデント解決後の復旧
  ├─ 自律度レベルを元に戻す
  ├─ 実施内容のフルログを提出
  └─ 事後レビューの実施（1営業日以内）
```

### 5.3 記録要件（緊急時）

緊急一時拡張中のすべてのAI決定は通常より厳密に記録する。

| 記録項目 | 記録先 |
|---------|-------|
| 一時拡張承認記録 | GitHub Issues（恒久保持） |
| 実施したAI操作ログ | `.claude/logs/decisions/` + `docs/15_audit_evidence/` |
| インシデント対応結果 | GitHub Issues |
| 事後レビュー記録 | GitHub Issues + `docs/15_audit_evidence/` |

## 7. 定期レビュースケジュール

### 7.1 レビュー体制

| レビュー種別 | 頻度 | 主担当 | 副担当 | 成果物 |
|-----------|------|-------|-------|-------|
| 自律度設定妥当性確認 | 月次 | Service Governance Authority | サービスオーナー | レビュー記録 |
| 一時拡張履歴の棚卸し | 月次 | Service Governance Authority | - | 改善提案（必要に応じて） |
| マトリックス全体見直し | 四半期 | Service Governance Authority | 全Agent Team代表 | 改訂版マトリックス |
| J-SOX適合性確認 | 半期 | 監査担当 | Service Governance Authority | 監査報告書 |
| 年次ポリシー改訂 | 年次 | Service Governance Authority | ステークホルダー | 改訂版本文書 |

### 7.2 レビュー観点

各定期レビューでは以下の観点を確認する。

- 実際のAI運用実績と設定の整合性（過剰制限・過剰自律がないか）
- インシデント・ロールバック発生状況と自律度の相関
- 一時拡張の発生頻度・適切性
- 新たなタスク種別への対応（マトリックスへの追加）
- 技術的変更（新ツール・新エージェント）への対応
