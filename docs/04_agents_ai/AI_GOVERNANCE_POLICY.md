# AI統治ポリシー（AI Governance Policy）

Version: 1.0
Status: Active
Classification: Internal / Governance
Owner: Service Governance Authority
Last Updated: 2026-03-02

---

## 1. 目的

本ポリシーは、ServiceMatrix における AI エージェント（ClaudeCode Agent Teams、SubAgent を含む）の
統治原則、運用規則、および制約を定義する。

AI は ServiceMatrix の補助者であり、統治の主体ではない。
すべての AI 活動は本ポリシーに準拠しなければならない。

---

## 2. 適用範囲

本ポリシーは以下に適用される。

- ClaudeCode メインセッション
- Agent Teams（Lead + Member 構成）
- SubAgent（軽量タスク実行）
- GitHub Actions 内の AI 連携処理
- 自動修復ループにおける AI 判断

---

## 3. AI 自律レベル定義

ServiceMatrix は AI の自律度を 5 段階で定義する。

### 3.1 レベル一覧

| レベル | 名称 | 説明 | 承認要件 |
|--------|------|------|----------|
| L0 | 手動（Manual） | AI は一切の自律動作を行わない。人間がすべてを実行する | なし（AI 不使用） |
| L1 | 提案（Advisory） | AI は分析・提案のみを行う。実行は人間が行う | 提案の閲覧のみ |
| L2 | 半自動（Semi-Autonomous） | AI はファイル編集・PR作成を行えるが、すべてに人間承認が必要 | 各操作前に承認 |
| L3 | 自動（Autonomous） | AI は commit / push / テスト実行を承認後に自律的に行う | バッチ承認可能 |
| L4 | 完全自律（Fully Autonomous） | AI が自律的にデプロイまで実行する | 原則禁止 |

### 3.2 デフォルト設定

ServiceMatrix の標準自律レベルは **L2（半自動）** とする。

L3 への昇格は以下の条件を満たす場合にのみ許可される。

- 対象リポジトリの CI がすべて Green であること
- 変更影響範囲が明確かつ限定的であること
- 人間管理者が明示的に L3 を許可していること

L4 は原則として使用しない。研究目的の場合のみ、隔離環境で許可される。

## 4. AI 意思決定の記録要件

### 4.1 記録必須条件

すべての AI 判断は以下を記録しなければならない。

- **タイムスタンプ**: ISO 8601 形式（例: 2026-03-02T14:30:00+09:00）
- **実行主体**: エージェント識別子（例: agent-teams-lead, subagent-lint-01）
- **判断種別**: 分析 / 提案 / 修正 / 承認要求 / 自動実行
- **入力データ**: 判断の根拠となった情報
- **出力結果**: AI が生成した判断・成果物
- **信頼度スコア**: AI の判断確信度（0.0 - 1.0）
- **承認要否**: 人間承認が必要かどうか
- **承認者**: 承認した人間の識別子（該当する場合）

### 4.2 記録保存場所

| 記録種別 | 保存先 | 保持期間 |
|----------|--------|----------|
| 設計判断 | GitHub Issues コメント | 永続 |
| 自動修復ログ | `.claude/logs/` + GitHub Actions ログ | 1 年 |
| CI 修復判断 | Pull Request コメント | PR 存続期間 |
| 監査対象判断 | `docs/15_audit_evidence/` | 5 年 |
| セッションログ | `.claude/sessions/` | 90 日 |

### 4.3 記録フォーマット

推奨フォーマットは JSON Lines（.jsonl）とする。

```json
{
  "timestamp": "2026-03-02T14:30:00+09:00",
  "agent_id": "agent-teams-lead",
  "decision_type": "auto_repair",
  "input": {"ci_failure": "lint_error", "file": "src/auth.ts", "line": 42},
  "output": {"action": "fix_applied", "diff": "+  const result = validate(input);"},
  "confidence": 0.95,
  "approval_required": true,
  "approved_by": null,
  "autonomy_level": "L2"
}
```

## 5. 承認ゲート設計

### 5.1 承認不要な操作（L0-L1 相当）

以下の操作は AI が自律的に実行できる。

- `git status` の実行
- `git diff` の実行
- ファイル読取（Read）
- コード分析・静的解析
- 提案文書の生成
- ログ閲覧

### 5.2 承認必須な操作（L2 相当）

以下の操作は人間承認を経て実行する。

- ファイル編集（Edit / Write）
- `git add` の実行
- `git commit` の実行
- Pull Request の作成
- Issue の作成・更新
- Agent Teams の spawn

### 5.3 厳格承認が必要な操作（L3 相当）

以下の操作は明示的な承認と理由記録が必要である。

- `git push` の実行
- Pull Request の merge
- ブランチ戦略の変更
- CI パイプラインの変更
- セキュリティ設定の変更
- 権限構成の変更

### 5.4 原則禁止の操作（L4 相当）

以下の操作は原則として禁止する。

- `git push --force`
- main / master ブランチへの直接変更
- 本番環境へのデプロイ
- 権限昇格を伴う変更
- セキュリティポリシーの無効化
- 監査ログの削除

## 6. AI の禁止事項

### 6.1 絶対的禁止事項

以下は ServiceMatrix における AI の絶対的禁止事項である。

1. **権限昇格禁止**: AI は自身の権限を昇格する操作を提案・実行してはならない
2. **記録なき変更禁止**: すべての変更は記録を伴わなければならない
3. **SLA 迂回禁止**: SLA 制約を回避する手段を提案してはならない
4. **無断コミット禁止**: 人間の承認なくコミットを作成してはならない
5. **無断 Push 禁止**: 人間の承認なく Push を実行してはならない
6. **承認なき Merge 禁止**: Pull Request の自動 Merge を行ってはならない
7. **秘密情報出力禁止**: ハードコードされた認証情報を生成してはならない
8. **認可回避禁止**: 認可チェックを迂回するコードを提案してはならない

### 6.2 条件付き禁止事項

以下は特定条件下で禁止される。

| 操作 | 禁止条件 | 許可条件 |
|------|----------|----------|
| 自動修復 | 設計変更を伴う場合 | 低リスク・テスト範囲内の場合 |
| Agent Teams 拡張 | 無断での追加 | 人間承認後 |
| CI 設定変更 | 既存パイプライン改変 | 新規追加かつ既存に影響なし |
| データスキーマ変更 | 破壊的変更 | 後方互換性維持の場合 |

## 7. AI ポリシー違反時の対応手順

### 7.1 違反検知

ポリシー違反は以下の手段で検知される。

1. **CI パイプライン検査**: GitHub Actions によるポリシー準拠チェック
2. **Hooks 検査**: pre-commit / pre-push フックによるリアルタイム検知
3. **監査ログ照合**: 定期的なログレビューによる事後検知
4. **Agent Teams 相互監視**: レビュー担当 Agent による検知

### 7.2 違反対応フロー

```
違反検知
  │
  ├─ 重大度判定
  │   ├─ Critical（権限昇格・データ漏洩）→ 即時停止 + 人間エスカレーション
  │   ├─ High（無断変更・SLA 違反）→ 操作ロールバック + 調査開始
  │   ├─ Medium（記録不備・手順逸脱）→ 警告 + 是正措置
  │   └─ Low（軽微な手順違反）→ ログ記録 + 次回防止策
  │
  ├─ 証跡保全
  │   ├─ 違反時のログスナップショット保存
  │   ├─ 関連 diff の保全
  │   └─ エージェント状態の記録
  │
  ├─ 是正措置
  │   ├─ ロールバック実行（必要な場合）
  │   ├─ ポリシー再確認
  │   └─ 再発防止策の策定
  │
  └─ 報告
      ├─ GitHub Issue での違反報告作成
      ├─ 監査ログへの記録
      └─ 関係者への通知
```

### 7.3 エスカレーション基準

| 重大度 | 対応時間 | エスカレーション先 | 必要アクション |
|--------|----------|-------------------|---------------|
| Critical | 即時 | プロジェクトオーナー | AI 即時停止・全変更ロールバック |
| High | 1 時間以内 | テクニカルリード | 操作停止・影響調査 |
| Medium | 24 時間以内 | チームリード | 是正計画策定 |
| Low | 次回定例時 | 担当者 | 記録・改善提案 |

## 8. AI エージェント間の統治

### 8.1 Agent Teams 内の統治

- Lead Agent は Teams 内の統治責任を持つ
- Member Agent は Lead の指示範囲内でのみ動作する
- 同一ファイルの同時編集は禁止する
- WorkTree による物理的分離を原則とする

### 8.2 SubAgent の統治

- SubAgent は呼び出し元のコンテキスト内で動作する
- SubAgent は独立した意思決定を行わない
- SubAgent の実行結果は呼び出し元が検証する

---

## 9. ポリシー更新手順

本ポリシーの変更は以下の手順に従う。

1. 変更提案を GitHub Issue として作成する
2. 影響分析を実施する
3. 関係者レビューを行う
4. 承認者の明示的承認を得る
5. Pull Request として変更を提出する
6. CI 検証を通過する
7. Merge 後、全エージェントに通知する
