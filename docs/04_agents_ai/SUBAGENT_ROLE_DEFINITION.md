# SubAgent 役割定義（SubAgent Role Definition）

Version: 1.0
Status: Active
Classification: Internal / Architecture
Owner: Service Governance Authority
Last Updated: 2026-03-02

---

## 1. 目的

本文書は、ServiceMatrix における SubAgent の定義、役割、適用場面、
および Agent Teams との違いを明確にする。

SubAgent は軽量かつ低コストなタスク実行手段であり、
Agent Teams と適切に使い分けることで、開発効率とコスト効率を最大化する。

---

## 2. SubAgent の定義

SubAgent とは、メインセッション（またはAgent Teams の Lead/Member）から
呼び出される **単一タスク実行用の軽量エージェント** である。

### 2.1 特徴

- **コンテキスト共有**: 呼び出し元と同一のリポジトリコンテキストで動作する
- **WorkTree 不要**: 独立した WorkTree を作成しない
- **短命**: タスク完了後に自動的に終了する
- **低コスト**: Agent Teams と比較してトークン消費が少ない
- **独立意思決定なし**: 呼び出し元の指示範囲内でのみ動作する

### 2.2 SubAgent の動作モデル

```
呼び出し元（Main / Lead / Member）
  │
  ├─ SubAgent 呼び出し
  │   ├─ タスク記述を渡す
  │   ├─ 必要なコンテキストを渡す
  │   └─ 制約条件を指定する
  │
  ├─ SubAgent 実行
  │   ├─ 指定タスクの実行
  │   ├─ 結果の生成
  │   └─ 呼び出し元への結果返却
  │
  └─ 呼び出し元が結果を検証・統合
```

## 3. SubAgent と Agent Teams の違い

### 3.1 比較表

| 項目 | SubAgent | Agent Teams |
|------|----------|-------------|
| 構成 | 単体 | Lead + Members（複数） |
| コンテキスト | 呼び出し元と共有 | 各 Agent が独立 |
| WorkTree | 不要 | 1 Agent = 1 WorkTree |
| ライフサイクル | 短命（タスク完了で終了） | 長命（チーム解散まで存続） |
| 通信 | 結果返却のみ | SendMessage / TaskSystem |
| コスト | 低 | 中〜高 |
| 適用規模 | 小タスク | 中〜大規模タスク |
| 並列度 | 呼び出し元が制御 | チーム内で自律的に並列 |
| 意思決定 | 呼び出し元に委任 | Lead が統括判断 |
| ファイル編集範囲 | 呼び出し元と同一制約 | WorkTree 内に限定 |

### 3.2 選択基準

以下のフローチャートで SubAgent と Agent Teams を選択する。

```
タスク受領
  │
  ├─ 単一ファイル修正か？
  │   ├─ Yes → SubAgent
  │   └─ No ↓
  │
  ├─ 複数レイヤーにまたがるか？
  │   ├─ Yes → Agent Teams
  │   └─ No ↓
  │
  ├─ ファイル間の依存関係が強いか？
  │   ├─ Yes → Agent Teams（依存管理が必要）
  │   └─ No ↓
  │
  ├─ 並列実行で効率化できるか？
  │   ├─ Yes → Agent Teams
  │   └─ No → SubAgent
  │
  └─ 不明な場合 → SubAgent で開始し、必要に応じて Agent Teams に移行
```

## 4. SubAgent の適用場面

### 4.1 推奨される適用場面

#### Lint 修正

- ESLint / Prettier の自動修正
- 型エラーの修正
- インポート順序の整理
- 未使用変数の除去

```
SubAgent 呼び出し例:
  タスク: "src/auth.ts の lint エラーを修正してください"
  タイプ: general-purpose
  制約: "ロジック変更は行わない。フォーマット修正のみ"
```

#### 単一ファイル改善

- 関数の命名改善
- コメント追加・更新
- 型定義の追加
- リファクタリング（小規模）

```
SubAgent 呼び出し例:
  タスク: "src/utils/validator.ts の validateInput 関数にJSDocを追加してください"
  タイプ: general-purpose
  制約: "既存のロジックは変更しない"
```

#### 軽量レビュー

- 特定ファイルのコード品質チェック
- セキュリティ脆弱性の簡易スキャン
- パフォーマンスの簡易分析

```
SubAgent 呼び出し例:
  タスク: "src/api/handler.ts のSQLインジェクション脆弱性をチェックしてください"
  タイプ: Explore
  制約: "レポートのみ生成。修正は行わない"
```

#### コード調査・探索

- 特定の関数の使用箇所調査
- 依存関係の分析
- 特定パターンの検索

```
SubAgent 呼び出し例:
  タスク: "validatePermission 関数がプロジェクト内のどこから呼ばれているか調査してください"
  タイプ: Explore
  制約: "呼び出し元のファイルパスと行番号を一覧化する"
```

### 4.2 推奨されない適用場面

以下の場合は Agent Teams を使用すべきである。

- 複数ファイルにまたがる設計変更
- API とUI の同時開発
- 複数仮説の並列検証
- 長時間にわたる反復的作業
- ファイル間の整合性確保が必要な作業

## 5. SubAgent の実行制約

### 5.1 コンテキスト制約

- SubAgent は呼び出し元のリポジトリコンテキストを共有する
- SubAgent 独自の永続状態は持たない
- SubAgent 間のコンテキスト共有はない（各 SubAgent は独立）

### 5.2 権限制約

- SubAgent は AI_GOVERNANCE_POLICY の自律レベル制約に従う
- SubAgent は呼び出し元の権限を超える操作を行えない
- SubAgent は git commit / push を直接実行しない（結果を呼び出し元に返す）

### 5.3 実行時間制約

- SubAgent は短時間での完了を前提とする
- 長時間実行が予想される場合は Agent Teams への移行を検討する
- タイムアウト上限はシステム設定に従う

### 5.4 ファイル操作制約

- SubAgent は呼び出し元が指定したファイル範囲内で作業する
- 指定外のファイルへの変更は原則禁止する
- 変更結果は呼び出し元が検証した後にのみ反映される

## 6. 利用可能な SubAgent タイプ

### 6.1 Explore

**目的**: コードベースの調査・探索

**適用場面**:
- ファイル構造の調査
- 特定パターンの検索
- 依存関係の分析
- アーキテクチャの把握

**特徴**:
- 読取専用の操作に特化
- 広範囲のコードベース探索が可能
- 結果をサマリとして返却

**呼び出し例**:
```
Agent ツール使用:
  subagent_type: Explore
  prompt: "認証モジュールの全体構造を調査し、
           主要なコンポーネントと依存関係を報告してください"
```

### 6.2 Plan

**目的**: 実装計画の策定

**適用場面**:
- 変更計画の立案
- リファクタリング計画の策定
- 影響範囲の分析

**特徴**:
- 分析と計画策定に特化
- 実際のファイル変更は行わない
- 構造化された計画文書を返却

### 6.3 General-Purpose

**目的**: 汎用的なタスク実行

**適用場面**:
- Lint 修正
- コード修正
- ドキュメント生成
- テスト追加

**特徴**:
- ファイル読取・編集が可能
- 汎用的なタスクに対応
- 呼び出し元が結果を検証

### 6.4 PR Review Toolkit 系

**目的**: Pull Request レビュー支援

**適用場面**:
- PR の差分分析
- コード品質チェック
- セキュリティレビュー
- テスト網羅性確認

**特徴**:
- PR の差分を入力として受取
- レビューコメントを生成
- 問題の重大度を分類

## 7. SubAgent の呼び出しパターン

### 7.1 直列呼び出し

一つの SubAgent が完了してから次を呼び出す。

```
Main → SubAgent-A（完了）→ SubAgent-B（完了）→ 結果統合
```

**適用場面**: 前のタスク結果が次のタスクの入力となる場合

### 7.2 並列呼び出し

複数の SubAgent を同時に呼び出す。

```
Main → SubAgent-A ─┐
     → SubAgent-B ─┤→ 結果統合
     → SubAgent-C ─┘
```

**適用場面**: 独立したタスクを並列実行する場合

### 7.3 条件分岐呼び出し

最初の SubAgent の結果に応じて次の SubAgent を選択する。

```
Main → SubAgent-A（分析）
         │
         ├─ 結果 X → SubAgent-B
         └─ 結果 Y → SubAgent-C
```

**適用場面**: 調査結果に応じて対応を変える場合

## 8. SubAgent のエラーハンドリング

### 8.1 SubAgent が失敗した場合

1. 呼び出し元がエラー内容を確認する
2. 入力パラメータを見直す
3. タスク記述を具体化して再試行する
4. 再試行でも失敗する場合は手動対応または Agent Teams に移行する

### 8.2 SubAgent のタイムアウト

1. タイムアウトが発生した場合は結果を破棄する
2. タスクを小さく分割して再試行する
3. 分割困難な場合は Agent Teams に移行する

## 9. SubAgent 利用時のベストプラクティス

### 9.1 タスク記述

- **具体的に書く**: 対象ファイル、関数名、行番号を明示する
- **制約を明示する**: 変更してよい範囲と変更禁止事項を記述する
- **期待する出力を示す**: 返却してほしい結果のフォーマットを指定する

### 9.2 コスト意識

- 単純なタスクに Agent Teams を使用しない
- 複雑なタスクに SubAgent を無理に適用しない
- 並列呼び出しは独立性が確認できた場合のみ行う

### 9.3 検証

- SubAgent の結果は必ず呼び出し元が検証する
- 自動的に結果を反映せず、内容を確認してから適用する
- 期待と異なる結果の場合はタスク記述を見直す
