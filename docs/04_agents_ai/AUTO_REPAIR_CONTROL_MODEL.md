# AI自動修復制御モデル（Auto Repair Control Model）

Version: 2.0
Status: Active
Classification: Internal / Operations
Owner: Service Governance Authority
Compliance: ITIL 4 / ISO/IEC 20000 / J-SOX
Last Updated: 2026-03-02

---

## 1. 目的

本文書は、ServiceMatrix における AI による自動修復の許可条件、禁止条件、
承認フロー、自律度レベル別制御、ログ記録要件、およびロールバック手順を定義する。

AI自動修復は CI の安定性維持と開発効率向上のための重要な機能であるが、
無制限な自動修復は統治原則に反するため、厳格な統制下で運用しなければならない。

---

## 2. 自動修復可能範囲

### 2.1 許可される修復対象

以下の変更は、適切なリスク評価と承認プロセスを経たうえで自動修復を許可する。

| 修復対象 | 具体例 | デフォルト自律度 |
|---------|-------|----------------|
| CI失敗（Lint） | ESLint・Pylint・Markdownlintエラー | L2 |
| CI失敗（フォーマット） | Prettier・Blackによるフォーマット不整合 | L2 |
| CI失敗（未使用インポート） | 未使用インポートの除去 | L2 |
| Markdown Lint | 見出し・リスト・リンク形式の修正 | L2 |
| YAML設定整合性 | インデント・型・必須フィールド修正 | L2（条件付き） |
| 依存関係（パッチ更新） | パッチバージョンの自動更新 | L2（条件付き） |
| テストデータ軽微修正 | テスト期待値の軽微な更新 | L1 |
| ドキュメント誤字脱字 | スペル・句読点の修正 | L2 |

### 2.2 自動修復の許可条件

AI による自動修復は、以下の **すべての条件** を同時に満たす場合のみ許可する。

| 条件 | 判定基準 | 検証方法 |
|------|---------|---------|
| 低リスク変更 | リスクスコア 0.4 未満 | リスク評価エンジン |
| 設計影響なし | データモデル・APIインターフェース変更を含まない | 変更対象ファイル種別確認 |
| テスト範囲内 | 既存テストでカバーされている範囲の修正 | カバレッジ分析 |
| 後方互換性維持 | 破壊的変更を含まない | インターフェース差分分析 |
| 変更量制限 | 変更行数100行未満・変更ファイル数4以下 | diff統計 |
| 単一責務 | 1つの問題に対する1つの修正 | 変更の凝集性確認 |

## 3. 自動修復禁止範囲

### 3.1 絶対禁止事項

以下のいずれかに該当する変更は、いかなる条件下でも自動修復を禁止する。

| 禁止対象 | 禁止理由 | 対応方針 |
|---------|---------|---------|
| データスキーマ変更 | データ整合性・マイグレーション整合の維持 | 設計レビュー後に人間が実施 |
| セキュリティ設定変更 | 認可・認証・暗号化の安全性確保 | セキュリティレビュー必須 |
| 設計・アーキテクチャ変更 | システム全体整合性の維持 | Change Issue作成・Agent Teamsレビュー |
| main直接変更 | ブランチ保護ポリシー違反 | featureブランチ経由のPR必須 |
| CI/CDパイプライン変更 | CI統治ルールの維持 | 変更管理プロセス経由 |
| 環境変数・シークレット変更 | セキュリティポリシー遵守 | 人間による手動操作のみ |
| APIインターフェース変更 | 後方互換性・SLA維持 | 影響分析・バージョニング必須 |
| 権限・ロール設定変更 | アクセス制御の完全性保護 | アクセス管理プロセス経由 |
| 依存関係メジャーバージョン更新 | 破壊的変更リスク | 互換性テスト・人間判断必須 |

### 3.2 条件付き禁止

以下の条件に該当する場合、自動修復を禁止する。

| 条件 | 禁止される場合 | 許可される場合 |
|------|-------------|-------------|
| 変更量 | 100行以上の変更 | 100行未満の変更 |
| 変更ファイル数 | 5ファイル以上 | 4ファイル以下 |
| テスト不在 | 変更箇所のテストが存在しない | 既存テストでカバー済み |
| 依存更新範囲 | メジャー・マイナーバージョン更新 | パッチバージョン更新のみ |
| リスクスコア | 0.4以上 | 0.4未満 |

## 4. 修復承認フロー

### 4.1 全体フロー

```
┌──────────────────────────────────────────────────────────────────┐
│  CI失敗 / 品質問題 検知                                           │
└───────────────────────────────┬──────────────────────────────────┘
                                │
                                ▼
┌──────────────────────────────────────────────────────────────────┐
│  Phase 1: 分析                                                    │
│  - 失敗原因の特定                                                  │
│  - 影響範囲の分析                                                  │
│  - 修復対象ファイルの特定                                           │
└───────────────────────────────┬──────────────────────────────────┘
                                │
                                ▼
┌──────────────────────────────────────────────────────────────────┐
│  Phase 2: リスク評価                                              │
│  - リスクスコア算出（0.0 - 1.0）                                   │
│  - 変更量・範囲の計測                                              │
│  - 自律度レベル判定                                                │
└────────┬─────────────────────┬──────────────────────────────────┘
         │                     │
     Low以下（<0.4）        Medium以上（>=0.4）
         │                     │
         ▼                     ▼
┌─────────────────┐   ┌─────────────────────────────────────────┐
│ Phase 3a:        │   │ Phase 3b:                               │
│ 修復案生成        │   │ AI提案作成                              │
│ （自動実行対象）  │   │ - GitHub Issueへの提案投稿              │
└────────┬────────┘   │ - 複数修復オプション提示                │
         │            │ - 各オプションのリスク・影響付記         │
         │            └──────────────┬──────────────────────────┘
         │                           │
         │                           ▼
         │            ┌─────────────────────────────────────────┐
         │            │ Phase 4: 人間確認・承認                  │
         │            │ - 提案内容のレビュー                     │
         │            │ - 承認 / 却下 / 修正指示                 │
         │            │ - 承認記録の保存                         │
         │            └──────────────┬──────────────────────────┘
         │                           │
         └───────────────────────────┤
                                     │
                                     ▼
                          ┌─────────────────────┐
                          │ Phase 5: 修復実行    │
                          │ - featureブランチ上  │
                          │ - 変更内容をログ記録 │
                          └──────────┬──────────┘
                                     │
                                     ▼
                          ┌─────────────────────┐
                          │ Phase 6: 検証        │
                          │ - CI再実行           │
                          │ - テスト通過確認     │
                          │ - 品質基準確認       │
                          └──────────┬──────────┘
                                     │
                          ┌──────────┴──────────┐
                          │                     │
                        成功                  失敗
                          │                     │
                          ▼                     ▼
                       完了               ロールバック
                    （ログ記録）          ／エスカレーション
```

### 4.2 リスク評価スコア定義

| リスクレベル | スコア範囲 | 自動修復 | 承認要件 |
|------------|----------|---------|---------|
| Negligible | 0.0 - 0.2 | 自動実行可 | ログ記録のみ |
| Low | 0.2 - 0.4 | 自動実行可 | ログ記録 + 事後通知 |
| Medium | 0.4 - 0.6 | 事前承認必要 | 人間承認後に実行 |
| High | 0.6 - 0.8 | 禁止 | Agent Teamsレビュー必須 |
| Critical | 0.8 - 1.0 | 禁止 | 設計レビュー必須 |

## 5. 自律度レベル別制御

### 5.1 自律度4段階の定義

| レベル | 名称 | 定義 | 人間関与 |
|-------|------|------|---------|
| L0 | 完全人間制御 | AIは情報提供のみ。すべての判断・実行を人間が行う | 全操作 |
| L1 | 推奨のみ | AIは修復案・手順を提示する。実行は人間が手動で行う | 実行のみ |
| L2 | 低リスク自動 | リスクスコアが Low 以下の修復を自動実行する。事後報告 | 中リスク以上 |
| L3 | 自動実行+報告 | 設定された範囲内で修復を自動実行し、結果を報告する | 例外・エスカレーション |

### 5.2 自律度レベル別の許可操作

| 操作 | L0 | L1 | L2 | L3 |
|-----|----|----|----|----|
| ファイル読み取り | 許可 | 許可 | 許可 | 許可 |
| 分析・調査 | 許可 | 許可 | 許可 | 許可 |
| 修復案生成・提示 | 許可 | 許可 | 許可 | 許可 |
| Lintエラー自動修正 | 禁止 | 禁止 | 許可 | 許可 |
| フォーマット自動修正 | 禁止 | 禁止 | 許可 | 許可 |
| 設定ファイル軽微修正 | 禁止 | 禁止 | 条件付き | 条件付き |
| PRの自動作成 | 禁止 | 禁止 | 禁止 | 許可 |
| commit・push | 禁止 | 禁止 | 禁止（featureブランチのみ） | 許可（featureのみ） |
| 設計変更 | 禁止 | 禁止 | 禁止 | 禁止 |
| セキュリティ変更 | 禁止 | 禁止 | 禁止 | 禁止 |
| main変更 | 禁止 | 禁止 | 禁止 | 禁止 |

### 5.3 自律度レベルのデフォルト設定

本番運用時のデフォルト自律度レベルは **L2** とする。

| コンテキスト | デフォルトレベル | 変更可否 |
|------------|---------------|---------|
| 通常開発時 | L2 | 承認後にL1/L3変更可 |
| セキュリティ対応時 | L0 | 変更禁止 |
| 設計変更時 | L0 | 変更禁止 |
| 緊急インシデント対応時 | L1 | 承認後にL2変更可 |
| CI自動修復ループ | L2 | 変更禁止 |

### 5.4 自律度レベルの変更プロセス

```
変更申請（Change Issue作成）
    │
    ▼
影響分析・リスク評価
    │
    ▼
Service Governance Authority 承認
    │
    ▼
変更実施・記録
    │
    ▼
定期レビュー（四半期）
```

## 6. 修復履歴のログ記録要件

### 6.1 必須記録項目

すべての自動修復試行（成功・失敗問わず）に対して以下を記録しなければならない。

```json
{
  "repair_id": "repair-20260302-001",
  "timestamp": "2026-03-02T14:30:00+09:00",
  "trigger": {
    "type": "ci_failure | quality_degradation | security_advisory",
    "source": "GitHub Actions ジョブ名 / ツール名",
    "error_message": "エラーメッセージ原文",
    "affected_file": "対象ファイルパス",
    "affected_line": 42
  },
  "analysis": {
    "root_cause": "根本原因の説明",
    "risk_score": 0.1,
    "risk_level": "Negligible | Low | Medium | High | Critical",
    "affected_files": ["ファイルパスのリスト"],
    "estimated_change_lines": 3
  },
  "autonomy_level": "L0 | L1 | L2 | L3",
  "repair_action": {
    "description": "実施した修復の説明",
    "diff_summary": "変更概要",
    "commit_hash": "コミットハッシュ（実行した場合）",
    "branch": "修復を実施したブランチ名"
  },
  "approval": {
    "required": true,
    "requested_at": "2026-03-02T14:30:05+09:00",
    "approved_by": "承認者ID（nullの場合は不要）",
    "approved_at": "2026-03-02T14:35:00+09:00",
    "approval_comment": "承認コメント"
  },
  "verification": {
    "ci_rerun_result": "pass | fail | skipped",
    "tests_passed": true,
    "coverage_delta": 0.0,
    "new_issues_introduced": false
  },
  "outcome": "success | failure | rollback | escalation",
  "duration_seconds": 300
}
```

### 6.2 ログ保存先と保持期間

| ログ種別 | 保存先 | 保持期間 | アーカイブ先 |
|---------|-------|---------|------------|
| 修復実行ログ | `.claude/logs/repairs/` | 1年 | `archive/repairs/` |
| CI連携ログ | GitHub Actionsログ | GitHub設定に従う | - |
| 承認記録 | GitHub Issues / PRコメント | 永続 | - |
| ロールバック記録 | `.claude/logs/rollbacks/` | 1年 | `archive/rollbacks/` |
| エスカレーション記録 | GitHub Issues + `.claude/logs/escalations/` | 1年 | - |

### 6.3 ログ完全性要件

- 自動修復の試行はすべて記録する（成功・失敗・キャンセルを含む）
- ログは追記のみ許可する（既存ログの変更・削除禁止）
- 日次でログファイルのSHA-256ハッシュ値を計算し、改ざん検知可能な形で保存する
- 監査時は `docs/15_audit_evidence/` に証拠ファイルとして提出できる形式で保存する

## 7. 修復失敗時のロールバック手順

### 7.1 自動ロールバック条件

以下のいずれかに該当する場合、修復後の変更を自動的にロールバックする。

| 条件 | ロールバック方法 |
|------|---------------|
| 修復後のCIが修復前より悪化した | `git revert {commit}` |
| 修復により新たなエラーが発生した | `git revert {commit}` |
| テストカバレッジが低下した | `git revert {commit}` |
| セキュリティスキャンで問題が検出された | `git revert {commit}` + 即時通知 |
| 3回の修復ループで解決しない | 全修復コミットを `git revert` + エスカレーション |

### 7.2 ロールバック実行手順

```
Step 1: ロールバック対象の特定
  ├─ 修復ログからコミットハッシュを取得
  └─ 変更内容と副作用を確認

Step 2: ロールバック実行（featureブランチ上）
  ├─ git revert {commit-hash} --no-edit
  ├─ 人間による承認（リスクLow以下の場合は事後承認可）
  └─ commit + push（featureブランチ）

Step 3: 検証
  ├─ CI再実行
  ├─ ロールバック前の状態に戻ったことを確認
  └─ 追加の副作用がないことを確認

Step 4: 記録
  ├─ ロールバック理由をログに記録（.claude/logs/rollbacks/）
  ├─ GitHub Issueにロールバック記録をコメントとして追加
  ├─ 再発防止策の検討と記録
  └─ 必要に応じて自動修復ルールを更新
```

### 7.3 ロールバック後の対応フロー

```
ロールバック完了
    │
    ▼
根本原因再分析
    │
    ├─ 自動修復可能 ──→ 修復ルール修正後に再試行
    └─ 自動修復不可 ──→ エスカレーション（GitHub Issue作成 + 人間対応依頼）
```

### 7.4 エスカレーション条件

以下のいずれかに該当する場合、自動修復を中断し人間にエスカレーションする。

- 修復ループが3回を超えた場合
- 累積変更量が100行を超えた場合
- ロールバック後も問題が解消しない場合
- リスクスコアがMedium以上に上昇した場合
- 設計変更が必要と判断された場合

## 8. CI自己修復ループ制限

### 8.1 ループ制限値

| 制限項目 | 上限値 | 超過時の処理 |
|---------|-------|------------|
| 最大ループ回数 | 3回 | エスカレーション |
| 1ループあたり最大変更量 | 50行 | ループ停止・人間確認 |
| 累積最大変更量 | 100行 | エスカレーション |
| 1セッション最大修復件数 | 10件 | 承認要求 |

## 9. 定期レビューとメトリクス

### 9.1 監視メトリクス

| メトリクス | 警告閾値 | 対応 |
|----------|--------|------|
| 自動修復成功率 | 80%未満 | ルール見直し |
| ロールバック発生率 | 10%以上 | リスク評価精度向上 |
| エスカレーション率 | 30%以上 | 人間対応プロセス強化 |
| 平均ループ回数 | 2.0以上 | 修復ロジック改善 |

### 9.2 定期レビュースケジュール

| レビュー種別 | 頻度 | 実施者 |
|------------|------|-------|
| 自動修復ログレビュー | 週次 | Service Governance Authority |
| 自律度レベル妥当性確認 | 月次 | サービスオーナー |
| 修復ルール改善検討 | 四半期 | Service Governance Authority |
| J-SOX証跡確認 | 半期 | 監査担当 |
