## 1. 目的と適用範囲

### 1.1 目的

本文書は、ServiceMatrix プロジェクトにおけるリスクアセスメントの標準プロセス、
手法、ツール、および責任体制を定義する。

ITIL 4 のリスク管理実践および ISO/IEC 20000-1 のリスクおよび機会の取り扱いに準拠し、
J-SOX IT 統制における内部統制の有効性維持に寄与する。

### 1.2 適用範囲

- ServiceMatrix システム全体（バックエンド API / フロントエンド / AI レイヤー / GitHub 統合）
- 本番環境、ステージング環境、開発環境
- 外部統合（GitHub API / Claude AI API）
- 運用プロセスおよび変更管理プロセス

---

## 2. リスクアセスメントプロセス

リスクアセスメントは以下の 5 フェーズで実施する。

```
識別 --> 分析 --> 評価 --> 対応 --> 監視
```

### 2.1 フェーズ 1: リスク識別（Risk Identification）

#### 目的

潜在的なリスクを網羅的に洗い出し、リスク登録簿に登録する。

#### 実施方法

| 手法 | 説明 | 実施者 |
|------|------|--------|
| ブレインストーミング | チームメンバーによる自由討論 | 全チーム |
| チェックリストレビュー | 標準リスクカテゴリに基づく確認 | リスクオーナー |
| 過去インシデント分析 | 過去の障害・インシデントからの学習 | 運用チーム |
| 変更リクエスト分析 | 変更提案からのリスク抽出 | 変更管理者 |
| AI 提案レビュー | Claude AI による自動リスク提案の人間レビュー | リスク委員会 |
| 外部情報収集 | CERT / NVD / ITIL ガイダンス参照 | セキュリティチーム |

#### リスク識別のトリガー

- 新機能・変更リクエストの提起
- インシデント・問題の発生
- 外部環境変化（法令改正 / 業界動向）
- 四半期定期レビュー
- 監査指摘事項

#### 識別基準

以下の問いに 1 つでも「Yes」の場合、リスクとして登録する。

1. システムの可用性に影響を与える可能性があるか？
2. データの機密性・完全性・可用性を損なう可能性があるか？
3. コンプライアンス違反を引き起こす可能性があるか？
4. AI の判断が意図しない結果をもたらす可能性があるか？
5. SLA 目標値を達成できなくなる可能性があるか？

---

### 2.2 フェーズ 2: リスク分析（Risk Analysis）

#### 目的

識別されたリスクの可能性（Likelihood）と影響度（Impact）を定量的に評価する。

#### 分析観点

**定性分析**
- リスクの性質・根本原因の理解
- 既存の統制措置の有効性評価
- リスク間の依存関係・連鎖リスクの特定

**定量分析**
- 可能性スコア（1-5）の算定
- 影響度スコア（1-5）の算定
- リスクスコア = 可能性 x 影響度（1-25）

詳細スコアリング定義は `RISK_SCORING_LOGIC.md` を参照。

#### AI 固有の分析項目

AI レイヤーに関連するリスクは以下の追加観点で分析する。

- 自律度分析: AI エージェントがどの程度自律的に判断・実行するか
- 誤判定連鎖: AI の誤判定が後続プロセスに与える影響範囲
- 説明可能性: AI の判断根拠が監査証拠として記録されているか
- バイアス評価: トレーニングデータや判断ロジックに偏りがないか

---

### 2.3 フェーズ 3: リスク評価（Risk Evaluation）

#### 目的

算定されたリスクスコアに基づき、対応の優先順位を決定する。

#### リスク許容基準

| リスクレベル | スコア範囲 | 対応方針 |
|------------|----------|---------|
| 重大（Critical） | 20-25 | 即時対応必須。全活動を停止し優先処理 |
| 高（High） | 15-19 | 緊急対応。1 週間以内に軽減計画策定 |
| 中（Medium） | 7-14 | 計画的対応。次スプリントまでに対応計画策定 |
| 低（Low） | 1-6 | 受容または監視継続。四半期レビューで確認 |

#### リスク受容基準

以下をすべて満たす場合のみ、リスクを受容（Accept）できる。

1. リスクスコアが 6 以下（低リスク）
2. リスクオーナーが書面で受容を承認
3. 定期監視の仕組みが存在する
4. J-SOX / ISO 20000 統制要件に影響しない

AI 関連リスクは追加承認が必要:
- AI ガバナンス委員会の承認
- 誤判定率の定量的根拠の提示

---

### 2.4 フェーズ 4: リスク対応（Risk Treatment）

#### 対応戦略

| 戦略 | 定義 | 適用条件 |
|------|------|---------|
| 回避（Avoid） | リスクを生じさせる活動を中止 | 重大リスク・代替手段あり |
| 軽減（Mitigate） | リスクの発生確率または影響度を低下 | 高・中リスクの主要戦略 |
| 転嫁（Transfer） | リスクを第三者（保険・外部委託）に移転 | 財務的リスク |
| 受容（Accept） | 対応コストがリスクを上回る場合に受容 | 低リスクのみ |

#### 対応計画の必須要素

- 対応戦略の明確な選択と理由
- 具体的な軽減措置（技術的・手続き的・人的）
- 対応期限
- 担当者とリスクオーナー
- 残留リスクのスコア評価
- 対応効果の測定指標（KRI: Key Risk Indicator）

#### AI 関連リスクへの対応原則

- AI エージェントへの権限付与は最小権限の原則を適用
- 重大リスクを伴う AI 判断には必ず人間の承認ステップを挿入
- AI 判断のロールバック機能を実装
- 継続的な AI 出力モニタリングを実施

---

### 2.5 フェーズ 5: リスク監視（Risk Monitoring）

#### 監視頻度

| リスクレベル | 監視頻度 | 担当者 |
|------------|---------|--------|
| 重大 | 毎日 | リスクオーナー + 委員会 |
| 高 | 毎週 | リスクオーナー |
| 中 | 毎月 | リスクオーナー |
| 低 | 四半期 | リスク管理者 |

#### 監視方法

**自動監視（CI/CD 統合）**
- GitHub Actions による自動セキュリティスキャン（毎コミット）
- SLA 監視ダッシュボード（リアルタイム）
- エラー率・レスポンスタイムのアラート（閾値超過時自動通知）
- AI 判断ログの自動集計（日次）

**手動監視（人間主導）**
- 週次インシデントレビューミーティング
- 月次リスク登録簿レビュー
- 四半期リスクアセスメントレビュー（詳細後述）

#### KRI（主要リスク指標）

| 指標 | 定義 | 閾値（警告） | 閾値（重大） |
|------|------|------------|------------|
| インシデント発生率 | 月次 P1/P2 インシデント件数 | 3 件/月 | 5 件/月 |
| SLA 達成率 | 月次 SLA 目標達成率 | 98% 未満 | 95% 未満 |
| AI 誤判定率 | AI 判断のうち人間が訂正した割合 | 3% 超 | 5% 超 |
| 未対応リスク件数 | 期限超過の対応未完了リスク数 | 3 件 | 5 件 |
| 変更成功率 | 変更リクエストのうち問題なく完了した割合 | 95% 未満 | 90% 未満 |

## 3. リスク登録簿フォーマット

### 3.1 登録簿フィールド定義

| フィールド名 | データ型 | 必須 | 説明 |
|------------|---------|------|------|
| risk_id | STRING | Y | 一意識別子。形式: RISK-YYYY-NNN（例: RISK-2026-001） |
| title | STRING | Y | リスクの簡潔なタイトル |
| description | TEXT | Y | リスクの詳細説明（原因・事象・結果の形式で記述） |
| category | ENUM | Y | リスクカテゴリ（Section 4 参照） |
| sub_category | STRING | N | サブカテゴリ |
| likelihood | INTEGER | Y | 可能性スコア（1-5）。定義は RISK_SCORING_LOGIC.md 参照 |
| impact | INTEGER | Y | 影響度スコア（1-5）。定義は RISK_SCORING_LOGIC.md 参照 |
| score | INTEGER | Y | リスクスコア = likelihood x impact（自動算出） |
| risk_level | ENUM | Y | LOW / MEDIUM / HIGH / CRITICAL（score から自動算出） |
| status | ENUM | Y | IDENTIFIED / ANALYZING / TREATING / MONITORING / CLOSED |
| owner | STRING | Y | リスクオーナーの氏名・役職 |
| identified_date | DATE | Y | リスク識別日 |
| review_date | DATE | Y | 次回レビュー予定日 |
| treatment_strategy | ENUM | Y | AVOID / MITIGATE / TRANSFER / ACCEPT |
| mitigation_measures | TEXT | Y | 具体的な軽減措置の記述 |
| residual_likelihood | INTEGER | N | 軽減後の残留可能性スコア |
| residual_impact | INTEGER | N | 軽減後の残留影響度スコア |
| residual_score | INTEGER | N | 残留リスクスコア（自動算出） |
| kri | TEXT | N | 主要リスク指標（測定方法・閾値） |
| related_controls | TEXT | N | 関連する統制措置（ISO 20000 / J-SOX 統制 ID） |
| related_risks | TEXT | N | 関連するリスク ID（カンマ区切り） |
| evidence_ref | TEXT | N | 証拠・根拠資料への参照 |
| last_updated | DATE | Y | 最終更新日 |
| updated_by | STRING | Y | 最終更新者 |
| approved_by | STRING | N | 承認者（受容・クローズ時は必須） |
| notes | TEXT | N | 補足事項 |

### 3.2 リスク登録簿エントリサンプル

```yaml
risk_id: RISK-2026-001
title: AI エージェントの誤判定による不適切な変更承認
description: |
  Claude AI エージェントが変更リクエストのリスク評価を誤り、
  高リスク変更を低リスクと判定した場合、
  適切な承認プロセスをバイパスして変更が実施されるリスク。
category: AI_SPECIFIC
sub_category: 誤判定
likelihood: 3
impact: 4
score: 12
risk_level: MEDIUM
status: TREATING
owner: AI ガバナンス委員会
identified_date: 2026-03-02
review_date: 2026-06-01
treatment_strategy: MITIGATE
mitigation_measures: |
  1. AI 評価スコアが HIGH 以上の場合は必ず人間の承認を必須化
  2. AI 判断ログの全記録と定期的な精度検証（月次）
  3. 誤判定率 5% 超過時の自動アラートと AI モデル再評価
  4. 変更承認の最終権限は人間のみに付与
residual_likelihood: 2
residual_impact: 3
residual_score: 6
kri: AI 誤判定率 < 5%（月次計測）
related_controls: AI-GOV-001, CHANGE-CTL-005
```

## 4. リスクカテゴリ定義

### 4.1 技術リスク（Technical Risk）

| サブカテゴリ | 説明 | 例 |
|------------|------|-----|
| インフラ障害 | サーバー・ネットワーク・クラウドの障害 | PostgreSQL 障害・CPU リソース枯渇 |
| ソフトウェア品質 | バグ・脆弱性・設計欠陥 | SQL インジェクション・メモリリーク |
| 依存関係 | 外部ライブラリ・API の変更・廃止 | FastAPI バージョンアップ・GitHub API 変更 |
| スケーラビリティ | 負荷増大への対応不足 | 同時接続増加によるタイムアウト |
| データ品質 | データの不整合・破損・消失 | DB マイグレーション失敗・重複データ |

### 4.2 運用リスク（Operational Risk）

| サブカテゴリ | 説明 | 例 |
|------------|------|-----|
| プロセス欠陥 | 運用手順の不備・未整備 | インシデント対応手順の未文書化 |
| 人的ミス | オペレーターの操作ミス | 誤ったコマンド実行・本番 DB 削除 |
| ナレッジ消失 | 担当者の退職・異動 | 単一担当者依存の運用知識 |
| キャパシティ不足 | リソース・人員の不足 | オンコール担当者不在 |
| SLA 違反 | サービスレベル目標の未達成 | 可用性 99.5% 未達 |

### 4.3 セキュリティリスク（Security Risk）

| サブカテゴリ | 説明 | 例 |
|------------|------|-----|
| 不正アクセス | 認証・認可の突破 | API トークン漏洩・権限昇格 |
| データ漏洩 | 機密情報の外部流出 | 監査ログの外部公開 |
| 攻撃 | DDoS・SQLi・XSS 等 | DoS 攻撃によるサービス停止 |
| 内部脅威 | 内部者による不正行為 | 権限ユーザーの不正データ参照 |
| サプライチェーン | 外部依存関係を経由した攻撃 | OSS ライブラリへのバックドア混入 |

### 4.4 コンプライアンスリスク（Compliance Risk）

| サブカテゴリ | 説明 | 例 |
|------------|------|-----|
| J-SOX 違反 | IT 全般統制の不備 | アクセス管理記録の欠如 |
| ISO 20000 不適合 | サービスマネジメント要件の未充足 | 変更管理手順の不遵守 |
| 個人情報 | 個人情報保護法・GDPR 違反 | 個人データの目的外利用 |
| 契約違反 | SLA・NDA・ライセンス違反 | オープンソースライセンス不遵守 |
| 規制変更 | 法令・規格の改訂への未対応 | ITIL ガイダンス改訂への未対応 |

### 4.5 AI 固有リスク（AI-Specific Risk）

| サブカテゴリ | 説明 | 例 |
|------------|------|-----|
| 誤判定 | AI が誤った判断を下す | 高リスク変更を低リスクと評価 |
| バイアス | 学習データや設計に起因する偏り | 特定変更種別への過度な承認傾向 |
| 自律過多 | AI が人間の監督なしに重要決定を実施 | 緊急変更の無承認実行 |
| 説明不能 | AI の判断根拠が追跡・説明できない | 監査時に判断根拠を提示できない |
| モデル劣化 | 時間経過による AI 精度低下 | システム変化に伴う判断精度低下 |
| プロンプト汚染 | 悪意ある入力による AI 操作 | プロンプトインジェクション攻撃 |

## 5. AI 固有リスク詳細分析

### 5.1 誤判定リスク

**リスク説明**

Claude AI がリスク評価・変更承認・インシデント優先度付けにおいて
誤った判断を下し、不適切なアクションが実行される。

**根本原因**
- 訓練データと実際の環境の乖離
- プロンプト設計の不備
- コンテキスト情報の不足
- モデルの確率的動作による出力のばらつき

**軽減措置**
1. 重要判断には必ず人間の最終承認を義務付ける
2. AI 判断には信頼度スコアを付与し、低信頼度の場合は人間にエスカレーション
3. AI 判断ログを完全記録し、月次精度検証を実施
4. 誤判定率 KPI を設定し、閾値超過時の即時通知を実装

**受容不可条件**
- 本番環境への変更承認を AI のみで実施することは受容不可
- インシデントの重大度分類（P1/P2）を AI のみで最終決定することは受容不可

### 5.2 バイアスリスク

**リスク説明**

AI の判断に系統的な偏りが存在し、特定のリスクを過小評価または過大評価する。

**根本原因**
- プロンプトに含まれる暗黙の前提
- 過去の判断履歴に基づく学習バイアス
- 特定の変更種別・担当者への一貫した評価傾向

**軽減措置**
1. 四半期ごとに AI 判断パターンの統計的分析を実施
2. 変更種別・担当者別の承認率異常を監視
3. バイアス検出時はプロンプト見直しと AI ガバナンス委員会への報告

### 5.3 自律過多リスク

**リスク説明**

AI エージェントが設計意図を超えて自律的に行動し、
人間の監督が及ばない領域で重要な判断・実行を行う。

**根本原因**
- 権限設計の不備（AI への過剰な権限付与）
- 承認フローのバイパス経路の存在
- 緊急時の例外処理における監督の抜け穴

**軽減措置**
1. AI エージェントの権限は最小権限の原則に基づき設計
2. 本番環境への Write 権限は AI に付与しない（Read + Propose のみ）
3. すべての AI 実行アクションは監査ログに記録
4. 定期的な権限棚卸しを実施（四半期）

## 6. リスクオーナー制度

### 6.1 リスクオーナーの定義

リスクオーナーとは、特定リスクの識別・評価・対応・監視に責任を持つ個人または役割である。

### 6.2 リスクオーナーの責務

| 責務 | 頻度 | 成果物 |
|------|------|--------|
| リスク状態の定期評価 | 監視頻度に従う | リスク登録簿の更新 |
| 軽減措置の実施確認 | 月次 | 進捗報告 |
| KRI の測定・報告 | 月次 | KRI レポート |
| リスク変化の通知 | 随時 | リスク変更通知 |
| 四半期レビューへの参加 | 四半期 | レビュー参加記録 |
| 監査証拠の保全 | 随時 | 証拠管理 |

### 6.3 カテゴリ別リスクオーナー

| リスクカテゴリ | 主担当オーナー | 代理オーナー |
|------------|-------------|-----------|
| 技術リスク | 技術リード | シニアエンジニア |
| 運用リスク | 運用責任者 | 運用チームリード |
| セキュリティリスク | セキュリティ担当 | 技術リード |
| コンプライアンスリスク | コンプライアンス担当 | マネージャー |
| AI 固有リスク | AI ガバナンス委員会 | 技術リード |

### 6.4 リスクオーナーアサインメント基準

- リスクの主要な影響を受ける業務領域の責任者を優先
- 軽減措置の実施権限を持つ者をアサイン
- 1 つのリスクに対してオーナーは 1 名のみ（連名不可）
- AI 固有リスクは必ず AI ガバナンス委員会をオーナーとする

## 7. 四半期リスクレビュープロセス

### 7.1 実施スケジュール

| 四半期 | 実施時期 | 対象 |
|--------|---------|------|
| Q1 レビュー | 1 月最終週 | 全リスク登録簿 |
| Q2 レビュー | 4 月最終週 | 全リスク登録簿 |
| Q3 レビュー | 7 月最終週 | 全リスク登録簿 |
| Q4 レビュー | 10 月最終週 | 全リスク登録簿 + 年次総括 |

### 7.2 レビュープロセス

**準備フェーズ（レビュー 1 週間前）**
1. リスク登録簿の最新状態確認（自動レポート生成）
2. 前四半期 KRI データの集計
3. インシデント・変更記録の照合
4. 新規識別リスクの一次評価
5. リスクオーナーへの事前資料配布

**実施フェーズ（レビュー当日）**
1. KRI 達成状況のレビュー（30 分）
2. 重大・高リスクの個別レビュー（60 分）
3. 軽減措置の進捗確認（30 分）
4. 新規リスクの審議（30 分）
5. クローズ候補リスクの承認（15 分）
6. 次四半期の優先リスク決定（15 分）

**アウトプット（レビュー 1 週間後までに）**
- 四半期リスクレビューレポートの発行
- リスク登録簿の更新確定
- 次四半期の対応優先リスト
- リスク委員会への報告（重大リスクは役員報告）

### 7.3 参加者

| 役割 | 参加区分 |
|------|---------|
| リスク管理責任者 | 必須 |
| 全リスクオーナー | 必須 |
| 技術リード | 必須 |

| 外部監査担当（年次） | 任意 |

### 7.4 年次リスクアセスメント

Q4 レビューに加え、年次で以下を実施する。

- リスクカテゴリおよびスコアリング基準の見直し
- リスク許容基準の経営層による再承認
- 外部脅威環境の変化に基づくリスク登録簿の全面見直し
- AI リスク管理手法の最新化（ベストプラクティス取込み）
- 翌年のリスク管理計画策定
