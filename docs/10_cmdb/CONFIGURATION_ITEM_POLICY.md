## 1. 概要

### 1.1 目的

本ポリシーは、ServiceMatrix CMDBにおける構成アイテム（CI: Configuration Item）の管理基準を定義する。CIの登録・変更・廃棄に関する一貫した手順と責任を明確にし、CMDB情報の正確性・完全性・最新性を維持することを目的とする。

### 1.2 適用範囲

- 全ITサービスを支えるインフラ・アプリケーション・サービスコンポーネント
- ServiceMatrixで管理されるすべてのCI
- CMDB管理に関わるすべての役割と担当者

## 2. CI管理の原則

### 2.1 基本原則

1. **網羅性**: サービス提供に関わるすべてのコンポーネントをCIとして登録する
2. **正確性**: CI情報は常に最新かつ正確であること
3. **追跡可能性**: CIの変更履歴はすべて記録される
4. **一貫性**: CI命名規則・分類体系を統一する
5. **責任明確化**: すべてのCIにオーナーを割り当てる
6. **自動化優先**: 可能な限り自動検出・自動更新を活用する

### 2.2 CI管理の範囲

| 管理対象 | 管理対象外 |
|----------|-----------|
| 本番環境のすべてのハードウェア/ソフトウェア | 個人のPC・モバイルデバイス |
| ステージング環境の主要コンポーネント | 開発環境の一時的リソース |
| サービスに影響するネットワーク機器 | 物理的なオフィス設備 |
| ビジネスサービス/ITサービス定義 | 外部SaaSのサービス内部構成 |
| 統治ドキュメント（ドキュメントCI） | 一般的なオフィスドキュメント |
| 担当者/チーム情報（PersonCI） | 組織図全体 |

## 3. CI登録基準

### 3.1 CI管理対象の判定基準

以下の条件を**1つ以上**満たすコンポーネントはCIとして登録する：

| 判定基準 | 説明 | 例 |
|----------|------|-----|
| **サービス影響** | ITサービスの提供に直接影響する | 本番サーバー、データベース、ロードバランサー |
| **変更管理対象** | 変更時にRFC承認が必要 | アプリケーションコード、設定ファイル |
| **SLA影響** | SLA測定・報告に影響する | 監視対象システム、APIエンドポイント |
| **監査要件** | J-SOX・コンプライアンス要件で管理が必要 | 財務システム、認証基盤 |
| **依存関係** | 他のCIから依存されている | 共有ライブラリ、共通サービス |
| **コスト管理** | 資産として財務管理が必要 | ハードウェア（50万円以上） |

### 3.2 CI管理対象外の判定基準

以下は原則としてCIとして管理しない：

- 消耗品（トナー、ケーブル等）
- 単価5万円未満の小物機器
- 開発・テスト環境の一時的リソース（ただし本番昇格時は登録）
- 個人所有のデバイス（BYOD）
- ベンダー管理のSaaSサービス（接続エンドポイントのみ登録）

### 3.3 CIタイプ別登録基準

#### Server（サーバー系）

| 項目 | 基準 |
|------|------|
| 登録対象 | 本番・ステージング環境のすべてのサーバー |
| 除外対象 | 開発環境の一時インスタンス（90日以内） |
| 自動検出 | 必須（Ansible/Terraform連携） |
| 棚卸し頻度 | 月次 |

#### Application（アプリケーション系）

| 項目 | 基準 |
|------|------|
| 登録対象 | バージョン管理されているすべてのアプリケーション |
| 粒度 | マイクロサービス単位（モジュール単位ではない） |
| 自動検出 | CI/CDパイプライン連携で自動登録 |
| 棚卸し頻度 | デプロイ時自動更新 |

#### Network（ネットワーク機器）

| 項目 | 基準 |
|------|------|
| 登録対象 | ルーター、スイッチ、ファイアウォール、VPN装置 |
| 除外対象 | アクセスポイント（個別管理表で管理） |
| 自動検出 | SNMP/NetFlow連携 |
| 棚卸し頻度 | 四半期 |

#### Database（データベース系）

| 項目 | 基準 |
|------|------|
| 登録対象 | 本番・ステージングのすべてのDBインスタンス |
| 粒度 | インスタンス単位（スキーマ単位ではない） |
| 自動検出 | DBサービス検出エージェント |
| 棚卸し頻度 | 月次 |

## 4. CI命名規則

### 4.1 CI ID体系

```
CI-{TYPE}-{SEQ}
```

| 要素 | 説明 | 例 |
|------|------|-----|
| CI | 固定プレフィックス | CI |
| TYPE | CIタイプ略称（3文字） | SRV, APP, NET, SVC, DB, STR, MID, PER, DOC, ENV |
| SEQ | 連番（3〜6桁、ゼロ埋め） | 001, 000012 |

#### CIタイプ略称一覧

| タイプ | 略称 | 連番範囲 |
|--------|------|---------|
| Server | SRV | 001-999 |
| Application | APP | 001-999 |
| Network | NET | 001-999 |
| Service | SVC | 001-999 |
| Database | DB | 001-999 |
| Storage | STR | 001-999 |
| Middleware | MID | 001-999 |
| Person | PER | 001-999 |
| Document | DOC | 001-999 |
| Environment | ENV | 001-099 |

### 4.2 CI名称規則

| ルール | 例 |
|--------|-----|
| 環境 + 役割 + 連番（サーバー） | prod-web-01, stg-api-01 |
| サービス名 + バージョン（アプリ） | ServiceMatrix App v2.1 |
| ロケーション + 種別 + 連番（ネットワーク） | dc1-fw-01, dc1-lb-01 |
| ビジネス名称（サービス） | Email Service, CRM Service |

### 4.3 命名禁止事項

- 個人名・担当者名の使用禁止
- 日本語文字の使用禁止（英数字・ハイフンのみ）
- 40文字超の名前禁止
- 同一名前の重複登録禁止（システムで自動チェック）

## 5. CI属性必須項目リスト

### 5.1 全CIタイプ共通必須項目

| フィールド名 | 型 | 説明 | バリデーション |
|-------------|-----|------|--------------|
| `ci_id` | VARCHAR | CI ID（命名規則準拠） | パターン検証必須 |
| `ci_type` | ENUM | CIタイプ | 定義済み値のみ |
| `name` | VARCHAR(200) | CI名称 | 最小3文字 |
| `status` | ENUM | ステータス | 定義済みライフサイクル値 |
| `owner_id` | UUID | 責任者ID | 実在ユーザー/チームへの参照 |
| `environment` | ENUM | 環境（Production/Staging/Development/DR） | 定義済み値のみ |
| `criticality` | ENUM | 重要度（Critical/High/Medium/Low） | 定義済み値のみ |
| `registered_by` | UUID | 登録者ID | 実在ユーザーへの参照 |

### 5.2 CI重要度分類

| 重要度 | ビジネス影響 | SLA対応 | 冗長構成要件 | 例 |
|--------|------------|---------|------------|-----|
| Critical | サービス全体に影響、事業停止リスク | P1 SLA適用 | 必須（冗長構成 + DR） | 本番DBプライマリ、認証基盤 |
| High | 主要機能に影響、業務遅延 | P2 SLA適用 | 推奨（冗長構成） | APIサーバー、ロードバランサ |
| Medium | 一部機能に影響、代替手段あり | P3 SLA適用 | 任意 | バッチサーバー、監視ツール |
| Low | 利便性への影響のみ | P4 SLA適用 | 不要 | 開発ツール、ドキュメントサーバー |

### 5.3 タイプ別必須属性（JSONB）

#### Server 必須属性

```json
{
  "ip_address": "string（CIDR形式）",
  "os_type": "string（例: RHEL9, Ubuntu22.04）",
  "os_version": "string",
  "cpu_cores": "integer",
  "memory_gb": "integer",
  "storage_gb": "integer",
  "datacenter": "string"
}
```

#### Application 必須属性

```json
{
  "version": "string（SemVer形式）",
  "repository_url": "string（GitHubリポジトリURL）",
  "language_stack": "string（例: Python3.12+FastAPI）",
  "health_check_url": "string（URL形式）"
}
```

#### Database 必須属性

```json
{
  "db_type": "string（例: PostgreSQL16）",
  "version": "string",
  "recovery_time_objective_hours": "integer",
  "recovery_point_objective_hours": "integer"
}
```

## 6. CI登録プロセス

### 6.1 登録フロー

```
[CI登録要求]
    ↓
[CI登録フォーム記入] → バリデーション（命名規則・必須項目）
    ↓
[影響分析] → 既存CIとの重複確認・依存関係確認
    ↓
[承認判定]
    ├─ Critical → ProcessOwner + CMDB Manager承認（4時間SLA）
    ├─ High → ProcessOwner承認（8時間SLA）
    ├─ Medium → TeamLead承認（24時間SLA）
    └─ Low → 自動承認（即時）
    ↓
[CMDB登録] → 自動通知（関係チーム）
    ↓
[初回棚卸し実施] → 7日以内
```

### 6.2 CI登録要求テンプレート

```markdown
# CI登録要求

## 基本情報
| 項目 | 値 |
|------|-----|
| CI名称 | |

| 環境 | [Production/Staging/Development/DR] |
| 重要度 | [Critical/High/Medium/Low] |
| オーナー | |
| 説明 | |

## タイプ別属性
（CIタイプに応じた属性を記入）

## 関係性
| 対象CI | 関係種別 | 説明 |
|--------|---------|------|
| | | |

## 登録理由
（新規導入、既存の未登録CI発見 等）

Labels: `cmdb/registration`
```

### 6.3 承認基準チェックリスト

| チェック項目 | 基準 |
|-------------|------|
| CI IDの一意性 | 既存CIと重複しないこと |
| 必須属性の完全性 | すべての必須属性が入力されていること |
| 命名規則の遵守 | 命名規則に準拠していること |
| オーナーの有効性 | 有効なユーザー/チームであること |
| 関係性の妥当性 | 参照先CIが存在し、関係種別が適切であること |
| 重要度の妥当性 | ビジネス影響に基づく適切な重要度であること |

## 7. CI変更プロセス

### 7.1 変更の分類

| 変更カテゴリ | 説明 | 承認要否 |
|-------------|------|---------|
| 属性更新（軽微） | バージョン更新、タグ変更 | CMDB管理者承認 |
| 属性更新（重要） | オーナー変更、重要度変更 | サービスマネージャー承認 |
| ステータス変更 | Active → Retired 等 | 変更管理プロセス準拠 |
| 関係性変更 | 新規依存関係の追加/削除 | CMDB管理者承認 |
| CI削除 | CMDBからのCI削除 | サービスマネージャー承認 |

### 7.2 変更記録必須事項

すべてのCI変更は以下の情報とともに `ci_history` テーブルに自動記録される。

| 記録項目 | 説明 |
|----------|------|
| history_id | 変更履歴ID（自動採番） |
| ci_id | 対象CI ID |
| change_type | 変更種別（create/update/delete/status_change） |
| field_name | 変更フィールド名 |
| old_value | 変更前の値 |
| new_value | 変更後の値 |
| changed_by | 変更者（user/agent） |
| changed_at | 変更日時（自動記録） |
| change_reason | 変更理由（必須・最小30文字） |
| related_issue | 関連GitHub Issue番号 |

## 8. CI棚卸しプロセス

### 8.1 棚卸しスケジュール

| 対象 | 頻度 | 方法 |
|------|------|------|
| Critical CI | 月次 | オーナー確認 + 自動属性チェック |
| High CI | 四半期 | オーナー確認 + サンプリング検証 |
| Medium/Low CI | 半期 | 自動チェック + 異常時のみ手動確認 |
| 全CI | 年次 | 全数棚卸し |

### 8.2 棚卸しチェックリスト

| チェック項目 | 確認内容 |
|-------------|---------|
| 存在確認 | CIが物理的/論理的に存在するか |
| ステータス確認 | CMDBのステータスが実態と一致するか |
| 属性確認 | IPアドレス、バージョン等が最新か |
| オーナー確認 | オーナーが在籍・担当しているか |
| 関係性確認 | 依存関係が実態と一致するか |
| 重要度確認 | ビジネス影響に応じた適切な重要度か |

### 8.3 自動検出連携

| ソース | 検出対象 | 連携方式 |
|--------|---------|---------|
| Terraform State | インフラCI（Server, Network） | Webhook + API |
| GitHub Actions | アプリケーションCI（Application） | CI/CDパイプライン |
| Ansible Inventory | サーバーCI（Server） | Playbook後処理 |
| Kubernetes API | コンテナ・サービスCI | カスタムオペレーター |
| AWS Config / GCP Asset | クラウドリソースCI | APIポーリング（15分間隔） |

## 9. CI廃止プロセス

### 9.1 廃止条件

以下のいずれかに該当する場合、CIの廃止を検討する。

| 条件 | 説明 |
|------|------|
| サービス終了 | CIが関連するサービスが終了した |
| リプレース | 後継CIに置き換えられた |
| 不要判定 | 棚卸しで不要と判定された |
| 障害廃止 | 修復不能な障害により使用不可 |

### 9.2 廃止フロー

```
[廃止候補の特定]
    ↓
[依存関係の確認] → 依存CIが存在する場合は廃止不可
    ↓
[廃止影響分析]
    ↓
[廃止申請Issue作成]
    ↓
[サービスマネージャー承認]
    ↓（承認後）
[ステータス: Retired に変更]
    ↓
[関係性の更新/削除]
    ↓（90日後）
[ステータス: Disposed に変更]
```

### 9.3 廃止後のデータ保持

| データ種別 | 保持期間 |
|-----------|---------|
| CI基本情報 | 廃止後3年 |
| CI変更履歴 | 廃止後7年（J-SOX監査要件） |
| 関係性情報 | 廃止後1年 |

## 10. 不正CI検出と対応手順

### 10.1 不正CI定義

| 分類 | 説明 |
|------|------|
| **未登録CI** | 実環境に存在するがCMDBに未登録のCI |
| **ゴーストCI** | CMDBに登録されているが実環境に存在しないCI |
| **属性不整合CI** | 実環境の情報とCMDB情報が著しく異なるCI |
| **無承認変更CI** | 承認なしに変更されたCI（変更管理回避） |
| **ポリシー違反CI** | 命名規則・必須項目・承認プロセス違反のCI |

### 10.2 検出方法

```
定期自動スキャン（日次）
    ├─ 実環境探索（Nmap/Ansible/クラウドAPI）
    ├─ CMDB照合
    └─ 差異リスト生成

変更イベント検出（リアルタイム）
    ├─ IaC変更検出（Terraform plan）
    ├─ デプロイイベント（CI/CD）
    └─ ネットワーク変更（SNMP）
```

### 10.3 不正CI対応手順

```
[不正CI検出]
    ↓
[重大度分類]
    ├─ CRITICAL: 本番環境の未登録CI・無承認変更 → 2時間以内対応
    ├─ HIGH: セキュリティCI不整合 → 4時間以内対応
    ├─ MEDIUM: その他のCI不整合 → 24時間以内対応
    └─ LOW: 軽微な属性不整合 → 72時間以内対応
    ↓
[根本原因調査]
    ├─ 未登録: 緊急登録フロー
    ├─ ゴースト: 廃棄確認・CMDB更新
    ├─ 属性不整合: 正確な情報で更新
    ├─ 無承認変更: インシデント起票・変更管理回収
    └─ ポリシー違反: 是正措置・再発防止策
    ↓
[是正実施（承認必須）]
    ↓
[再スキャンによる解消確認]
    ↓
[インシデントクローズ + 再発防止記録]
```

### 10.4 不正CI KPI

| KPI | 目標値 | 測定頻度 |
|-----|--------|---------|
| 未登録CI検出率 | 0%（ゼロトレランス） | 週次 |
| CMDB精度 | 95%以上 | 月次 |
| 不正CI平均解消時間（Critical） | 2時間以内 | 月次 |
| 無承認変更発生件数 | 月3件以下 | 月次 |
| 棚卸し完了率 | 100% | 四半期 |

## 11. CMDB管理体制

### 11.1 役割と責任

| 役割 | 責任 |
|------|------|
| CMDB管理者 | CMDB全体の管理・品質保証・アクセス制御 |
| CIオーナー | 担当CIの情報正確性の維持 |
| サービスマネージャー | サービスCIの重要度・関係性の承認 |
| 変更マネージャー | CI変更の変更管理プロセス連携 |
| 監査担当 | CMDB棚卸しの実施・監査レポート作成 |

### 11.2 アクセス制御

| 操作 | CMDB管理者 | CIオーナー | サービスマネージャー | 運用チーム | 一般ユーザー |
|------|-----------|-----------|-------------------|-----------|------------|
| CI参照 | Yes | Yes | Yes | Yes | Yes（公開CIのみ） |
| CI登録 | Yes | 申請のみ | 申請のみ | 申請のみ | No |
| CI属性変更 | Yes | 担当CIのみ | 担当サービスCIのみ | No | No |
| CI削除 | Yes | No | 申請のみ | No | No |
| 関係性変更 | Yes | 担当CIのみ | No | No | No |
| レポート参照 | Yes | Yes | Yes | Yes | No |

## 12. コンプライアンス要件

### 12.1 J-SOX対応

- 財務報告に影響するCIは必ず `criticality: Critical` で登録
- すべてのCI変更は変更理由とRFC番号を記録
- 四半期棚卸し結果は内部監査部門へ提出
- CI廃棄記録は最低7年間保管

### 12.2 ISO/IEC 20000対応

- CMDB精度95%以上の維持（月次測定）
- 変更プロセスとCMDB更新の連携を確保
- CIライフサイクル全体の追跡可能性を確保

### 12.3 ITIL 4対応

- Configuration Management Practice準拠
- CIの正確性・完全性・最新性の維持
- サービス影響分析への活用

## 13. ポリシー違反への対応

| 違反内容 | 対応 |
|---------|------|
| 命名規則違反 | システムによる登録拒否 + 是正依頼通知 |
| 必須項目未入力 | システムによる保存拒否 |
| 無承認変更 | 変更の自動ロールバック + インシデント起票 |
| 棚卸し未実施 | CMDB Managerへのエスカレーション + ProcessOwner報告 |
| 不正CI放置（期限超過） | 自動隔離 + 情報セキュリティへのエスカレーション |
