# 優先度マトリックス定義

ServiceMatrix Priority Matrix Definition

Version: 1.0
Status: Active
Owner: Service Operations Authority
Classification: ITIL 4 Aligned
準拠規格: ITIL 4 / ISO/IEC 20000 / J-SOX

---

## 1. 目的と適用範囲

### 1.1 目的

本ドキュメントは ServiceMatrix における優先度定義、影響度・緊急度マトリックス、
および各プロセスにおける優先度決定ルールを定義する。
すべての担当者・AI Agent が一貫した基準でチケットを優先度付けし、
SLA 遵守と資源の最適配分を実現することを目的とする。

### 1.2 適用範囲

- インシデント管理における優先度決定
- 問題管理における優先度決定
- 変更リクエストにおける優先度決定
- サービスリクエストにおける優先度決定
- AI Agent による自動優先度算出

### 1.3 優先度の定義原則

優先度は以下の2軸の組合せで決定する：

- **影響度（Impact）**: サービスまたはビジネスに対する現在の影響の大きさ
- **緊急度（Urgency）**: 対応を開始または完了しなければならない速度

> 優先度 = f（影響度, 緊急度）

## 2. 優先度定義

### 2.1 優先度レベル一覧

| 優先度 | ラベル | 色 | 概要 |
|--------|-------|----|------|
| P1 | Critical | 赤 | 重大なサービス停止またはビジネスへの致命的影響。即時対応必須 |
| P2 | High | オレンジ | 主要サービスの重大な機能低下。緊急対応必須 |
| P3 | Medium | 黄 | サービスの部分的な機能低下。計画的対応 |
| P4 | Low | 緑 | 軽微な影響または将来のリスク。通常スケジュールで対応 |

### 2.2 各優先度の詳細定義

#### P1 - Critical（致命的）

```
定義: 本番環境での完全なサービス停止、またはビジネスクリティカルな
      機能が完全に利用不能な状態。回避策が存在しない。

典型例:
  - 本番 API の全面的な応答不能
  - データベースの完全停止
  - 認証基盤の障害（全ユーザーのアクセス不能）
  - セキュリティ侵害（データ漏洩・不正アクセス）
  - 法的コンプライアンス違反の確定
  - 財務システムの完全停止

影響:
  - 全ユーザーまたは50%以上のユーザーがサービス利用不能
  - 重大な財務損失が発生中または差し迫っている
  - 監査対応や法的報告に直接影響
```

#### P2 - High（重大）

```
定義: 主要なビジネス機能が重大に低下しているが、部分的な回避策が
      存在する状態。SLA 達成が危機的状況。

典型例:
  - 本番 API のレスポンスタイム著しい劣化（SLA 超過）
  - インシデント管理機能の主要部分が利用不能
  - 通知・アラート機能の完全停止
  - バックアップ/フェイルオーバーの失敗
  - パフォーマンスの大幅な低下（通常の3倍以上のレスポンス時間）

影響:
  - 20〜49% のユーザーが影響を受けている
  - ビジネス機能が著しく低下しているが、一部継続可能
  - SLA 違反のリスクが高い（残時間25%以下）
```

#### P3 - Medium（中程度）

```
定義: 一部のユーザーまたは機能に影響があるが、ビジネスの主要機能は
      継続可能。十分な回避策が存在する状態。

典型例:
  - 特定の機能やレポートが利用不能
  - パフォーマンスの軽度な低下
  - 一部ユーザーへの機能制限
  - 非クリティカルな自動化の停止
  - UI/UXの軽微な問題

影響:
  - 5〜19% のユーザーが影響を受けている
  - ビジネス機能は継続しているが、効率が低下している
  - 回避策が存在し、業務継続が可能
```

#### P4 - Low（低）

```
定義: ほとんど影響のない軽微な問題、または将来のリスクとなり得る
      軽微な不具合。

典型例:
  - ドキュメントの誤り
  - UI の軽微な表示問題
  - 将来的な機能改善要求
  - パフォーマンスの微小な劣化（SLA 範囲内）
  - 設定の最適化提案
  - ナレッジベースの更新依頼

影響:
  - 5% 未満のユーザーへの影響、または影響なし
  - ビジネス機能への実質的な影響なし
  - 将来のリスクとして記録・管理
```

## 3. 影響度 × 緊急度 マトリックス

### 3.1 影響度定義

| 影響度 | レベル | 基準 |
|--------|-------|------|
| 高（High） | 3 | 全ユーザー影響 / ビジネスクリティカル機能停止 / 財務的重大損失 |
| 中（Medium） | 2 | 一部ユーザー影響 / 主要機能低下 / 中程度の財務影響 |
| 低（Low） | 1 | 軽微な影響 / 非クリティカル機能の問題 / 財務影響軽微 |

### 3.2 緊急度定義

| 緊急度 | レベル | 基準 |
|--------|-------|------|
| 高（High） | 3 | 即時対応必須 / SLA 違反が差し迫っている / ビジネス停止中 |
| 中（Medium） | 2 | 数時間以内に対応が必要 / SLA 違反リスクあり |
| 低（Low） | 1 | 通常スケジュールで対応可能 / SLA に余裕がある |

### 3.3 優先度決定マトリックス

| 影響度 ↓ / 緊急度 → | 高（Urgency: High） | 中（Urgency: Medium） | 低（Urgency: Low） |
|---------------------|--------------------|-----------------------|-------------------|
| **高（Impact: High）** | **P1 - Critical** | **P2 - High** | **P2 - High** |
| **中（Impact: Medium）** | **P2 - High** | **P3 - Medium** | **P3 - Medium** |
| **低（Impact: Low）** | **P3 - Medium** | **P3 - Medium** | **P4 - Low** |

### 3.4 マトリックス適用例

| シナリオ | 影響度 | 緊急度 | 優先度 |
|---------|--------|-------|--------|
| 本番 API 全面停止 | 高 | 高 | P1 |
| 重要レポート機能の停止（月次締め前日） | 中 | 高 | P2 |
| バックグラウンドジョブの失敗（即日影響なし） | 低 | 中 | P3 |
| UI 表示の軽微な崩れ | 低 | 低 | P4 |
| 認証エラー（一部ユーザー） | 中 | 高 | P2 |
| セキュリティ脆弱性の検出（未悪用） | 高 | 中 | P2 |
| 本番 DB の完全損失リスク | 高 | 高 | P1 |
| ドキュメント更新要求 | 低 | 低 | P4 |

## 4. インシデント優先度決定ルール

### 4.1 基本ルール

1. インシデント登録時、担当者または AI Agent が影響度・緊急度を評価する
2. マトリックスに従い優先度を決定する
3. 優先度は状況の変化に応じて変更できる（変更理由の記録必須）
4. P1/P2 の優先度引き下げには、サービスマネージャーの承認が必要

### 4.2 自動優先度上昇ルール

以下の条件を満たした場合、AI Agent は優先度を自動的に上昇させ、承認を促す：

| 条件 | 現優先度 | 提案優先度 |
|------|---------|-----------|
| SLA 残時間が25%以下になった | P3 | P2 |
| SLA 残時間が10%以下になった | P2 | P1 |
| 影響ユーザー数が当初見積もりを50%超えた | P2/P3 | 1段階上昇 |
| セキュリティ侵害の可能性が判明した | 任意 | P1 |
| 関連インシデントが3件以上発生した | P3 | P2 |
| 重要顧客からのエスカレーションがあった | P3/P4 | P2 |

### 4.3 インシデント優先度変更記録

優先度を変更する場合、以下を記録する：

```
## 優先度変更記録
- 変更日時: {ISO 8601}
- 変更前優先度: {P1|P2|P3|P4}
- 変更後優先度: {P1|P2|P3|P4}
- 変更理由: {具体的な理由}
- 変更者: {担当者名またはAI Agent}
- 承認者: {承認者名（P1/P2の引き下げ時のみ必須）}
```

## 5. 変更リクエスト優先度ルール

### 5.1 変更優先度定義

変更リクエストにおける優先度は、リスク・緊急性・ビジネス価値を考慮して決定する。

| 優先度 | 適用基準 | リードタイム |
|--------|---------|------------|
| P1 - Emergency Change | セキュリティ侵害の修復、本番障害の緊急修正 | 即時〜4時間 |
| P2 - Urgent Change | SLA 違反防止のための変更、重大リスクの軽減 | 24時間以内 |
| P3 - Normal Change | 計画的な機能追加・改善 | 通常リードタイム（1〜4週間） |
| P4 - Low Change | 軽微な改善、将来的な最適化 | 次回定期リリース |

### 5.2 緊急変更（Emergency Change）の条件

以下のすべてを満たす場合のみ緊急変更として処理する：

1. 本番インシデント P1/P2 の解消に直接必要な変更であること
2. 通常の変更プロセスを待つ時間的余裕がないこと
3. 変更の影響範囲が明確であること
4. ロールバック手順が確立されていること
5. ECC（緊急変更委員会）の口頭承認が得られていること（事後文書化可）

## 6. SLA 目標時間

### 6.1 インシデント SLA 目標

| 優先度 | 初回応答時間 | 担当者アサイン | 解決目標時間 | エスカレーション閾値 |
|--------|------------|-------------|------------|------------------|
| **P1 - Critical** | 5分以内 | 15分以内 | 1時間以内 | 30分（SLA残50%） |
| **P2 - High** | 15分以内 | 30分以内 | 4時間以内 | 2時間（SLA残50%） |
| **P3 - Medium** | 1時間以内 | 2時間以内 | 24時間以内 | 12時間（SLA残50%） |
| **P4 - Low** | 4時間以内 | 8時間以内 | 72時間以内 | 36時間（SLA残50%） |

> 「初回応答」= 担当者が GitHub Issue にコメントを追加、または状態を更新した時刻
> 「解決目標」= 営業時間ベース（P1 は24時間365日）

### 6.2 変更リクエスト SLA 目標

| 優先度 | 受付確認 | 影響評価完了 | 承認判定 | 実施完了 |
|--------|---------|-----------|---------|---------|
| **P1 - Emergency** | 15分以内 | 1時間以内 | 2時間以内 | 4時間以内 |
| **P2 - Urgent** | 1時間以内 | 4時間以内 | 8時間以内 | 24時間以内 |
| **P3 - Normal** | 4時間以内 | 2営業日以内 | 5営業日以内 | スケジュール日 |
| **P4 - Low** | 1営業日以内 | 1週間以内 | 2週間以内 | 次回定期リリース |

### 6.3 サービスリクエスト SLA 目標

| 優先度 | 受付確認 | 処理開始 | 完了目標 |
|--------|---------|---------|---------|
| 緊急（パスワードリセット等） | 15分以内 | 即時 | 30分以内 |
| 標準（アカウント作成等） | 2時間以内 | 4時間以内 | 1営業日以内 |
| 計画（環境構築等） | 4時間以内 | 1営業日以内 | 3営業日以内 |

## 7. AI 支援優先度自動算出ロジック

### 7.1 算出フロー

```
インシデント・チケット登録
        |
        v
  AI Agent: テキスト解析
  - キーワード抽出（停止, 全滅, 緊急, etc.）
  - 影響サービス特定
  - エラーコード/ログ解析
        |
        v
  影響度スコア算出 (0-10)
  ┌─────────────────────────────────────┐
  │ 要素1: 影響ユーザー数推定           │
  │   全体の50%以上 → +4点             │
  │   全体の20-49% → +3点             │
  │   全体の5-19%  → +2点             │
  │   全体の5%未満 → +1点             │
  │                                    │
  │ 要素2: サービスクリティカル度        │
  │   コアサービス停止   → +3点        │
  │   主要機能低下       → +2点        │
  │   補助機能低下       → +1点        │
  │                                    │
  │ 要素3: 財務/コンプライアンス影響    │
  │   法的違反・重大財務損失 → +3点    │
  │   中程度の財務影響      → +2点    │
  │   軽微な財務影響        → +1点    │
  └─────────────────────────────────────┘
        |
        v
  緊急度スコア算出 (0-10)
  ┌─────────────────────────────────────┐
  │ 要素1: SLA 残時間比率               │
  │   残25%以下  → +4点               │
  │   残50%以下  → +3点               │
  │   残75%以下  → +2点               │
  │   残75%以上  → +1点               │
  │                                    │
  │ 要素2: ビジネスコンテキスト         │
  │   重要期間（締め処理・監査中）→+3点 │
  │   通常業務への直接影響中   → +2点  │
  │   影響が潜在的         → +1点     │
  │                                    │
  │ 要素3: 拡散リスク                  │
  │   拡大傾向が確認された → +3点      │
  │   拡大の可能性がある  → +2点      │
  │   拡大リスク低        → +1点      │
  └─────────────────────────────────────┘
        |
        v
  優先度マッピング
  ┌─────────────────────────────────────┐
  │ 影響度スコア ≥ 7 AND 緊急度 ≥ 7 → P1 │
  │ 影響度スコア ≥ 5 OR  緊急度 ≥ 7 → P2 │
  │ 影響度スコア ≥ 3 OR  緊急度 ≥ 4 → P3 │
  │ その他                          → P4 │
  └─────────────────────────────────────┘
        |
        v
  担当者への提案（承認必須）
  ※ 自動確定はしない（P1/P2の場合は自動通知も実施）
```

### 7.2 AI 算出精度の監視

| 指標 | 目標値 | 計測頻度 |
|------|--------|---------|
| 優先度提案の正解率 | 85% 以上 | 月次 |
| P1/P2 の見落とし率 | 1% 以下 | 月次 |
| 過大評価率（P1提案が実際P3） | 10% 以下 | 月次 |

### 7.3 AI 提案の人間確認フロー

```
AI 優先度提案
    |
    v
担当者レビュー（必須）
    |
    +-- P1/P2 提案: 5分以内に確認または修正
    |
    +-- P3/P4 提案: 30分以内に確認または修正
    |
    v
確定優先度が AI 提案と異なる場合:
    → 理由を記録（AI 学習のためのフィードバック）
    → 月次で AI 算出ロジックを見直し
```

## 8. 優先度に関する例外処理

### 8.1 VIP・重要顧客の扱い

重要顧客からの問合せは、通常より1段階優先度を高く設定することができる。ただし：

- 対象顧客リストは Service Operations Authority が管理する
- 対象顧客リストは四半期ごとにレビューする
- 優先度引き上げの理由は必ずチケットに記録する

### 8.2 重複インシデントの優先度

同一事象に起因する複数のインシデントが登録された場合：

1. 最も高い優先度を親インシデントの優先度とする
2. 重複インシデントは親にリンクしてクローズする
3. SLA タイマーは最初に登録されたインシデントの時刻から計測する

### 8.3 優先度変更の承認

| 変更方向 | 承認要件 |
|---------|---------|
| P4 → P3 | 担当者権限で変更可（理由記録必須） |
| P3 → P2 | チームリード承認 |
| P2 → P1 | サービスマネージャー承認 |
| P1 → P2 | サービスマネージャー承認 + 解決確認 |
| P2 → P3 | チームリード承認 + 状況説明 |
| P3 → P4 | 担当者権限で変更可（理由記録必須） |

## 9. KPI とメトリクス

### 9.1 優先度管理 KPI

| KPI | 目標値 | 計測頻度 |
|-----|--------|---------|
| P1 SLA 遵守率 | 99% 以上 | 月次 |
| P2 SLA 遵守率 | 95% 以上 | 月次 |
| P3 SLA 遵守率 | 90% 以上 | 月次 |
| P4 SLA 遵守率 | 80% 以上 | 月次 |
| 優先度変更頻度（平均） | 1件あたり1回以下 | 月次 |
| AI 優先度提案採用率 | 80% 以上 | 月次 |

### 9.2 優先度分布の監視

健全な優先度分布の目安（インシデント全体に対する比率）：

| 優先度 | 健全な比率 | 要注意レベル |
|--------|----------|------------|
| P1 | 2% 以下 | 5% 超過で根本原因調査 |
| P2 | 10% 以下 | 20% 超過で問題管理強化 |
| P3 | 50〜70% | - |
| P4 | 20〜40% | - |

P1/P2 の割合が継続的に高い場合は、サービス品質の根本的な問題として問題管理プロセスに引き渡す。
