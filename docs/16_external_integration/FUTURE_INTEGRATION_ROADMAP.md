# 将来統合ロードマップ

ServiceMatrix Future Integration Roadmap

Version: 1.0
Status: Active
Classification: Internal Strategic Document
Last Updated: 2026-03-02

---

## 1. 概要

本ドキュメントは、ServiceMatrix の将来的な外部システム統合ロードマップを定義する。
現在の統合基盤（GitHub中心）を出発点として、段階的に統合範囲を拡大し、
エンタープライズ ITSM プラットフォームとしての機能を強化する計画を示す。

---

## 2. ロードマップ概観

### 2.1 フェーズ別計画

| フェーズ | 期間 | テーマ | 目標 |
|---------|------|--------|------|
| Phase 1 | 2026 Q1-Q2 | コア統合確立 | GitHub + 通知基盤の安定稼働 |
| Phase 2 | 2026 Q3-Q4 | 監視・運用統合 | Datadog/PagerDuty 連携完成 |
| Phase 3 | 2027 Q1-Q2 | エンタープライズ統合 | LDAP/SSO/CMDB 外部連携 |
| Phase 4 | 2027 Q3-Q4 | AI 強化統合 | AI 自律度向上・分析統合 |
| Phase 5 | 2028 以降 | エコシステム拡張 | マーケットプレイス統合 |

## 4. Phase 2: 監視・運用統合（2026 Q3-Q4）

### 4.1 APM・監視統合

| システム | 統合内容 | 期待効果 |
|---------|---------|---------|
| Datadog | APM・ログ・メトリクスの統合ビュー | 障害検知から ITSM 自動連携 |
| Prometheus/Grafana | カスタムメトリクスのダッシュボード統合 | SLA リアルタイム監視 |
| Sentry | エラートラッキングからインシデント自動作成 | MTTR 短縮 |

```
Datadog → ServiceMatrix 統合フロー:
  1. Datadog がアラート発火
  2. ServiceMatrix Webhook エンドポイントに通知
  3. アラート内容を解析してインシデント自動作成
  4. 優先度を Datadog 深刻度から自動マッピング
  5. 担当者への自動アサイン
  6. SLA タイマー開始
```

### 4.2 インシデント対応統合

| システム | 統合内容 | 期待効果 |
|---------|---------|---------|
| PagerDuty | P1 インシデントのオンコール自動呼び出し | 24/7 対応体制 |
| OpsGenie | アラートグルーピング・エスカレーション | ノイズ削減 |
| Statuspage | サービスステータスの自動公開 | ユーザー透明性向上 |

## 5. Phase 3: エンタープライズ統合（2027 Q1-Q2）

### 5.1 アイデンティティ・認証統合

| システム | 統合内容 | 優先度 |
|---------|---------|--------|
| Active Directory / LDAP | ユーザー認証・グループ同期 | P1 |
| SAML 2.0 / OIDC SSO | シングルサインオン対応 | P1 |
| Azure AD | Microsoft エコシステム統合 | P2 |
| Okta | クラウド IAM 統合 | P2 |

```
SSO 統合設計（SAML 2.0）:
  IdP: Active Directory Federation Services (ADFS)
  SP: ServiceMatrix
  属性マッピング:
    email → user.email
    displayName → user.name
    groups → user.roles (RBAC マッピング)
  セッション: JWT + リフレッシュトークン（最大 8 時間）
```

## 6. Phase 4: AI 強化統合（2027 Q3-Q4）

### 6.1 AI 分析・予測統合

| 機能 | 説明 | 技術スタック |
|------|------|------------|
| 予測的インシデント検知 | メトリクス異常を事前検知 | Datadog ML + Claude API |
| 自動根本原因分析 | ログ・トレースから RCA を自動生成 | LLM + Graph Analysis |
| 変更リスク自動評価 | 過去の変更データから AI がリスク評価 | ML Model + Claude API |
| SLA 違反予測 | SLA 違反を事前に予測・警告 | 時系列予測モデル |

### 6.2 自動化レベルの段階的向上

```
AI 自律度進化ロードマップ:
  現在（L1）:
    → AI は提案のみ、すべて人間が承認・実行

  2027 Q1（L2）:
    → 低リスク変更の自動実行（設定変更・スケール調整）
    → 条件: リスクスコア 20 以下 + テスト成功 + 夜間バッチ時間帯

  2027 Q3（L3）:
    → 中リスク変更の条件付き自動実行
    → 条件: 人間のレビュー承認 + ロールバック計画確認済み

  2028（L4）:
    → SRE 型の自律的インシデント対応
    → 条件: 完全な AI 統治フレームワーク確立後
```

### 6.3 LLM 統合強化

| 機能 | Claude API 活用方法 | 統治要件 |
|------|-------------------|---------|
| インシデント要約 | 長文ログから要約生成 | L1 自律（人間レビュー必須）|
| 変更レビュー支援 | PR コードの影響分析 | L1 自律 |
| SOP 自動生成 | インシデント対応手順の自動生成 | L2 自律 |
| ポストモーテム生成 | インシデントデータから自動生成 | L2 自律 |

## 7. Phase 5: エコシステム拡張（2028 以降）

サードパーティが統合を開発・公開できるエコシステムを構築する。

| 機能 | 説明 |
|------|------|
| Integration SDK | 外部開発者向け統合開発キット |
| Webhook API 公開 | ServiceMatrix イベントの外部購読 |
| パートナーポータル | 統合の申請・審査・公開 |
| 統合マーケットプレイス | コミュニティ統合のディレクトリ |

### 7.2 業界標準プロトコル対応

| プロトコル | 用途 | 想定時期 |
|----------|------|---------|
| CloudEvents | イベント標準フォーマット対応 | 2027 |
| OpenTelemetry | 分散トレーシング標準対応 | 2027 |
| AsyncAPI | 非同期 API 仕様の標準化 | 2028 |
| ITIL 4 Practice API | ITSM 標準 API スキーマ | 2028 |

## 8. 技術的考慮事項

### 8.1 統合アーキテクチャの進化

| 現在 | 将来 |
|------|------|
| 同期 REST API 中心 | 非同期イベントドリブン中心 |
| GitHub Webhook のみ | マルチソース Webhook 対応 |
| ポイントツーポイント | メッセージブローカー（Kafka等）経由 |
| 単一クラウド | マルチクラウド対応 |

### 8.2 移行時の互換性戦略

```
バージョニング戦略:
  - Webhook ペイロードのスキーマバージョニング（v1, v2...）
  - 旧スキーマのサポート期間: 新バージョン公開後 12 ヶ月
  - 非推奨通知: 廃止 6 ヶ月前に通知
  - 移行ガイド: 各バージョンアップ時に提供
```
