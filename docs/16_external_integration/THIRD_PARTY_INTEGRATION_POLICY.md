# サードパーティ統合ポリシー

ServiceMatrix Third-Party Integration Policy

Version: 1.0
Status: Active
Classification: Internal Governance Document
Last Updated: 2026-03-02

---

## 1. 概要

本ドキュメントは、ServiceMatrix とサードパーティシステム・サービスを統合する際の
ポリシー・評価基準・承認フロー・運用規則を定義する。
J-SOX IT全般統制および情報セキュリティ要件を満たす統合のみを許可する。

---

## 2. 統合分類

### 2.1 統合種別定義

| 分類 | 説明 | 例 |
|------|------|-----|
| コアインテグレーション | 業務プロセスに不可欠な統合 | GitHub, Slack |
| 補助インテグレーション | 業務効率化を目的とした統合 | Datadog, PagerDuty |
| オプションインテグレーション | 個別ユーザーの選択による統合 | 個人カレンダー連携 |

### 2.2 統合リスクレベル

| リスクレベル | 定義 | 承認者 |
|------------|------|--------|
| Critical | データの外部転送・認証情報共有 | CISO + CTO |
| High | 重要データへの読み取りアクセス | セキュリティ責任者 + テックリード |
| Medium | 一般データの読み書きアクセス | テックリード |
| Low | 通知・メタデータのみ | 開発リード |

## 3. 承認済み統合サービス

### 3.1 承認済みコアインテグレーション

| サービス | 用途 | リスク | 認証方式 | 承認日 |
|---------|------|--------|---------|--------|
| GitHub | ソースコード管理・ガバナンス | Medium | GitHub App | 2026-03-01 |
| Slack | 通知・アラート配信 | Low | Bot Token (OAuth 2.0) | 2026-03-01 |
| Microsoft Teams | 企業チャット通知 | Low | Incoming Webhook | 2026-03-01 |
| PagerDuty | P1 インシデント緊急通知 | Low | API Key | 2026-03-01 |

### 3.2 承認済み補助インテグレーション

| サービス | 用途 | リスク | 認証方式 | 承認日 |
|---------|------|--------|---------|--------|
| Datadog | APM・ログ監視 | Medium | API Key | 2026-03-01 |
| Prometheus/Grafana | メトリクス可視化 | Low | 内部ネットワーク | 2026-03-01 |
| Sentry | エラートラッキング | Medium | DSN Key | 2026-03-01 |
| LDAP/Active Directory | ユーザー認証 | High | サービスアカウント | 2026-03-01 |

### 3.3 未承認統合（要申請）

以下のカテゴリは未承認であり、利用前に審査が必要。

| カテゴリ | 理由 |
|---------|------|
| AI/ML 外部サービス | データプライバシー評価が必要 |
| 海外クラウドサービス（個人データ含む） | 国外移転規制の確認が必要 |
| 財務・会計システム | J-SOX 統制要件の評価が必要 |
| 医療関連システム | 個人情報保護法の評価が必要 |

## 4. 統合評価フレームワーク

### 4.1 評価基準

新規統合申請時に以下の観点で評価する。

| 評価項目 | チェック内容 | 必須/任意 |
|---------|------------|---------|
| セキュリティ | 認証方式・暗号化・脆弱性履歴 | 必須 |
| コンプライアンス | 法令遵守・ISO認証・SOC 2 Type II | 必須 |
| データプライバシー | 個人情報取扱い・データ所在地 | 必須 |
| 可用性 | SLA・障害履歴・サポート体制 | 必須 |
| 運用性 | ドキュメント整備・API安定性 | 推奨 |
| コスト | ライセンス費用・従量課金 | 推奨 |

### 4.2 評価スコアリング

| 評価項目 | 配点 |
|---------|------|
| セキュリティ | 40 点 |
| コンプライアンス | 30 点 |
| データプライバシー | 20 点 |
| 可用性 | 10 点 |
| 合計 | 100 点 |

| スコア | 判定 | アクション |
|-------|------|-----------|
| 80 点以上 | 承認 | 標準承認フロー |
| 60-79 点 | 条件付き承認 | 追加条件を付して承認 |
| 60 点未満 | 非承認 | 代替案を検討 |

## 5. 統合申請フロー

### 5.1 申請から承認までのフロー

```
統合申請フロー:
  1. 申請者が Integration Request Issue を作成
  2. セキュリティチームが評価スコアリングを実施（5営業日以内）
  3. リスクレベルに応じた承認者がレビュー
  4. 承認の場合: 統合設計仕様書を作成
  5. 統合実装 PR を提出
  6. CI/CD セキュリティチェックを通過
  7. ステージング環境での動作検証
  8. 本番デプロイ
  9. 統合台帳に登録
```

### 5.2 申請 Issue テンプレート

```
タイトル: [統合申請] {サービス名} - {用途}

## 申請概要
- サービス名: {名称}
- 用途: {具体的な利用目的}
- 申請者: @{github_login}
- 必要日程: {利用開始希望日}

## 統合詳細
- 認証方式: {OAuth/APIKey/Webhook等}
- アクセスするデータ: {データ種別・量}
- データ転送方向: {受信/送信/双方向}
- データ保存先: {ベンダー側/自社のみ}

## セキュリティ情報
- SOC 2 Type II 認証: {有/無}
- ISO 27001 認証: {有/無}
- 脆弱性開示ポリシー: {URLまたは記載}
- プライバシーポリシー: {URL}

## 代替案検討
{代替案を検討した場合はその内容を記載}
```

## 6. 統合実装基準

### 6.1 技術実装要件

| 要件 | 内容 |
|------|------|
| 秘密情報管理 | API キーは Kubernetes Secret または Vault で管理、コードに含めない |
| 通信暗号化 | すべての外部通信は TLS 1.2 以上を使用 |
| 入力検証 | 外部からのデータはすべてバリデーション実施 |
| エラーハンドリング | 外部サービス障害時はサーキットブレーカーを適用 |
| タイムアウト設定 | 外部 API 呼び出しには必ずタイムアウトを設定（最大 30 秒）|
| ログ記録 | 外部 API 呼び出しの成否・応答時間を記録 |

### 6.2 認証情報ライフサイクル

| 種別 | ローテーション | 管理 |
|------|--------------|------|
| API キー | 四半期 | Vault / K8s Secret |
| OAuth トークン | 自動更新（有効期限管理） | アプリ内キャッシュ |
| サービスアカウント | 年次 | IAM |
| Webhook Secret | 半年 | K8s Secret |

## 7. 依存関係管理

### 7.1 統合台帳

すべての承認済み統合を台帳で一元管理する。

| 項目 | 内容 |
|------|------|
| 台帳保存場所 | GitHub リポジトリ内 `docs/16_external_integration/registry/` |
| 更新タイミング | 統合追加・変更・削除時 |
| レビュー頻度 | 四半期 |
| 責任者 | テックリード |

### 7.2 廃止・削除フロー

```
統合廃止フロー:
  1. 廃止申請 Issue を作成
  2. 影響範囲を分析（依存機能・データの確認）
  3. データ移行・退避計画の作成
  4. 承認者の承認取得
  5. 段階的無効化（Feature Flag 活用）
  6. 認証情報の失効
  7. コードの削除（PR 経由）
  8. 台帳から削除
  9. 監査ログに廃止記録
```

## 8. セキュリティ・コンプライアンス要件

### 8.1 J-SOX 観点での統制要件

| 統制 | 要件 |
|------|------|
| アクセス管理（AC） | 外部サービスへのアクセス権は最小権限 + 定期レビュー |
| 変更管理（CM） | 統合設定変更は承認フローを経る |
| 監査証跡（AT） | 外部サービスへの重要操作はすべて監査ログに記録 |

### 8.2 情報セキュリティ要件

| 要件 | 詳細 |
|------|------|
| 機密データの外部送信禁止 | 個人情報・認証情報・財務データの外部送信は原則禁止 |
| データマスキング | 開発・テスト環境では本番データの代わりに匿名化データを使用 |
| ベンダーリスク評価 | 重要統合の提供事業者は年次でセキュリティ評価を実施 |

## 9. インシデント対応

### 9.1 統合インシデントの分類

| 種別 | 説明 | 優先度 |
|------|------|--------|
| 認証情報漏洩 | API キーや秘密情報が漏洩した可能性 | P1 緊急 |
| 不正アクセス | 統合経由の不正なデータアクセス | P1 緊急 |
| サービス障害 | 統合先サービスの障害による業務影響 | P2 高 |
| 性能劣化 | 統合応答時間の著しい悪化 | P3 中 |

### 9.2 認証情報漏洩時の対応

```
緊急対応手順（認証情報漏洩）:
  1. 即座に該当 API キー/トークンを失効させる（5分以内）
  2. セキュリティ責任者とCISOに即時通知
  3. 被害範囲の調査開始（アクセスログ分析）
  4. 新しい認証情報を発行・設定
  5. インシデント報告書を作成（24時間以内）
  6. 根本原因を特定し再発防止策を実装
```
