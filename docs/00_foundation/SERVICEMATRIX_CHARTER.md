## 2. 使命（Mission）

ServiceMatrix は、GitHub および自動化ワークフローとネイティブ統合された、
AI 駆動型・多次元型 IT サービス統治フレームワークを確立するために存在する。

本システムの使命は以下の5項に集約される。

### 2.1 運用の透明性確保

すべてのサービス運用活動は記録・追跡・可視化されなければならない。
暗黙の運用、記録なき変更、追跡不能な判断は許容しない。

### 2.2 構造化されたサービス統治の実施

ITIL 4 の指導原則に準拠し、プロセス・ポリシー・承認フローを
体系的に設計・運用する。場当たり的な対応ではなく、
決定論的な統治メカニズムを適用する。

### 2.3 知的自動化の実現

ClaudeCode Agent Teams を中核とするAI知能層により、
分析・提案・低リスク修復を自動化する。ただし、
統治ゲートを迂回する自動化は許容しない。

### 2.4 監査対応可能な証跡管理の維持

すべてのサービス状態変更・承認判断・AI判断には、
タイムスタンプ付きの証跡が残されなければならない。
内部監査・外部監査の両方に耐えうる記録品質を維持する。

### 2.5 標準フレームワークの調和

ITIL 4、ISO/IEC 20000、内部統制（J-SOX）、DevOps プラクティスを
対立させるのではなく、統合・調和させる設計思想を採用する。

## 3. ビジョン（Vision）

ServiceMatrix は、以下の4軸が交差する構造的マトリクスアーキテクチャ上に構築された、
自己進化型サービス統治エンジンとなることを目指す。

```
Process × Intelligence × DevOps × Governance
```

この4軸の交差により、従来の ITSM ツールでは実現困難であった
以下の価値を提供する。

- 手動調整コストの大幅削減
- 統治の分断・サイロ化の防止
- 適応的知性を備えた決定論的制御
- 社内外のサービス領域への拡張性
- リアルタイム統治状態の可視化

---

## 4. コア統治軸（Matrix Model）

ServiceMatrix は、以下の4つの統治軸の交差により構成される。
いかなる構成要素も、これら統治軸の外で動作してはならない。

### 4.1 プロセス軸（Process Axis）

ITIL 4 のサービスバリューチェーンに準拠し、
以下のコアプロセスを統治対象とする。

| プロセス | 目的 | 主要KPI |
|----------|------|---------|
| インシデント管理 | サービス復旧の迅速化 | MTTR, 初回解決率 |
| サービス要求管理 | 定型サービスの効率的提供 | 平均処理時間, 満足度 |
| 変更管理 | 変更リスクの最小化 | 変更成功率, CAB効率 |
| 問題管理 | 根本原因の特定と排除 | 既知エラー解決率 |
| リリース管理 | 安全なデプロイの実現 | デプロイ頻度, 失敗率 |

### 4.2 インテリジェンス軸（Intelligence Axis）

AI による知的支援層を提供する。

| 機能 | 説明 |
|------|------|
| ClaudeCode Agent Teams | 複数AIエージェントによる並列作業 |
| SubAgent オーケストレーション | 軽量タスクの委任と管理 |
| 自動影響分析 | 変更・インシデントの波及範囲を自動推定 |
| 自動修復ループ | ポリシー範囲内での自動修復実行 |
| リスクスコアリング | 変更・インシデントのリスクを定量評価 |
| トレンド分析 | インシデント傾向の検出と予測 |

### 4.3 DevOps軸（DevOps Axis）

GitHub エコシステムとのネイティブ統合を実現する。

| 要素 | 統治との関係 |
|------|-------------|
| GitHub Issues | サービスチケットの一元管理 |
| Pull Request | 変更承認フローの実装 |
| GitHub Actions | CI/CD パイプラインと自動検証 |
| ブランチ統治 | main保護とフロー制御 |
| WorkTree 分離 | Agent並列作業の衝突回避 |
| Labels / Milestones | 分類・優先度・進捗の可視化 |

### 4.4 統治軸（Governance Axis）

組織統治とコンプライアンスの実装層を提供する。

| 要素 | 目的 |
|------|------|
| SLA 強制 | サービスレベル目標の自動監視と逸脱検知 |
| 監査証跡保持 | すべての判断・変更の記録保全 |
| コンプライアンス検証 | ポリシー準拠の自動チェック |
| 証拠保全 | 監査対応のための証拠アーカイブ |
| ポリシー適用 | 統治ポリシーの自動実施 |
| 職務分掌（SoD） | 権限の適切な分離と相互牽制 |

## 5. 統治原則（Governance Principles）

ServiceMatrix のすべての設計・運用判断は、以下の原則に従う。

### 原則1: 決定論的運用の優先

場当たり的な運用ではなく、事前に定義されたプロセス・ポリシー・承認フローに
基づく運用を行う。例外処理も、事前に定義された例外ルートに従う。

### 原則2: 追跡可能性の確保

すべてのサービス状態変更には、以下の4要素が記録される。

- 記録（何が変更されたか）
- 理由（なぜ変更されたか）
- 実行主体（誰が/何が変更したか）
- タイムスタンプ（いつ変更されたか）

### 原則3: 安全な範囲での自動化

自動化は統治ゲートの範囲内でのみ実施される。
自動化がポリシーを迂回することは許容しない。

### 原則4: 人間承認の介在

高リスク変更、ポリシー逸脱、権限昇格には、
必ず人間による承認判断を介在させる。

### 原則5: 監査可能性の必須化

すべての運用活動は、内部監査および外部監査に
耐えうる品質で記録・保全される。

### 原則6: AI 統治境界の遵守

AI は補助・分析・提案を行うが、統治そのものを代替しない。
AI の判断にも統治ゲートが適用される。

## 6. アーキテクチャ原則（Architectural Doctrine）

ServiceMatrix は以下の設計特性を持つ。

### 6.1 GitHub ネイティブ

ServiceMatrix は GitHub のエコシステムを基盤として構築される。
独自のチケットシステムや承認フローを構築するのではなく、
GitHub Issues、Pull Request、Actions、Labels を活用する。

### 6.2 AI 拡張型

人間の判断能力を代替するのではなく、拡張する。
AI は情報収集・分析・提案・低リスク自動化に特化し、
最終判断は統治ポリシーに基づいて行われる。

### 6.3 Issue 駆動型

すべてのサービス活動は Issue を起点とする。
Issue を経由しない変更・作業は存在しない。

### 6.4 ポリシー強制型

ポリシーは文書として存在するだけでなく、
GitHub Actions、ブランチ保護ルール、承認フローとして
技術的に強制される。

### 6.5 CI 検証型

すべての変更は CI パイプラインによる自動検証を通過する。
CI の結果は統治判断の入力として使用される。

## 7. 権限範囲（Scope of Authority）

### 7.1 統治対象

ServiceMatrix は以下を統治対象とする。

- IT サービス運用プロセス（Incident / Change / Problem / Request / Release）
- 変更ライフサイクル管理
- SLA 測定と逸脱管理
- インシデント優先度判定とエスカレーション
- リスク評価とスコアリング
- 統治検証ロジックと自動チェック
- CI/CD パイプライン統治
- AI エージェントの行動統治

### 7.2 統治対象外

ServiceMatrix は以下を代替するものではない。

- 人事管理システム（HRM）
- 財務 ERP システム
- 外部法的機関の権限
- 物理インフラ管理（データセンター設備等）

## 8. AI 統治条項（AI Governance Clause）

### 8.1 AI エージェントの権限

ServiceMatrix 内の AI エージェント（ClaudeCode Agent Teams, SubAgent）は、
以下の権限を持つ。

| 行動 | 権限 | 条件 |
|------|------|------|
| 分析の実施 | 許可 | 記録が必要 |
| 提案の作成 | 許可 | 記録が必要 |
| 低リスク自動修復 | 条件付き許可 | ポリシー範囲内、記録必須 |
| 高リスク変更 | 禁止 | 人間承認が必要 |
| 権限昇格 | 禁止 | いかなる場合も |

### 8.2 AI 自律レベル

AI の自律レベルは以下の3段階で設定可能である。

| レベル | 名称 | 説明 |
|--------|------|------|
| L1 | Advisory | 分析・提案のみ。実行は人間が行う |
| L2 | Assisted | 低リスク操作の自動実行。高リスクは人間承認 |
| L3 | Supervised | 広範な自動化。ただし監視と事後レビューを必須とする |

### 8.3 AI 判断の記録要件

AI が行ったすべての判断には、以下が記録される。

- 判断内容
- 入力データ
- 適用したロジック/ポリシー
- 出力結果
- 信頼度スコア（可能な場合）
- タイムスタンプ

## 9. コンプライアンス遵守（Compliance Commitment）

ServiceMatrix は以下のフレームワーク・標準に整合する。

### 9.1 ITIL 4

ITIL 4 の指導原則を設計の基盤とする。

- 価値に着目する
- 現状から始める
- フィードバックをもとに反復して進化する
- 協力し、可視性を高める
- 包括的に考え、取り組む
- 実用的にシンプルを保つ
- 最適化し、自動化する

### 9.2 ISO/IEC 20000

IT サービスマネジメントシステム（SMS）の国際標準に
適合する設計を行う。特に以下の要求事項を考慮する。

- サービスマネジメントシステムの確立と維持
- サービスレベル管理
- 変更管理
- インシデント管理
- 問題管理

### 9.3 内部統制（J-SOX）

IT 全般統制の観点から、以下を実装する。

- アクセス管理
- 変更管理統制
- 職務分掌
- 監査証跡

### 9.4 職務分掌（Segregation of Duties）

以下の職務は分離されなければならない。

- 変更の申請者と承認者
- 開発と本番環境操作
- テスト実施者とテスト承認者
- 監査実施者と被監査者

## 10. 進化方針（Evolution Policy）

ServiceMatrix は継続的に進化する統治システムである。

### 10.1 変更プロセス

すべてのアーキテクチャ変更・ポリシー変更は、以下の手順に従う。

1. Change Issue の作成
2. 影響分析の実施
3. リスク評価の記録
4. Pull Request の提出
5. CI による自動検証
6. レビューと承認
7. Merge と反映
8. 事後レビュー

### 10.2 バージョン管理

本憲章を含むすべての統治文書は Git によりバージョン管理され、
変更履歴が追跡可能である。

### 10.3 定期レビュー

本憲章は四半期ごとにレビューされ、必要に応じて改定される。
レビュー結果は記録として保全される。

## 11. 不可侵原則（Non-Negotiables）

以下は絶対的原則であり、いかなる理由があっても違反してはならない。

| # | 原則 | 説明 |
|---|------|------|
| 1 | 隠れた自動化の禁止 | すべての自動化は明示的に定義・記録される |
| 2 | 無言の権限昇格の禁止 | 権限変更は必ず承認プロセスを経る |
| 3 | 記録されない変更の禁止 | すべての変更は証跡として残る |
| 4 | SLA 迂回の禁止 | SLA 目標は最優先で遵守される |
| 5 | ログなき AI 判断の禁止 | AI のすべての判断は記録される |
| 6 | 承認なき本番変更の禁止 | 本番環境への変更は承認を必須とする |
| 7 | 監査証跡の削除禁止 | 一度記録された証跡は改ざん・削除不可 |

| 統治透明性 | 監査証跡カバレッジ | 100% |
| 自動化率向上 | 定型作業自動化率 | 80%以上 |
