## 1. 文書の目的

本文書は、ServiceMatrix の主要プロセスにおける役割と責任の分担を RACI マトリックスとして定義する。

RACI マトリックスは、各活動における責任の所在を明確化し、職務分掌（SoD: Segregation of Duties）の
遵守と統治の透明性を確保するための基盤文書である。

J-SOX の IT 全般統制要件、および ISO/IEC 20000 の責任分担要件を充足するために本文書を維持する。

---

## 2. RACI 記号の定義

| 記号 | 英語 | 日本語 | 説明 |
|------|------|--------|------|
| **R** | Responsible | 実行責任 | タスクを実際に遂行する役割。複数可 |
| **A** | Accountable | 説明責任 | 最終的な成果に対して責任を負う役割。各タスクに必ず1名のみ |
| **C** | Consulted | 協議先 | 実行前後に意見・情報を求められる役割（双方向コミュニケーション） |
| **I** | Informed | 報告先 | 進捗・結果を通知される役割（一方向コミュニケーション） |
| **(空白)** | — | 関与なし | 当該タスクへの関与なし |

**基本原則**:
- 各タスクには必ず1つの **A**（説明責任）が存在する
- **A** は複数名に分割しない（責任の曖昧化を防止する）
- **R** が設定されていないタスクは存在しない
- **A** と **R** は同一人物が兼任可能だが、SoD 制約に留意する

## 3. ロール定義

ServiceMatrix における主要ロールは以下のとおり定義する。

| 略称 | ロール名 | 説明 |
|------|---------|------|
| **SysA** | システム管理者 | システム全体の技術的管理・維持を担当。インフラ・セキュリティ設定の実施権限を持つ |
| **PO** | プロセスオーナー | 各 ITIL プロセスの設計・改善・KPI 達成に対する説明責任を持つ管理職 |
| **CM** | 変更管理者 | 変更プロセスの統括・CAB 運営・変更承認の調整を行う |
| **Sec** | セキュリティ担当者 | セキュリティポリシーの策定・リスク評価・セキュリティ審査を担当 |
| **EU** | エンドユーザー | システムを利用する業務担当者。インシデント・サービス要求の起票者 |
| **AIG** | AI 統治担当者 | AI エージェントの統治・自律レベル管理・AI 判断ログの監査を担当 |

## 5. RACI 適用規則と職務分離（SoD）制約

### 5.1 禁止される役割組み合わせ

以下の組み合わせは、同一人物が兼務することを禁止する。

| # | 役割A（実施者） | 役割B（承認・監査者） | 理由 |
|---|--------------|---------------------|------|
| 1 | 変更申請者 | 変更承認者（CM） | 変更の牽制・不正防止 |
| 2 | 変更実施者（SysA） | 変更承認者（CM） | 環境変更の独立性確保 |
| 3 | AI 修復実行（AIG） | AI 修復承認者（SysA） | AI 行動の人間監督保証 |
| 4 | セキュリティ設定変更者（SysA） | セキュリティ監査者（Sec） | セキュリティ審査の独立性 |
| 5 | インシデント対応者（SysA） | インシデント SLA 最終承認者（PO） | プロセス品質の独立性 |
| 6 | AI 統治ポリシー策定者（AIG） | AI 統治監査者（外部監査） | AI 統治の客観性保証 |

### 5.2 GitHub との連携設定

RACI の A（説明責任者）は、GitHub において以下の設定と連動する。

| RACI 役割 | GitHub 設定 |
|-----------|------------|
| 変更管理の A（CM） | CODEOWNERS に記載・PR の Required Reviewer |
| セキュリティの A（Sec） | セキュリティ関連ディレクトリの CODEOWNERS |
| AI 統治の A（AIG） | `.claude/` ディレクトリの CODEOWNERS |
| インシデント管理の A（PO） | Incident ラベル付き Issue の担当者 |
| SLA 管理の A（PO） | SLA 関連ラベルの通知先 |

## 6. RACI レビューと改訂

### 6.1 定期レビュー

本文書は以下のサイクルで見直す。

| レビュートリガー | 対応方法 |
|----------------|---------|
| 組織変更・役割変更 | 即時改訂 |
| 新規プロセス追加 | プロセス追加時に同時作成 |
| 監査指摘事項 | 指摘内容に基づき改訂 |
| 定期レビュー | 年次（1月第1営業日） |

### 6.2 改訂手順

1. RACI 変更の提案（GitHub Issue 起票）
2. 影響を受けるプロセスオーナーへの協議（C）
3. SoD 制約との整合性確認
4. プロセスオーナー（PO）の承認（A）
5. Pull Request による本文書の更新
6. CI 検証通過後 Merge
7. GitHub 権限設定への反映
