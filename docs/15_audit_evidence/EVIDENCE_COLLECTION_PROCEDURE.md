# エビデンス収集手順

ServiceMatrix Evidence Collection Procedure

Version: 1.0
Status: Active
Classification: Internal Audit Document
Last Updated: 2026-03-02

---

## 1. 概要

本ドキュメントは、ServiceMatrix におけるエビデンス収集の
手順・責任分担・スケジュールを定義する。
J-SOX / ISO 20000 / ISO 27001 の監査対応に必要な
エビデンスを効率的かつ確実に収集するための標準手順を提供する。

---

## 2. 収集プロセス概要

### 2.1 全体フロー

```
[フェーズ1] エビデンス自動収集（常時稼働）
  +-- 監査ログ自動記録
  +-- 変更記録自動取得
  +-- テスト結果自動保存
  +-- SLA メトリクス自動計測
  |
  v
[フェーズ2] エビデンス検証（日次）
  +-- ハッシュチェーン検証
  +-- 完全性チェック
  +-- 欠損検出
  |
  v
[フェーズ3] エビデンス整理（月次）
  +-- 統制要件との紐づけ確認
  +-- メタデータ補完
  +-- 分類・タグ付け
  |
  v
[フェーズ4] 監査対応パッケージ作成（四半期 / 随時）
  +-- エビデンスパッケージ生成
  +-- 監査チームへの提供
  +-- 質問対応・追加収集
```

### 2.2 責任分担

| 役割 | 責任 |
|------|------|
| SystemAdmin | エビデンス収集基盤の維持・運用 |
| Auditor | エビデンスの検証・レビュー・監査対応 |
| ChangeManager | 変更管理エビデンスの品質保証 |
| ProcessOwner | 各プロセスエビデンスの正確性担保 |
| DevOps | CI/CD関連エビデンスの自動収集維持 |
| AI Agent | エビデンス自動分類・欠損検出支援（L2） |

## 3. 自動収集エビデンス

### 3.1 監査ログ（リアルタイム収集）

| ログ種別 | 収集内容 | 収集頻度 | 保存先 |
|---------|---------|---------|--------|
| 認証ログ | ログイン/ログアウト、認証試行 | リアルタイム | Audit Trail DB |
| 操作ログ | CRUD操作、設定変更 | リアルタイム | Audit Trail DB |
| アクセスログ | API呼び出し、画面遷移 | リアルタイム | Audit Trail DB |
| 権限変更ログ | ロール変更、権限付与/剥奪 | リアルタイム | Audit Trail DB |
| AI判断ログ | AI入力、ロジック、出力、承認結果 | リアルタイム | Audit Trail DB |

#### 監査ログレコード構造

```
監査ログレコード:
  id: UUID（一意識別子）
  timestamp: ISO 8601 UTC
  actor_id: 操作者ID（ユーザーまたはAI Agent）
  actor_type: "user" | "ai_agent" | "system"
  action: "create" | "read" | "update" | "delete" | "approve" | "reject"
  resource_type: "incident" | "change_request" | "ci" | "user" | ...
  resource_id: 対象リソースID
  details: 操作詳細（JSON）
  ip_address: 接続元IPアドレス
  user_agent: クライアント情報
  result: "success" | "failure" | "denied"
  previous_hash: 前レコードのハッシュ値
  hash: 現レコードのハッシュ値（SHA-256）
```

### 3.2 変更管理エビデンス

| エビデンス | 収集元 | 収集タイミング |
|-----------|--------|--------------|
| RFC（変更要求） | GitHub Issue | Issue作成時 |
| リスク評価結果 | Risk Engine | RFC作成時に自動実行 |
| 影響分析結果 | CMDB Core | RFC作成時に自動実行 |
| CABレビュー記録 | GitHub Issue コメント | レビュー実施時 |
| 承認記録 | GitHub PR 承認 | PR承認時 |
| テスト結果 | CI/CD Pipeline | テスト実行時 |
| デプロイ記録 | CI/CD Pipeline | デプロイ実行時 |
| PIR（事後レビュー） | GitHub Issue | PIR完了時 |

### 3.3 テスト・品質エビデンス

| エビデンス | 収集元 | 保存形式 |
|-----------|--------|---------|
| ユニットテスト結果 | Vitest | JUnit XML |
| カバレッジレポート | c8 | HTML + JSON |
| 統合テスト結果 | Vitest + Supertest | JUnit XML |
| E2Eテスト結果 | Playwright | HTML + JSON |
| セキュリティスキャン結果 | CodeQL / Semgrep / ZAP | SARIF / JSON |
| 依存関係スキャン結果 | Dependabot / Snyk | JSON |
| パフォーマンステスト結果 | k6 | JSON + HTML |
| 品質ゲート判定結果 | CI/CD Pipeline | JSON |

### 3.4 SLA・メトリクスエビデンス

| エビデンス | 収集元 | 収集頻度 |
|-----------|--------|---------|
| SLA達成率 | SLA Engine | リアルタイム計測、月次集計 |
| SLA違反記録 | SLA Engine | 違反検出時 |
| インシデント応答時間 | Incident Manager | インシデント毎 |
| インシデント解決時間 | Incident Manager | インシデント毎 |
| 変更成功率 | Change Manager | 変更完了毎 |
| システム可用性 | Monitoring | 常時計測、月次集計 |

## 4. 手動収集エビデンス

### 4.1 手動収集が必要なエビデンス

| エビデンス | 担当 | 頻度 | 手順 |
|-----------|------|------|------|
| アクセス権限レビュー記録 | Auditor | 四半期 | 権限一覧取得→レビュー→承認記録 |
| リスク評価レビュー記録 | ProcessOwner | 半年 | リスク登録簿レビュー→更新記録 |
| サービスレベルレビュー記録 | ProcessOwner | 四半期 | SLAレポートレビュー→改善計画 |
| 事業継続計画テスト記録 | SystemAdmin | 年次 | DR訓練実施→結果記録 |
| セキュリティ教育記録 | HR/Admin | 年次 | 教育実施→受講記録 |
| 外部監査報告書 | Auditor | 年次 | 外部監査→報告書受領 |

### 4.2 手動収集手順

各手動収集エビデンスに対し、以下の標準手順に従う。

```
手動エビデンス収集手順:
  1. エビデンス収集タスクの発行（GitHub Issue）
  2. 担当者の割り当て
  3. エビデンスの作成/取得
  4. エビデンスの検証（二重チェック）
  5. メタデータの付与
     - 統制ID
     - 収集日時
     - 収集者
     - 対象期間
  6. エビデンスの保存（指定ストレージ）
  7. エビデンスの登録（マッピングシステム）
  8. 収集タスクの完了
```

## 5. エビデンス検証

### 5.1 自動検証（日次）

| 検証項目 | 方法 | 失敗時の対応 |
|---------|------|------------|
| ハッシュチェーン整合性 | SHA-256チェーン検証 | Critical アラート発報 |
| エビデンス件数 | 前日比での異常検出 | Warning アラート発報 |
| タイムスタンプ連続性 | ギャップ検出 | 調査タスク起票 |
| 保存先アクセス可能性 | ストレージヘルスチェック | Critical アラート発報 |
| 暗号化状態 | 保存データの暗号化確認 | 即時対応 |

### 5.2 手動検証（月次）

| 検証項目 | 担当 | 方法 |
|---------|------|------|
| サンプリング検証 | Auditor | ランダムサンプルの内容確認 |
| 統制マッピング確認 | Auditor | 統制要件とエビデンスの紐づけ確認 |
| 網羅性確認 | Auditor | 全統制要件に対するエビデンスの有無 |
| メタデータ正確性 | Auditor | メタデータの正確性スポットチェック |

### 5.3 エビデンス欠損時の対応

| 欠損種別 | 対応 | 期限 |
|---------|------|------|
| 自動収集の一時停止 | システム復旧後にバックフィル | 24時間以内 |
| 手動収集の遅延 | 担当者へのリマインダー、エスカレーション | 7日以内 |
| データ損失 | バックアップからの復元、インシデント起票 | 即時 |
| 改竄検出 | インシデント起票、フォレンジック調査 | 即時 |

## 6. エビデンス保存・保持

### 6.1 保存ポリシー

| エビデンス種別 | 保持期間 | 法的根拠 | 保存先 |
|--------------|---------|---------|--------|
| 監査ログ | 7年 | J-SOX（金融商品取引法） | DB + オブジェクトストレージ |
| 変更管理記録 | 7年 | J-SOX | GitHub + バックアップ |
| テスト結果 | 3年 | 内部規程 | CI/CD + オブジェクトストレージ |
| SLA記録 | 7年 | ISO 20000 | DB + オブジェクトストレージ |
| AI判断記録 | 7年 | AI統治ポリシー | DB + オブジェクトストレージ |
| インシデント記録 | 7年 | J-SOX / ISO 20000 | GitHub + バックアップ |
| 設定変更記録 | 永続 | ISO 27001 | Git リポジトリ |

### 6.2 アーカイブプロセス

```
アーカイブプロセス（月次実行）:
  1. 保持期間超過データの特定
  2. アーカイブ対象のリスト生成
  3. アーカイブ先への転送
  4. 転送の完全性検証（チェックサム照合）
  5. 元データの削除（保持期間超過分のみ）
  6. アーカイブログの記録
```

### 6.3 暗号化要件

| 保存状態 | 暗号化方式 |
|---------|-----------|
| 保存時（at rest） | AES-256-GCM |
| 転送時（in transit） | TLS 1.3 |
| バックアップ | AES-256-GCM |
| アーカイブ | AES-256-GCM + 鍵の分離保管 |

## 7. 監査対応手順

### 7.1 監査通知受領時の対応

```
監査通知受領
  |
  v
[Day 1-2] 監査範囲の確認
  +-- 対象統制の特定
  +-- 対象期間の確認
  +-- 必要エビデンスの一覧作成
  |
  v
[Day 3-5] エビデンスパッケージ作成
  +-- 自動収集エビデンスの抽出
  +-- 手動収集エビデンスの確認
  +-- パッケージの生成・検証
  |
  v
[Day 5-7] 内部レビュー
  +-- エビデンスの品質確認
  +-- 欠損の特定・補完
  +-- パッケージの最終確認
  |
  v
[Day 7-10] 監査チームへの提供
  +-- エビデンスパッケージの提出
  +-- 説明資料の準備
  +-- 質問対応体制の確立
```

### 7.2 監査質問対応

| 対応レベル | 対応者 | 対応時間 |
|-----------|--------|---------|
| エビデンス補足説明 | Auditor | 24時間以内 |
| 追加エビデンス提供 | 各プロセスオーナー | 48時間以内 |
| 技術的説明 | DevOps / テックリード | 48時間以内 |
| AI関連説明 | AI統治責任者 | 48時間以内 |

## 8. エビデンス品質管理

### 8.1 品質基準

| 基準 | 定義 | 検証方法 |
|------|------|---------|
| 正確性 | エビデンスの内容が事実を正確に反映 | サンプリング検証 |
| 完全性 | 必要な情報がすべて含まれている | 必須フィールドチェック |
| 適時性 | 適切なタイミングで収集されている | タイムスタンプ検証 |
| 一貫性 | エビデンス間で矛盾がない | 相互参照検証 |
| 信頼性 | 改竄されていない | ハッシュチェーン検証 |

### 8.2 品質メトリクス

| メトリクス | 目標 | 測定頻度 |
|-----------|------|---------|
| 自動収集率 | 90%以上 | 月次 |
| エビデンス欠損率 | 0% | 日次 |
| ハッシュ検証成功率 | 100% | 日次 |
| 監査対応所要時間 | 10営業日以内 | 監査毎 |
| 監査指摘件数 | 0件 | 監査毎 |
