# 監査ログ保管ポリシー（Audit Log Retention Policy）

ServiceMatrix 監査ログ統治仕様
Version: 1.0
Status: Active
Classification: Confidential - Internal Governance Document
Applicable Standard: ITIL 4 / ISO/IEC 20000 / J-SOX / 金融商品取引法
Last Updated: 2026-03-02
Owner: セキュリティ統治担当 / Auditor

---

## 1. 目的

本ドキュメントは、ServiceMatrixにおいて生成・収集される監査ログの
種別・保管期間・完全性保証・アクセス制御・廃棄手順を定義する。

J-SOX（金融商品取引法に基づく内部統制報告制度）要件への準拠を確保し、
システム操作の透明性と追跡可能性を保証することを目的とする。

---

## 2. ログ種別と保管期間

### 2.1 ログ分類マトリックス

| ログ種別 | 説明 | 保管期間 | 根拠 |
|---------|------|---------|------|
| 操作ログ | システム・アプリケーションへの操作履歴 | **1年** | 内部運用基準 |
| 監査ログ | アクセス制御・権限変更・承認操作の記録 | **7年** | J-SOX要件 |
| セキュリティログ | 認証・認可・セキュリティイベントの記録 | **3年** | ISO/IEC 27001 |
| AI判断ログ | AIエージェントの提案・判断・操作記録 | **3年** | AI統治要件 |
| 変更管理ログ | 変更申請・承認・実施の記録 | **7年** | J-SOX要件 |
| SLA/KPIメトリクスログ | サービスレベル計測データ | **2年** | SLA管理要件 |
| システムイベントログ | エラー・警告・情報イベント | **1年** | 運用要件 |
| アクセスログ（Web） | HTTP/API アクセスログ | **1年** | 運用・セキュリティ要件 |
| BreakGlass使用ログ | 緊急アクセス使用記録 | **永久保管** | J-SOX特別要件 |

### 2.2 ログ種別の詳細定義

#### 操作ログ（Operation Log）

```
対象：
  - ServiceMatrix APIへのすべてのリクエスト・レスポンス
  - データベースCRUD操作（クエリレベル）
  - GitHub Actions実行ログ
  - Agent Teams の操作実行記録

記録項目：
  - タイムスタンプ（UTC / マイクロ秒精度）
  - 操作者ID / エージェントID
  - 操作種別（CREATE / READ / UPDATE / DELETE）
  - 対象リソース
  - 実行結果（成功 / 失敗 / エラーコード）
  - IPアドレス / ユーザーエージェント
```

#### 監査ログ（Audit Log）

```
対象：
  - ユーザー認証（ログイン / ログアウト / 失敗）
  - 権限変更（ロール付与 / 剥奪）
  - 承認操作（変更承認 / インシデントクローズ等）
  - 設定変更（システム設定 / SLA定義 / ポリシー変更）
  - 特権操作のすべて
  - J-SOX関連業務処理操作

記録項目：
  - タイムスタンプ（UTC / マイクロ秒精度）
  - 操作者ID
  - 操作内容の詳細
  - 変更前後の値
  - 関連するIssue番号 / 承認記録
  - ハッシュチェーン値（改ざん防止）
```

#### セキュリティログ（Security Log）

```
対象：
  - 認証失敗（連続失敗を含む）
  - 認可拒否イベント
  - 不審なアクセスパターン検知
  - IPブロック / アンブロック
  - セキュリティポリシー変更
  - 脆弱性検知アラート

記録項目：
  - タイムスタンプ（UTC / マイクロ秒精度）
  - イベント種別
  - 関与したユーザーID / IPアドレス
  - 検知ルール識別子
  - 対応処理（自動対応内容）
```

## 3. J-SOX要件との対応

### 3.1 J-SOX適用対象業務とログ対応

| J-SOX管理対象業務 | 対応ログ種別 | 保管期間 | 証跡として使用 |
|-----------------|------------|---------|--------------|
| 変更承認プロセス | 監査ログ | 7年 | 変更の承認者・日時・内容 |
| アクセス権管理 | 監査ログ | 7年 | 権限付与・変更・削除の証跡 |
| 本番環境デプロイ | 操作ログ + 監査ログ | 7年 | デプロイ実施者・承認・内容 |
| 財務データ操作 | 監査ログ | 7年 | データ変更の完全な証跡 |
| SLA報告データ生成 | SLA/KPIメトリクスログ | 2年 | 計測データの信頼性証明 |

### 3.2 J-SOX監査対応の証跡提供

J-SOX監査時に提供すべき証跡の一覧：

| 証跡種別 | 提供形式 | 対応ログ |
|---------|---------|---------|
| 職務分離の実施証明 | 承認操作ログのエクスポート | 監査ログ |
| アクセス権棚卸し記録 | 権限変更ログのエクスポート | 監査ログ |
| 変更管理の実施証明 | PR/Issue + 承認ログ | 監査ログ |
| ITシステムの稼働記録 | 可用性メトリクス | 操作ログ |
| 不正アクセス対応記録 | インシデント + セキュリティログ | セキュリティログ |

## 4. ログ完全性保証（改ざん防止・ハッシュチェーン）

### 4.1 ハッシュチェーン方式

ServiceMatrixは監査ログの改ざんを検知するため、ハッシュチェーン方式を採用する。

```
ログエントリN:
  {
    "timestamp": "...",
    "event_data": {...},
    "previous_hash": "hash_of_entry_N-1",
    "entry_hash": SHA256(event_data + previous_hash + timestamp)
  }
```

**効果：**
- 単一エントリの改ざんは後続する全エントリのハッシュ不整合として検出される
- ログの削除・挿入も不整合として検出される

### 4.2 完全性検証プロセス

| 検証タイミング | 検証方法 | 担当者 |
|-------------|---------|-------|
| 日次 | 自動ハッシュチェーン整合性チェック | システム自動 |
| 週次 | ランダムサンプリング手動検証 | Auditor |
| 監査時 | 対象期間の全エントリ検証 | Auditor + 外部監査人 |

### 4.3 ログの保護措置

| 保護措置 | 内容 |
|---------|------|
| 書き込み専用アクセス | ログ生成プロセスは書き込みのみ可能（削除・更新不可） |
| ストレージ暗号化 | 保管中のログはAES-256で暗号化 |
| 転送時暗号化 | ログ転送はTLS 1.3を使用 |
| オフサイト複製 | 監査ログ（7年保管）は別リージョン/別システムに複製 |
| アクセス制限 | Auditorロールのみ参照可（後述） |

### 4.4 タイムスタンプ整合性

- **NTPサーバー同期**：すべてのシステムは信頼できるNTPサーバーと同期
- **UTCタイムゾーン統一**：すべてのログはUTCで記録
- **マイクロ秒精度**：イベントシーケンスの正確な記録

## 5. ログアクセス制御

### 5.1 ロール別アクセス権

| ロール | 操作ログ | 監査ログ | セキュリティログ | AI判断ログ |
|--------|---------|---------|---------------|----------|
| SystemAdmin | 参照可 | 参照可 | 参照可 | 参照可 |
| Auditor | **参照可（主担当）** | **参照可（主担当）** | **参照可（主担当）** | 参照可 |
| ProcessOwner | 自担当分のみ参照可 | 自担当操作分のみ | 参照不可 | 参照可 |
| ChangeManager | 変更関連のみ参照可 | 変更関連のみ | 参照不可 | 参照不可 |
| Operator | 参照不可 | 参照不可 | 参照不可 | 参照不可 |
| EndUser | 自己ログのみ参照可 | 参照不可 | 参照不可 | 参照不可 |
| AIAgent | 参照不可 | 参照不可 | 参照不可 | 自己ログのみ |

### 5.2 監査ログアクセスの特別規定

監査ログへのアクセスは以下の追加統制を適用する：

1. **アクセス自体をログ記録**：監査ログへのアクセスは別のメタ監査ログに記録
2. **エクスポート承認**：監査ログのエクスポートにはProcessOwnerの承認が必要
3. **目的の記録**：アクセス時に参照目的を記録することを義務付ける
4. **アクセス期間の記録**：閲覧開始・終了時刻を記録

### 5.3 ログへのアクセス要求手順

```
1. アクセス申請
   - 対象ログ種別・期間・参照目的をIssue起票

2. 承認
   - ProcessOwner（監査ログ）またはSystemAdmin（セキュリティログ）が承認

3. 一時アクセス付与
   - 承認後、必要期間限定のアクセス権付与

4. 記録
   - アクセス開始・操作・終了を記録

5. 権限失効
   - 目的完了または期限到来時に自動失効
```

## 6. ログ削除プロセス（保管期間満了後の安全削除）

### 6.1 削除スケジュール管理

- ログの保管期間満了は自動的に検知・記録する
- 削除実施前に、Auditorによる最終確認を実施する
- 削除は四半期ごとにバッチで実施（通常・緊急の場合を除く）

### 6.2 安全削除手順

```
ステップ1: 削除対象の特定
  - 保管期間満了ログの一覧を生成
  - 進行中の監査・調査への影響確認

ステップ2: 削除前確認
  - Auditorによる削除対象リストの確認・承認
  - J-SOX保管期間中のログが含まれていないことを確認

ステップ3: 削除実施
  - 暗号化鍵の削除によるCrypto Erase（推奨）
  - または安全なファイル削除ツールによる上書き削除

ステップ4: 削除記録
  - 削除実施日時・削除対象・実施者を記録
  - この削除記録自体は永久保管

ステップ5: 検証
  - 削除完了の確認
  - ストレージの完全性チェック
```

### 6.3 削除禁止期間

以下の期間中は、関連するログの削除を禁止する：

| 禁止理由 | 禁止期間 |
|---------|---------|
| J-SOX監査実施中 | 監査完了まで |
| セキュリティインシデント調査中 | 調査完了・是正措置完了まで |
| 法的調査・訴訟 | 解決まで |
| 外部監査実施中 | 監査完了まで |

## 7. バックアップとリカバリ

### 7.1 ログバックアップ戦略

| ログ種別 | バックアップ頻度 | バックアップ保管期間 | バックアップ先 |
|---------|--------------|-----------------|-------------|
| 監査ログ（7年保管） | リアルタイム複製 + 日次バックアップ | 保管期間と同等 | オフサイト（別リージョン） |
| セキュリティログ（3年保管） | 日次バックアップ | 保管期間と同等 | オフサイト |
| 操作ログ（1年保管） | 日次バックアップ | 保管期間と同等 | 同一リージョン別ストレージ |

### 7.2 バックアップ完全性検証

- **毎月**：バックアップからのリストアテストを実施
- **四半期**：完全リストアシナリオのテスト
- **検証記録**：テスト結果はテスト台帳に記録し保管

### 7.3 ログリカバリ手順

```
1. 障害検知
   - ログストレージ障害の検知
   - 影響範囲の特定

2. 代替記録への切り替え
   - バックアップストレージへの書き込み切り替え
   - 記録の継続性を確保

3. 主ストレージのリカバリ
   - システム復旧後、バックアップからリストア
   - ハッシュチェーン整合性の確認

4. ギャップ検証
   - 障害期間中のログの存在確認
   - 欠損ログの特定と記録

5. 事後報告
   - リカバリ完了報告の作成
   - 再発防止策の策定
```

## 8. コンプライアンス監視

### 8.1 定期確認事項

| 確認事項 | 頻度 | 担当者 |
|---------|------|-------|
| ログ保管量・ストレージ使用率 | 週次 | Operator |
| ハッシュチェーン整合性チェック | 日次（自動）/ 週次（手動確認） | Auditor |
| バックアップ成功確認 | 日次 | Operator |
| 保管期間満了ログの確認 | 月次 | Auditor |
| アクセスログのレビュー | 月次 | Auditor |

### 8.2 年次コンプライアンスレポート

Auditorは年次にて以下を含むコンプライアンスレポートを作成する：

- ログ保管状況サマリー
- J-SOX対応ログの完全性証明
- ハッシュチェーン検証結果
- バックアップ・リカバリテスト結果
- ポリシー例外事項と対応
- 改善推奨事項
