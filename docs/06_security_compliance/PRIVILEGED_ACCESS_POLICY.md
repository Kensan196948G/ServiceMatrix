# 特権アクセスポリシー（Privileged Access Policy）

ServiceMatrix 特権アクセス統治仕様
Version: 1.0
Status: Active
Classification: Confidential - Internal Governance Document
Applicable Standard: ITIL 4 / ISO/IEC 20000 / J-SOX / ISO/IEC 27001
Last Updated: 2026-03-02
Owner: セキュリティ統治担当 / SystemAdmin

---

## 1. 目的

本ドキュメントは、ServiceMatrixにおける特権アクセスの定義・管理・監視に関するポリシーを定義する。

特権アクセスの管理不備はシステム全体への影響を及ぼす重大なリスクであり、
J-SOX要件・ISO/IEC 27001要件に対応するため、厳格な統制を確立する。

---

## 2. 特権アクセスの定義と範囲

### 2.1 特権アクセスの定義

特権アクセスとは、通常の業務ユーザーに与えられる範囲を超えた、
システムやデータへの高い権限を伴うアクセスを指す。

### 2.2 特権アクセスの分類

| 分類 | 説明 | 対象ロール |
|------|------|-----------|
| システム管理権限 | OS・インフラ・データベースへの管理者アクセス | SystemAdmin |
| アプリケーション管理権限 | ServiceMatrixアプリケーション設定・全データへのアクセス | SystemAdmin / ProcessOwner |
| セキュリティ管理権限 | アクセス制御設定・監査ログ・セキュリティポリシーの変更 | SystemAdmin |
| AI統治管理権限 | AIエージェント権限設定・自律レベル変更 | AI統治担当 |
| 緊急アクセス（BreakGlass） | 通常手続きを迂回した緊急操作 | 指定者2名のみ |

### 2.3 特権操作の具体例

以下の操作は「特権操作」として分類し、特別な統制を適用する：

**システム層：**
- データベースへの直接SQL実行（READ / WRITE）
- サーバー設定ファイルの変更
- 本番環境への直接デプロイ
- ネットワーク設定の変更

**アプリケーション層：**
- ユーザーアカウントの作成・削除・権限変更
- SLA定義・KPI閾値の変更
- システム設定パラメータの変更
- 監査ログのエクスポート・削除

**GitHub / CI/CD層：**
- Branch Protection Rulesの変更
- GitHub Organizationの設定変更
- CI/CDパイプライン設定の変更
- Secretsの追加・変更・削除

**AI統治層：**
- AIエージェントの自律レベル変更
- AIエージェントへの新規権限付与
- AIエージェントアクションのバイパス承認

## 3. Just-in-Time（JIT）アクセス原則

### 3.1 JITアクセスの概念

特権アクセスは常時付与するのではなく、必要な時に必要な期間だけ付与する
Just-in-Time（JIT）方式を基本原則とする。

```
申請 → 承認 → 一時的な特権付与 → 操作実行 → 自動失効 → 記録
```

### 3.2 JITアクセス申請フロー

```
ステップ1: 申請
  - 対象システム・操作内容を明示
  - 業務上の必要性を説明
  - 要求する権限レベルと期間を指定
  - GitHub IssueまたはServiceMatrix内で申請

ステップ2: 承認
  - ProcessOwnerまたはSystemAdminが審査
  - 必要性・リスクの確認
  - 承認または却下の記録

ステップ3: 一時権限付与
  - 承認後、指定期間のみ有効な権限を付与
  - 最大期間：通常操作は4時間 / 計画作業は8時間

ステップ4: 操作実行
  - 承認された操作のみを実行
  - 全操作はセッションログに記録

ステップ5: 自動失効
  - 指定期間経過後に権限を自動失効
  - 操作完了後は手動での早期返却を推奨

ステップ6: 事後報告
  - 実施した操作の概要を記録
  - 異常・問題が発生した場合はインシデント起票
```

### 3.3 JIT適用範囲外

以下のロールは常時特権が必要なため、JIT適用対象外とするが、
代わりに強化された監視を適用する：

| ロール | 常時特権範囲 | 追加統制 |
|--------|------------|---------|
| SystemAdmin（2名まで） | システム全体の管理権限 | 全操作ログの月次レビュー |
| Auditor | 監査ログへの読み取り専用アクセス | アクセスログの記録 |

## 4. 特権操作の事前承認要件

### 4.1 承認マトリックス

| 特権操作カテゴリ | 一次承認者 | 二次承認者（必要な場合） | 承認期限 |
|----------------|-----------|----------------------|---------|
| DB直接操作（本番） | ProcessOwner | SystemAdmin | 24時間以内 |
| 本番環境設定変更 | ProcessOwner | SystemAdmin | 24時間以内 |
| ユーザー権限変更 | ProcessOwner | - | 4時間以内 |
| GitHub設定変更 | SystemAdmin | ProcessOwner | 4時間以内 |
| AIエージェント権限変更 | AI統治担当 | SystemAdmin | 4時間以内 |
| セキュリティポリシー変更 | SystemAdmin | ProcessOwner | 24時間以内 |
| 緊急アクセス（BreakGlass） | BreakGlass権限者（1名） | - | 即時実行可（事後報告必須） |

### 4.2 変更管理プロセスとの連携

特権操作は原則として変更管理プロセスと連携させる：

- **通常変更**：変更管理プロセス（CAB承認）を経て実施
- **標準変更**：事前定義された手順に従い実施（個別承認不要）
- **緊急変更**：緊急承認フローを使用し事後承認も可

## 5. 特権セッションの記録と監視

### 5.1 セッション記録要件

すべての特権セッションは以下の情報を記録する：

| 記録項目 | 説明 |
|---------|------|
| セッションID | 一意の識別子 |
| 実施者ID | 操作者のユーザーID |
| 開始日時 | セッション開始のタイムスタンプ |
| 終了日時 | セッション終了のタイムスタンプ |
| アクセス対象 | 操作したシステム・リソース |
| 実行コマンド / 操作 | 実行した全コマンド・操作の詳細 |
| 承認参照番号 | 対応する承認Issue番号 |
| 操作結果 | 成功 / 失敗 / 一部成功 |
| 備考 | 異常事態・気づきの記録 |

### 5.2 リアルタイム監視

特権セッションは以下の方法でリアルタイム監視を実施する：

- **GitHubアクティビティ監視**：GitHub Audit Logを通じた操作監視
- **アプリケーションログ監視**：API操作ログのリアルタイム収集
- **アラート設定**：以下の場合に即時アラートを発報
  - 承認なしの特権操作試行
  - 業務時間外の特権操作
  - BreakGlassアカウントの使用
  - 異常に多数の操作実行

### 5.3 ログ保管

特権セッションのログは以下の期間保管する：
- 通常特権操作ログ：3年
- BreakGlass使用ログ：7年（J-SOX要件対応）
- 詳細はAUDIT_LOG_RETENTION_POLICY.mdを参照

## 6. 特権アカウントのライフサイクル管理

### 6.1 特権アカウントの種別

| 種別 | 説明 | 管理方法 |
|------|------|---------|
| 名前付き特権アカウント | 特定の個人に紐づく特権アカウント | 個人認証 + JIT |
| サービスアカウント | 自動化・CI/CD用のサービス専用アカウント | 証明書認証 + スコープ制限 |
| BreakGlassアカウント | 緊急時専用アカウント | 厳格な管理 / 通常使用禁止 |
| AIエージェントアカウント | AIエージェント専用のマシンアカウント | スコープ制限 + 全操作ログ |

### 6.2 ライフサイクルステージ

```
申請 → 審査 → 作成 → 運用 → 定期レビュー → 無効化 → 廃棄
```

| ステージ | 実施内容 | 担当者 |
|---------|---------|-------|
| 申請 | 特権アカウント必要性の申請 | 申請者 |
| 審査 | 業務必要性・リスク評価 | ProcessOwner |
| 作成 | アカウント作成・初期設定 | SystemAdmin |
| 運用 | JIT方式での利用 | 利用者 |
| 定期レビュー | 必要性確認・棚卸し（月次） | SystemAdmin |
| 無効化 | 不要判定または業務変更時の停止 | SystemAdmin |
| 廃棄 | アカウント完全削除・ログ保管 | SystemAdmin |

### 6.3 特権アカウント棚卸し

- **実施頻度**：月次
- **実施者**：SystemAdmin
- **確認内容**：
  - 最後のアクセス日時（90日以上未使用は要確認）
  - 業務上の必要性の継続確認
  - 権限スコープの妥当性
  - パスワード・証明書の有効期限

## 7. 緊急アクセス手順（BreakGlass）

### 7.1 BreakGlass手順の位置づけ

BreakGlass手順は、通常の承認プロセスが機能しない緊急事態においてのみ使用する。
誤用・濫用は重大な統治違反とみなす。

### 7.2 BreakGlass使用条件

以下の条件をすべて満たす場合にのみ使用を許可する：

1. 通常の承認者（ProcessOwner）に連絡がつかない
2. システム障害またはセキュリティインシデントにより即時対応が必要
3. 対応遅延による被害がBreakGlass使用リスクを上回ると判断される

### 7.3 BreakGlass実施手順

```
ステップ1: 使用判断
  - 使用条件を満たしていることを確認
  - 使用理由を記録（テキスト・時刻）

ステップ2: BreakGlass認証
  - 専用の物理的に保管された認証情報を取得
  - BreakGlass使用を開始した時刻を記録

ステップ3: 最小限の操作
  - 目的達成に必要な最小限の操作のみを実施
  - すべての操作を手動で記録

ステップ4: 操作完了後の処置
  - 使用した認証情報の即時失効
  - 新しい認証情報の生成
  - 物理的保管場所への格納

ステップ5: 事後報告（2時間以内）
  - GitHub Issueへのインシデント起票
  - 実施した操作の完全な記録
  - 使用理由・承認不能の経緯の説明
  - 通常プロセスへの切り替え確認

ステップ6: 事後承認と審査
  - ProcessOwner + SystemAdminによる事後承認
  - 必要に応じて追加の是正措置
```

### 7.4 BreakGlass権限者の管理

- 権限保有者：最大2名（SystemAdmin）
- 認証情報の保管：金庫または厳格に管理された安全な場所
- 定期確認：四半期ごとに認証情報の有効性を確認
- 使用記録：すべての使用事例を永久保管

### 7.5 BreakGlass使用の禁止事項

- 通常業務での使用
- 承認プロセスが面倒なための使用
- 未承認変更の実施
- 他者へのBreakGlass認証情報の共有

## 8. AI統治担当者の特権制限

### 8.1 AI統治担当者の特権範囲

AI統治を担当するロール（AIAgentロール含む）には、以下の特権制限を適用する：

| 操作 | AI統治担当（人間） | AIエージェント |
|------|------------------|--------------|
| AIエージェント自律レベルの変更 | ProcessOwner承認後に可能 | 不可（自己変更禁止） |
| AIエージェントへの権限付与 | SystemAdmin承認後に可能 | 不可 |
| AI判断ログの参照 | 可能（Auditorレベル） | 読み取り可能（自己ログのみ） |
| AI判断の承認・却下 | 可能 | 不可（自己承認禁止） |
| BreakGlass使用 | 不可 | 不可 |
| セキュリティポリシーの変更 | SystemAdmin承認後に可能 | 不可 |

### 8.2 AIエージェントの特権統制原則

1. **自己承認禁止**：AIエージェントは自身の提案を自分で承認することを禁止
2. **権限昇格禁止**：AIエージェントは自律的に権限を昇格させることを禁止
3. **スコープ遵守**：定義されたスコープ外の操作を試みた場合は即時停止
4. **透明性確保**：すべてのAI操作は人間が確認可能な形で記録
5. **人間監督必須**：高影響操作には常に人間の最終承認を要求

## 9. コンプライアンスと監査対応

### 9.1 J-SOX対応

本ポリシーは以下のJ-SOX要件に対応する：

| J-SOX要件 | 対応内容 |
|----------|---------|
| IT全般統制 - アクセス管理 | 特権アクセスのJIT管理・棚卸し |
| IT全般統制 - 変更管理 | 特権変更の承認プロセス |
| 職務分離 | BreakGlass権限の限定・承認プロセスの分離 |
| 監査証跡 | 特権操作の完全ログ記録 |

### 9.2 定期監査事項

| 監査項目 | 頻度 | 監査者 |
|---------|------|-------|
| 特権アカウント一覧レビュー | 月次 | Auditor |
| BreakGlass使用履歴レビュー | 都度（使用時） | ProcessOwner |
| 特権操作ログのサンプリング | 四半期 | Auditor |
| JITアクセス申請・承認記録 | 四半期 | Auditor |
| AIエージェント権限の妥当性 | 月次 | AI統治担当 |
