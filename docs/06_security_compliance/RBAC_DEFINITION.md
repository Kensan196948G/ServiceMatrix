Version: 1.0
Status: Active
Classification: Internal / Security
Owner: Service Governance Authority
Last Updated: 2026-03-02

---

## 1. 目的

本文書は、ServiceMatrix における Role-Based Access Control（RBAC）の
ロール定義、権限割当、および管理手順を定義する。

すべてのアクセス制御は本 RBAC 定義に基づいて実施される。

---

## 2. RBAC モデル概要

### 2.1 RBAC の基本構造

```
ユーザー/エージェント
  │
  └─ ロール（1人に複数ロール割当可能）
       │
       └─ 権限セット（ロールごとに定義）
            │
            └─ リソース × 操作（CRUD + 実行）
```

### 2.2 設計原則

- **ロールベース**: 個人ではなくロールに権限を割り当てる
- **最小権限**: 各ロールには必要最小限の権限のみを付与する
- **職務分掌**: 相互牽制が必要な権限は異なるロールに分離する
- **明示的拒否**: 明示的に許可されていない操作はすべて拒否する

## 3. ロール定義

### 3.1 Admin（管理者）

**概要**: ServiceMatrix の全権限を持つ最上位ロール

**責務**:
- システム全体の管理・設定
- ロール割当の管理
- セキュリティポリシーの管理
- 緊急時の対応

**付与条件**:
- プロジェクトオーナーまたは指名された管理者のみ
- 最小人数での運用（原則 2 名以下）

**制約**:
- Admin であっても自己承認は禁止
- すべての Admin 操作は監査ログに記録される

### 3.2 Change Manager（変更管理者）

**概要**: 変更管理プロセスを統括するロール

**責務**:
- 変更要求の承認・却下
- 変更影響の評価
- リリース承認
- 変更スケジュールの管理

**付与条件**:
- 変更管理プロセスの知識を有すること
- ITIL 変更管理の理解があること

**制約**:
- 自身が提出した変更要求の承認は禁止
- 設計変更は Architecture Review Board と協議

### 3.3 Incident Manager（インシデント管理者）

**概要**: インシデント対応を統括するロール

**責務**:
- インシデントの優先度判定
- エスカレーション管理
- インシデント解決の追跡
- 事後レビューの実施

**付与条件**:
- インシデント管理プロセスの知識を有すること
- 緊急対応の経験があること

**制約**:
- インシデント対応のためのコード変更権限はあるが、通常の機能開発は不可

### 3.4 Operator（オペレーター）

**概要**: 日常的な開発・運用を行うロール

**責務**:
- コードの実装・修正
- テストの作成・実行
- PR の作成・更新
- ドキュメントの作成・更新

**付与条件**:
- プロジェクトの開発メンバーであること

**制約**:
- 自身の PR の承認は不可
- セキュリティ設定の変更は不可
- 本番環境への直接アクセスは不可

### 3.5 Viewer（閲覧者）

**概要**: 閲覧専用のロール

**責務**:
- ドキュメント・コードの閲覧
- Issue・PR のコメント
- レポートの閲覧

**付与条件**:
- プロジェクトへの閲覧アクセスが承認されていること

**制約**:
- コード・ドキュメントの編集は不可
- 設定変更は不可

### 3.6 AI Agent（AI エージェント）

**概要**: AI エージェント専用のロール

**責務**:
- コード分析・レビュー
- 提案の生成
- 承認後のコード修正
- テスト実行

**付与条件**:
- 自律レベルが設定されていること
- AI_GOVERNANCE_POLICY に準拠していること

**制約**:
- 自律レベルに応じた操作制限（AI_AUTONOMY_LEVEL_MATRIX 参照）
- 人間承認なしの commit/push は L3 以上のみ
- merge は不可
- セキュリティ設定の変更は不可
- main ブランチへの直接変更は不可

## 5. GitHub Teams との対応

### 5.1 チーム構成

| ServiceMatrix ロール | GitHub Team | Repository 権限 |
|---------------------|------------|-----------------|
| Admin | @org/sm-admins | Admin |
| Change Manager | @org/sm-change-managers | Maintain |
| Incident Manager | @org/sm-incident-managers | Triage + Write |
| Operator | @org/sm-operators | Write |
| Viewer | @org/sm-viewers | Read |
| AI Agent | (GitHub App / Bot Account) | Write* |

*: Branch Protection Rules により制限

### 5.2 チーム階層

```
@org/sm-all（全メンバー）
  ├─ @org/sm-admins
  ├─ @org/sm-change-managers
  ├─ @org/sm-incident-managers
  ├─ @org/sm-operators
  └─ @org/sm-viewers
```

## 6. ロール変更手順と承認要件

### 6.1 ロール付与手順

```
1. ロール付与申請
   ├─ 申請者: 対象者の上長または Admin
   ├─ 申請先: GitHub Issue（ロール変更テンプレート使用）
   └─ 記載事項: 対象者、付与ロール、理由、期間

2. 承認
   ├─ 承認者: Admin（Viewer/Operator の場合）
   ├─ 承認者: Admin + Change Manager（Change Manager 以上の場合）
   └─ 承認記録: Issue コメントに記録

3. 適用
   ├─ GitHub Teams への追加
   ├─ ServiceMatrix ロール設定の更新
   └─ 適用日時の記録

4. 通知
   ├─ 対象者への通知
   ├─ 関連チームへの通知
   └─ 監査ログへの記録
```

### 6.2 ロール変更（昇格・降格）手順

```
1. 変更申請
   ├─ 現在のロールと変更後のロールを明記
   ├─ 変更理由を記載
   └─ 影響範囲を分析

2. 承認
   ├─ 昇格: Admin の承認必須
   ├─ 降格: Admin の承認必須（懲戒降格の場合は即時適用後に記録）
   └─ 承認記録の保存

3. 適用・通知
   ├─ GitHub Teams メンバーシップの変更
   ├─ ServiceMatrix ロール設定の更新
   └─ 関係者への通知
```

### 6.3 ロール剥奪手順

```
1. 即時剥奪が必要な場合
   ├─ 退職・異動
   ├─ セキュリティインシデント
   └─ 懲戒処分

2. 手順
   ├─ GitHub Teams からの除外
   ├─ アクティブセッションの無効化
   ├─ API トークンの無効化
   └─ 監査ログへの記録

3. 事後処理
   ├─ 剥奪記録の保存
   ├─ 引継ぎ事項の確認
   └─ 残存アクセスの確認
```

## 7. ロールの兼務ルール

### 7.1 兼務可能な組み合わせ

| ロール A | ロール B | 兼務 | 理由 |
|---------|---------|------|------|
| Admin | Change Manager | 可（非推奨） | 職務分掌の観点から非推奨 |
| Admin | Operator | 可（非推奨） | 自己レビューリスク |
| Change Manager | Operator | 不可 | 変更承認と変更実施の分離 |
| Incident Manager | Operator | 可 | インシデント対応の迅速性 |
| Operator | Viewer | 不要 | Operator は Viewer の権限を包含 |

### 7.2 兼務時の制約

- 兼務ロールであっても、職務分掌の原則は維持する
- 兼務ロールの一方で作成した成果物を、もう一方のロールで承認することは禁止する
- 兼務の状態は定期的に見直し、不要な兼務は解消する

## 8. ロール棚卸し

### 8.1 棚卸し頻度

| 対象 | 頻度 | 実施者 |
|------|------|--------|
| 全ロール割当 | 四半期 | Admin |
| 特権ロール（Admin/Change Manager） | 月次 | Admin |
| AI Agent ロール | 月次 | Governance Team |
| 退任者チェック | 発生時即時 | Admin |

### 8.2 棚卸し手順

1. 現在のロール割当一覧を出力する
2. 各メンバーのロールの妥当性を確認する
3. 不要なロールを特定する
4. 変更が必要な場合はロール変更手順に従う
5. 棚卸し結果を記録する
