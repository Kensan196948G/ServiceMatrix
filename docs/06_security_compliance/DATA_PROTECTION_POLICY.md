# データ保護ポリシー（Data Protection Policy）

ServiceMatrix データ保護統治仕様
Version: 1.0
Status: Active
Classification: Confidential - Internal Governance Document
Applicable Standard: 個人情報の保護に関する法律（個人情報保護法）/ GDPR参考 / ISO/IEC 27001
Last Updated: 2026-03-02
Owner: データ保護担当 / セキュリティ統治担当

---

## 1. 目的

本ドキュメントは、ServiceMatrixが取り扱うデータの保護に関する
方針・要件・手順を定義する。

個人情報保護法・GDPR（参考）・ISO/IEC 27001に準拠し、
データの機密性・完全性・可用性を確保することを目的とする。

---

## 2. 個人情報取扱い方針

### 2.1 個人情報の定義

ServiceMatrixが取り扱う個人情報の範囲：

| 情報種別 | 具体例 | 格納箇所 |
|---------|-------|---------|
| 利用者識別情報 | ユーザーID・氏名・メールアドレス | PostgreSQL ユーザーテーブル |
| 認証情報 | パスワードハッシュ・MFAシード | PostgreSQL 認証テーブル |
| 操作履歴 | 誰がいつ何を操作したかの記録 | 監査ログ |
| GitHub連携情報 | GitHubユーザー名・トークン（ハッシュ） | 設定ストア |
| 連絡先情報 | エスカレーション先電話番号・メール | 設定ドキュメント |

### 2.2 個人情報の収集原則

1. **目的の特定**：収集前に利用目的を特定・明示する
2. **必要最小限の収集**：業務遂行に必要な最小限の個人情報のみ収集する
3. **同意の取得**：収集時に本人の同意を得る（可能な場合）
4. **適正な取得**：偽りの手段による取得をしない

### 2.3 個人情報の利用原則

1. 収集時に特定した目的の範囲内でのみ利用する
2. 目的外利用は原則禁止（例外：法令に基づく場合等）
3. 利用目的が変更された場合は本人への通知または同意取得を行う

## 3. データ分類

### 3.1 データ分類定義

ServiceMatrixは取り扱うデータを以下の3分類で管理する：

| 分類 | 定義 | 具体例 |
|------|------|-------|
| **機密（Confidential）** | 外部への漏洩・不正アクセスにより重大な損害が生じるデータ | 個人情報・認証情報・セキュリティ設定・暗号鍵 |
| **社外秘（Internal）** | 組織内部でのみ共有されるべき業務データ | インシデント詳細・変更内容・SLA計測データ・設計文書 |
| **一般（Public）** | 公開可能または公開されているデータ | 公開ドキュメント・README・サービスカタログ概要 |

### 3.2 分類別取扱い要件

| 要件項目 | 機密 | 社外秘 | 一般 |
|---------|------|--------|------|
| 暗号化（保存時） | 必須（AES-256） | 推奨 | 不要 |
| 暗号化（転送時） | 必須（TLS 1.3） | 必須（TLS 1.3） | 推奨 |
| アクセス制御 | 個人単位での制限 | ロールベース制限 | 制限なし |
| ログ記録 | 全アクセス記録 | 更新操作記録 | 不要 |
| バックアップ | 日次・オフサイト | 日次 | 任意 |
| 廃棄方法 | 安全削除（Crypto Erase） | 安全削除 | 通常削除 |
| 第三者提供 | 原則禁止（特別条件のみ） | 要承認 | 自由 |

### 3.3 データ分類の適用手順

1. 新規データ作成・登録時にデータオーナーが分類を判断
2. 分類が不明な場合は「機密」として暫定分類し、後にレビュー
3. データ分類の変更は変更管理プロセスに従う

## 4. 暗号化要件

### 4.1 保存時暗号化（Encryption at Rest）

| 対象データ | 暗号化方式 | 鍵管理 |
|----------|----------|-------|
| PostgreSQL 機密データ列 | AES-256（列レベル暗号化） | アプリケーション鍵管理 |
| PostgreSQL データベース全体 | AES-256（ストレージレベル） | インフラ鍵管理 |
| バックアップファイル | AES-256 | 専用バックアップ鍵 |
| ログファイル（機密含む） | AES-256 | ログ管理鍵 |
| 設定ファイル（認証情報含む） | 環境変数 / シークレット管理ツール | GitHub Secrets / Vault |

### 4.2 転送時暗号化（Encryption in Transit）

| 通信経路 | 要件 |
|---------|------|
| クライアント ↔ APIサーバー | TLS 1.3以上（TLS 1.0 / 1.1は禁止） |
| APIサーバー ↔ データベース | TLS 1.3以上（同一VPC内でも適用） |
| サービス間通信 | mTLS（相互TLS認証） |
| GitHubとの通信 | HTTPS / SSH（SHA-256以上） |
| バックアップ転送 | TLS 1.3以上 |

### 4.3 暗号化鍵管理

| 鍵種別 | ローテーション頻度 | 保管方法 |
|--------|--------------|---------|
| マスター暗号化鍵 | 年次 | HSM または安全な鍵管理サービス |
| アプリケーション暗号化鍵 | 四半期 | シークレット管理ツール |
| TLS証明書 | 年次（または自動更新） | 証明書管理サービス |
| APIトークン / シークレット | 90日 | GitHub Secrets |

## 5. データ所在地管理

### 5.1 データ所在地の定義

| データ種別 | 所在地 | 管理方針 |
|----------|-------|---------|
| 本番データベース | 指定データセンター / クラウドリージョン | 特定リージョンに限定 |
| バックアップデータ | 本番と別リージョン | 国内別リージョンを優先 |
| GitHubリポジトリ | GitHub（米国）サーバー | GitHub利用規約に準拠 |
| ログデータ | 本番と同一リージョン + オフサイトバックアップ | 保管期間に応じた管理 |

### 5.2 国外移転の管理

個人情報を国外に移転する場合（GitHubへのコード管理等）は：

1. 移転先国の個人情報保護水準を確認する
2. 必要に応じて本人の同意を取得する
3. 移転に関する記録を保管する

### 5.3 クラウドサービス利用時のデータ管理

| クラウドサービス | 格納データ種別 | 注意事項 |
|--------------|------------|---------|
| GitHub | ソースコード・設定文書 | 機密情報のプッシュ禁止 / .gitignore設定 |
| GitHub Actions | CI/CDログ・Secrets | Secretsのマスク設定必須 |
| クラウドDB | 業務データ | 暗号化 + リージョン制限 |

## 6. 第三者提供制限

### 6.1 第三者提供の原則

個人情報の第三者提供は原則として禁止とし、以下の場合にのみ例外的に許可する：

| 例外事由 | 条件 |
|---------|------|
| 本人の同意 | 明示的な同意取得と記録 |
| 法令に基づく場合 | 法令の根拠を明示・記録 |
| 委託先への提供 | 委託契約における安全管理義務の明示 |
| 共同利用 | 共同利用の範囲・目的・管理者を事前に通知 |

### 6.2 委託先管理

個人情報の取扱いを外部委託する場合：

1. 委託先の安全管理体制を事前に確認する
2. 委託契約に個人情報保護条項を含める
3. 委託先の管理状況を定期的に確認する（年次監査）
4. 委託先一覧と監査結果を記録・保管する

### 6.3 AIサービスへのデータ提供制限

AIエージェント・外部AIサービスへのデータ提供に関する特別規定：

| 制限事項 | 内容 |
|---------|------|
| 個人情報の外部AI送信禁止 | 個人情報を含むデータを外部AIサービスに送信しない |
| 機密データの分析禁止 | 機密分類のデータをAI分析の入力に使用しない |
| データ最小化 | AIへの入力データは目的達成に必要な最小限に留める |
| ログ記録 | AIへのデータ提供は全件記録する |

## 7. データ削除手順（Right to be Forgotten対応）

### 7.1 データ削除の対象

以下の場合にデータ削除を実施する：

| 削除事由 | 対象 | 期限 |
|---------|------|------|
| 本人からの削除要求 | 対象者の個人情報 | 要求受付後30日以内 |
| 利用目的の達成 | 目的に関連するデータ | 達成確認後速やかに |
| 同意の撤回 | 同意に基づき収集したデータ | 撤回確認後速やかに |
| 保管期間満了 | 期間超過データ | 四半期バッチ処理 |
| 退職・契約終了 | 対象者のユーザー情報 | 最終勤務日から90日後 |

### 7.2 削除手順

```
ステップ1: 削除要求の受付
  - 本人確認を実施
  - 削除対象データの範囲を確認
  - GitHub Issueで削除リクエストを記録

ステップ2: 削除可否の判断
  - 法令保管義務（J-SOX等）との照合
  - 進行中の法的手続きとの確認
  - 削除可能な場合は承認

ステップ3: データの特定
  - 削除対象データがどのシステム・テーブルに存在するかを特定
  - バックアップ・ログへの影響確認

ステップ4: 削除実施
  - 本番データベースからの削除
  - バックアップからの削除スケジュール設定
  - ログへの対応（個人情報部分の匿名化）

ステップ5: 削除確認
  - 削除完了の確認テスト
  - 関連システムへの反映確認

ステップ6: 記録と通知
  - 削除完了の記録（削除実施記録自体は保管）
  - 本人への削除完了通知
```

### 7.3 削除困難なデータの処置

以下のデータは即時削除が困難なため、代替措置を適用する：

| データ種別 | 理由 | 代替措置 |
|----------|------|---------|
| 監査ログ内の個人情報 | J-SOX保管要件 | 匿名化または仮名化処理 |
| バックアップテープ | 上書き困難 | 暗号化鍵の削除（Crypto Erase） |
| Gitコミット履歴 | Git特性上の困難 | git-filter-branchによる除去 + 全クローンの更新 |

## 8. インシデント時のデータ漏洩対応

### 8.1 データ漏洩インシデントの分類

| 分類 | 定義 | 対応レベル |
|------|------|---------|
| レベル1（軽微） | 個人情報が含まない内部データの漏洩 | 内部対応のみ |
| レベル2（中程度） | 少数の個人情報（100件未満）の漏洩 | 内部対応 + 本人通知検討 |
| レベル3（重大） | 大規模個人情報漏洩（100件以上）または機密情報漏洩 | 規制当局報告 + 本人通知 |

### 8.2 データ漏洩発生時の対応フロー

```
検知
  ↓
初期封じ込め（30分以内）
  ├─ 漏洩経路の遮断
  ├─ 影響範囲の初期確認
  └─ セキュリティ担当への即時報告

インシデント評価（2時間以内）
  ├─ 漏洩データの種別・件数の確認
  ├─ 漏洩先の特定（可能な範囲で）
  └─ 被害範囲の評価

対応判断（4時間以内）
  ├─ レベル3の場合 → 規制当局への報告検討（72時間以内）
  ├─ 本人通知の要否判断
  └─ 外部専門家（弁護士・セキュリティ専門家）への相談

是正措置
  ├─ 漏洩の完全な封じ込め
  ├─ データの回収・削除依頼（可能な範囲で）
  └─ 再発防止策の実施

報告・記録
  ├─ 内部インシデントレポートの作成
  ├─ 規制当局への報告（該当する場合）
  ├─ 本人への通知（該当する場合）
  └─ 全プロセスの記録保管
```

### 8.3 規制当局への報告要件

個人情報保護法に基づく報告義務：

| 報告要件 | 期限 | 報告先 |
|---------|------|-------|
| 個人情報保護委員会への速報 | 漏洩認識後3〜5日以内 | 個人情報保護委員会 |
| 個人情報保護委員会への確報 | 30日以内（不正目的の場合60日） | 個人情報保護委員会 |
| 本人への通知 | 可能な限り速やかに | 被害者本人 |

### 8.4 漏洩後の再発防止プロセス

1. 根本原因分析の実施
2. 技術的対策の強化
3. 組織的・管理的対策の見直し
4. 再発防止策の実施
5. 実施後の効果確認テスト
6. 全関係者への教育・訓練

## 9. データ保護影響評価（DPIA）

### 9.1 DPIA実施基準

以下に該当する処理を行う場合、DPIAを実施する：

- 大規模な個人情報の処理を開始する場合
- 個人情報処理システムの大規模な変更
- 新しい技術（AI等）を個人情報処理に活用する場合
- 機密性の高い個人情報の処理

### 9.2 DPIA手順

1. 処理の説明と必要性の評価
2. 処理の目的と比例性の評価
3. リスクの特定と評価
4. リスク対策の検討と実装
5. DPIAドキュメントの作成と承認
6. 定期的なレビューと更新
