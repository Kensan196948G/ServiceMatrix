# J-SOX内部統制マッピング（J-SOX Internal Control Mapping）

ServiceMatrix J-SOX統制仕様
Version: 1.0
Status: Active
Classification: Confidential - Internal Governance Document
Applicable Standard: 金融商品取引法（J-SOX）/ PCAOB AS 2201 参考
Last Updated: 2026-03-02
Owner: 内部統制担当 / Auditor

---

## 1. 目的

本ドキュメントは、ServiceMatrixにおけるJ-SOX（金融商品取引法に基づく内部統制報告制度）への
対応状況を文書化し、IT全般統制（ITGC）とIT業務処理統制（ITAC）の
マッピングを明示する。

経営者による内部統制評価を支援し、外部監査人への証跡提供を可能にする。

---

## 2. J-SOX適用範囲

### 2.1 適用範囲の定義

ServiceMatrixは以下の範囲においてJ-SOX内部統制の対象となる：

| 適用範囲 | 対象内容 |
|---------|---------|
| ITシステム範囲 | ServiceMatrix本体（Python/FastAPI/PostgreSQL/Next.js） |
| データ範囲 | 財務報告に影響を与えるサービスマネジメントデータ |
| プロセス範囲 | 変更管理・アクセス管理・IT運用管理 |
| 組織範囲 | ServiceMatrix利用部門全体 |

### 2.2 適用外の範囲

- 財務会計システムとの直接統合（現時点では統合なし）
- 個人利用端末のローカル環境

## 3. IT全般統制（ITGC）マッピング

IT全般統制は「ITシステムが信頼性高く動作するための基盤的な統制」である。

### 3.1 アクセス管理統制

**統制目的：** 適切な権限を持つ者のみがシステムにアクセスできることを保証する

| 統制ID | 統制内容 | ServiceMatrix実装 | 証跡 | テスト方法 |
|--------|---------|-----------------|------|---------|
| ITGC-AC-01 | ユーザーアカウントの承認付与 | RBAC_DEFINITION.md / 権限申請プロセス | 申請・承認GitHub Issue | 申請記録のサンプリング確認 |
| ITGC-AC-02 | 退職・異動時の権限即時剥奪 | 退職者処理手順 / HR連携プロセス | 権限変更ログ（監査ログ） | 退職者リストとアカウント状況の照合 |
| ITGC-AC-03 | 特権アクセスの制限と監視 | PRIVILEGED_ACCESS_POLICY.md / JIT管理 | 特権操作ログ | 特権使用ログのサンプリングレビュー |
| ITGC-AC-04 | 権限の定期棚卸し（四半期） | アクセス権レビュープロセス | 棚卸し結果報告書 | 棚卸し実施記録の確認 |
| ITGC-AC-05 | 職務分離の実施 | RBAC設計 / 承認ゲート設計 | ロール割当記録 | 相互排他ロールの同時付与がないことの確認 |
| ITGC-AC-06 | AIエージェントアクセス管理 | AI_GOVERNANCE_POLICY.md / AIロール定義 | AIエージェント操作ログ | AIエージェント権限の妥当性確認 |

### 3.2 変更管理統制

**統制目的：** システムへの変更が適切に承認・テスト・実施され、不正変更を防止する

| 統制ID | 統制内容 | ServiceMatrix実装 | 証跡 | テスト方法 |
|--------|---------|-----------------|------|---------|
| ITGC-CM-01 | すべての変更に変更申請を要求 | CHANGE_MANAGEMENT_PROCESS.md / GitHub PR必須 | GitHub PR / Issue記録 | 変更のPRなし直コミットがないことの確認 |
| ITGC-CM-02 | 変更の承認者・申請者の分離 | PR承認ルール（自己承認禁止） | GitHub PR承認記録 | 申請者と承認者の同一性チェック |
| ITGC-CM-03 | 本番適用前のテスト実施 | GitHub Actions CI / テストパイプライン | CIテスト結果ログ | CI通過なしのマージがないことの確認 |
| ITGC-CM-04 | 緊急変更の事後承認管理 | 緊急変更プロセス / 事後承認手順 | 緊急変更Issue + 事後承認記録 | 緊急変更の承認完了率確認 |
| ITGC-CM-05 | 変更ロールバック手順 | BACKUP_AND_RECOVERY_MODEL.md | ロールバック実施記録 | ロールバック手順の存在と実施能力テスト |
| ITGC-CM-06 | AI提案変更の人間承認 | AI統治承認ゲート | AI提案・承認ログ | AI提案の全件に人間承認が付与されていること |

### 3.3 IT運用管理統制

**統制目的：** システムが安定して稼働し、問題が適切に管理されることを保証する

| 統制ID | 統制内容 | ServiceMatrix実装 | 証跡 | テスト方法 |
|--------|---------|-----------------|------|---------|
| ITGC-OP-01 | インシデント管理プロセス | INCIDENT_MANAGEMENT_PROCESS.md | インシデントチケット | インシデント記録の完全性確認 |
| ITGC-OP-02 | 問題管理プロセス | PROBLEM_MANAGEMENT_PROCESS.md | 問題チケット / KEDB | 問題の根本原因分析記録の確認 |
| ITGC-OP-03 | バックアップの定期実施と確認 | BACKUP_AND_RECOVERY_MODEL.md | バックアップ実施ログ | バックアップ成功率・リストアテスト記録 |
| ITGC-OP-04 | 可用性監視 | SLA_DEFINITION.md / 監視設計 | 可用性メトリクス | SLA達成状況の確認 |
| ITGC-OP-05 | キャパシティ管理 | CAPACITY_MANAGEMENT.md | リソース使用量ログ | キャパシティ閾値超過アラートの対応記録 |
| ITGC-OP-06 | 運用ログの保管 | AUDIT_LOG_RETENTION_POLICY.md | 各種運用ログ | ログ保管状況・完全性の確認 |

### 3.4 システム開発管理統制

**統制目的：** システム開発・変更が統制されたプロセスで実施されることを保証する

| 統制ID | 統制内容 | ServiceMatrix実装 | 証跡 | テスト方法 |
|--------|---------|-----------------|------|---------|
| ITGC-SD-01 | 開発環境と本番環境の分離 | Branch戦略 / 環境設計 | GitHub Branch保護ルール | 本番へのdirect push禁止の確認 |
| ITGC-SD-02 | コードレビュー必須 | PULL_REQUEST_POLICY.md | PR承認記録 | レビューなしマージがないことの確認 |
| ITGC-SD-03 | 自動テストの実施 | GitHub Actions CI設定 | CIテスト結果 | CI必須チェックの設定確認 |
| ITGC-SD-04 | リリース管理 | 14_release_management | リリース記録 | リリースプロセスの遵守確認 |
| ITGC-SD-05 | セキュリティテスト | SAST/DAST設定 | セキュリティスキャン結果 | セキュリティチェックのCI組み込み確認 |

## 4. IT業務処理統制（ITAC）マッピング

IT業務処理統制は「業務プロセスに組み込まれた自動化された統制」である。

### 4.1 完全性統制

**統制目的：** 処理されるべきすべての取引・イベントが漏れなく処理されること

| 統制ID | 統制内容 | ServiceMatrix実装 | 検証方法 |
|--------|---------|-----------------|---------|
| ITAC-CO-01 | インシデント全件記録の保証 | GitHub Issue必須化 / APIバリデーション | レコード数の連番確認・欠番チェック |
| ITAC-CO-02 | 変更全件の追跡 | GitHub PRとIssueの連携必須 | 未紐付けPRの不存在確認 |
| ITAC-CO-03 | SLAタイマーの自動起動 | インシデント登録時の自動タイムスタンプ | タイムスタンプ設定の確認テスト |

### 4.2 正確性統制

**統制目的：** 処理されるデータが正確であること

| 統制ID | 統制内容 | ServiceMatrix実装 | 検証方法 |
|--------|---------|-----------------|---------|
| ITAC-AC-01 | 入力データバリデーション | FastAPIスキーマバリデーション | バリデーションロジックのコードレビュー |
| ITAC-AC-02 | ステータス遷移制御 | 有限状態機械による遷移管理 | 不正遷移の試みに対するエラー応答確認 |
| ITAC-AC-03 | SLA計算ロジックの正確性 | SLA_CALCULATION_LOGIC.md / 自動計算 | テストケースによる計算結果検証 |

### 4.3 制限的処理統制

**統制目的：** 二重処理・不正処理の防止

| 統制ID | 統制内容 | ServiceMatrix実装 | 検証方法 |
|--------|---------|-----------------|---------|
| ITAC-RE-01 | 重複インシデント検知 | 重複検出ロジック | 同一インシデントの二重登録テスト |
| ITAC-RE-02 | 承認の二重実施防止 | 承認状態管理 | 承認済みレコードへの再承認試行テスト |
| ITAC-RE-03 | AIエージェントの自己承認防止 | 承認ゲート設計 | AI提案者によるセルフ承認試行テスト |

### 4.4 承認統制

**統制目的：** 適切な権限を持つ者による承認の実施

| 統制ID | 統制内容 | ServiceMatrix実装 | 検証方法 |
|--------|---------|-----------------|---------|
| ITAC-AP-01 | 変更の承認ゲート | RBAC + 承認ワークフロー | 権限なしユーザーによる承認試行テスト |
| ITAC-AP-02 | P1インシデントの自動エスカレーション | 自動エスカレーションロジック | 閾値到達後のエスカレーション発動確認 |
| ITAC-AP-03 | SLA違反時の自動通知 | SLA_BREACH_HANDLING_MODEL.md | 通知送信ログの確認 |

## 5. 統制テスト計画

### 5.1 テスト種別

| テスト種別 | 説明 | 頻度 |
|----------|------|------|
| 設計有効性テスト | 統制が設計どおりに存在するかの確認 | 年次（または統制変更時） |
| 運用有効性テスト | 統制が期間を通じて正しく機能しているかの確認 | 半期（または年次） |
| 再実施テスト | 以前の問題が是正されたかの確認 | 問題発見後随時 |

### 5.2 統制テスト年間スケジュール

| 時期 | テスト対象 | テスト種別 | 担当 |
|------|----------|---------|------|
| 4月 | アクセス管理統制（ITGC-AC） | 設計有効性 + 運用有効性 | Auditor |
| 6月 | 変更管理統制（ITGC-CM） | 設計有効性 + 運用有効性 | Auditor |
| 8月 | IT運用管理統制（ITGC-OP） | 運用有効性 | Auditor |
| 10月 | システム開発管理統制（ITGC-SD）+ ITAC | 設計有効性 + 運用有効性 | Auditor |
| 12月 | 全統制の年次総合テスト | 設計有効性 + 運用有効性 | Auditor + 外部監査人 |

### 5.3 テストサンプルサイズ基準

| 対象期間の母集団 | サンプルサイズ |
|--------------|------------|
| 1件 | 1件（全件） |
| 2〜9件 | 全件 |
| 10〜24件 | 5件 |
| 25〜99件 | 15件 |
| 100〜499件 | 25件 |
| 500件以上 | 40件 |

## 6. 経営者評価サポート機能

### 6.1 ServiceMatrixが提供する経営者評価支援

ServiceMatrixは以下の機能により経営者による内部統制評価を支援する：

| 支援機能 | 内容 | 提供方法 |
|---------|------|---------|
| 統制証跡の自動収集 | GitHub Issue / PRの操作記録の自動保管 | GitHubリポジトリ |
| KPI・SLAレポート | 内部統制有効性の定量的証拠 | KPI_DEFINITION.md / 月次レポート |
| 異常検知アラート | 統制失敗の早期発見 | 自動アラート機能 |
| AI統治レポート | AIエージェントの操作透明性証明 | AI判断ログ |
| ハッシュチェーン監査 | 証跡改ざんのないことの証明 | AUDIT_LOG_RETENTION_POLICY.md |

### 6.2 経営者評価レポートの構成要素

年次の経営者内部統制評価報告書に含めるべき情報：

```
1. 内部統制の概要
   - 適用範囲
   - 統制フレームワーク（ITGC / ITAC）
   - 評価期間

2. IT全般統制の評価結果
   - アクセス管理統制：評価結果サマリー
   - 変更管理統制：評価結果サマリー
   - IT運用管理統制：評価結果サマリー
   - システム開発管理統制：評価結果サマリー

3. IT業務処理統制の評価結果
   - 各ITAC統制の有効性評価

4. 発見された問題点と是正措置
   - 統制上の弱点（Control Deficiency）
   - 重要な弱点（Significant Deficiency）
   - 実施した是正措置

5. 結論
   - 内部統制は有効に機能している / 重要な弱点がある

6. 添付資料
   - 統制テスト結果一覧
   - 統制証跡サンプル
```

### 6.3 統制上の問題発見時のエスカレーション

| 問題レベル | 定義 | 報告先 | 期限 |
|----------|------|-------|------|
| 軽微な問題（Control Deficiency） | 統制に軽微な問題があるが財務報告への影響は低い | ProcessOwner | 発見後30日以内 |
| 重要な弱点（Significant Deficiency） | 財務報告に悪影響を与える可能性がある | 経営層 + 外部監査人 | 発見後5営業日以内 |
| 重大な弱点（Material Weakness） | 財務報告に重大な虚偽表示をもたらす可能性がある | 経営者 + 監査役 + 外部監査人 | 即日報告 |
