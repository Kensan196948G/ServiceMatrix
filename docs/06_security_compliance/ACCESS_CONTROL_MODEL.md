# アクセス制御モデル（Access Control Model）

Version: 1.0
Status: Active
Classification: Internal / Security
Owner: Service Governance Authority
Last Updated: 2026-03-02

---

## 1. 目的

本文書は、ServiceMatrix におけるアクセス制御の設計原則、
制御方式、および運用ルールを定義する。

ServiceMatrix は多層防御の原則に基づき、
すべてのリソースへのアクセスを適切に制御する。

---

## 2. アクセス制御の基本原則

### 2.1 最小権限の原則（Principle of Least Privilege）

すべてのユーザーおよびエージェントは、
業務遂行に必要な最小限の権限のみを付与される。

- 不要な権限は付与しない
- 一時的に必要な権限は期限付きで付与する
- 権限の棚卸しを定期的に実施する

### 2.2 職務分掌の原則（Segregation of Duties）

相互牽制が必要な業務は、異なる担当者（またはロール）に分離する。

| 分離すべき業務の組み合わせ | 理由 |
|--------------------------|------|
| 変更要求の提出 と 変更の承認 | 自己承認防止 |
| コード作成 と コードレビュー | 品質担保 |
| デプロイ実行 と デプロイ承認 | リリース統制 |
| アクセス権設定 と アクセス権承認 | 権限管理統制 |
| 監査証跡の作成 と 監査証跡の変更 | 証跡保全 |

### 2.3 Need-to-Know の原則

情報へのアクセスは、業務上知る必要がある者にのみ許可する。

- 機密分類に応じたアクセス制御を適用する
- プロジェクト間の情報隔離を維持する
- 退任時は速やかにアクセスを無効化する

## 3. アクセス制御方式

### 3.1 制御方式の階層

ServiceMatrix は以下の 3 層でアクセス制御を実施する。

```
┌─────────────────────────────────────────────┐
│  Layer 1: GitHub ネイティブ制御               │
│  ├─ Repository 権限（Admin/Write/Read）      │
│  ├─ Branch Protection Rules                  │
│  ├─ CODEOWNERS                               │
│  └─ GitHub Teams                             │
├─────────────────────────────────────────────┤
│  Layer 2: ServiceMatrix ロール制御            │
│  ├─ RBAC（Role-Based Access Control）        │
│  ├─ 操作別権限マトリクス                      │
│  └─ ロール別ビュー制御                        │
├─────────────────────────────────────────────┤
│  Layer 3: AI エージェント制御                  │
│  ├─ 自律レベル制御（L0-L4）                   │
│  ├─ 承認ゲート制御                            │
│  └─ 操作種別制御                              │
└─────────────────────────────────────────────┘
```

### 3.2 Layer 1: GitHub ネイティブ制御

GitHub の標準機能を利用したアクセス制御。

#### Repository 権限

| 権限レベル | 説明 | 対象ロール |
|-----------|------|----------|
| Admin | リポジトリの全権限 | Admin |
| Maintain | ブランチ管理・設定変更 | Change Manager |
| Write | コード push・PR 作成 | Operator |
| Triage | Issue 管理・ラベル操作 | Incident Manager |
| Read | 閲覧のみ | Viewer |

#### Branch Protection Rules

main ブランチに対して以下の保護ルールを適用する。

- PR 必須（直接 push 禁止）
- 最低 1 名のレビュー承認必須
- CI ステータスチェック必須
- 管理者も保護ルール適用
- force push 禁止
- ブランチ削除禁止

#### CODEOWNERS

重要なファイルに対して CODEOWNERS を設定する。

```
# セキュリティ関連
docs/06_security_compliance/   @security-team
.github/workflows/             @devops-team

# 統治関連
docs/01_governance/            @governance-team
CLAUDE.md                      @admin-team

# アーキテクチャ関連
docs/02_architecture/          @architecture-team
```

### 3.3 Layer 2: ServiceMatrix ロール制御

RBAC に基づくアプリケーションレベルのアクセス制御。
詳細は `RBAC_DEFINITION.md` を参照。

### 3.4 Layer 3: AI エージェント制御

AI エージェントの操作権限は自律レベルに基づいて制御される。
詳細は `AI_AUTONOMY_LEVEL_MATRIX.md` を参照。

## 4. 認証モデル

### 4.1 人間ユーザーの認証

| 認証方式 | 用途 | 必須/推奨 |
|---------|------|----------|
| GitHub SSO | リポジトリアクセス | 必須 |
| 二要素認証（2FA） | GitHub アカウント保護 | 必須 |
| SSH キー認証 | Git 操作 | 推奨 |
| GPG 署名 | コミット署名 | 推奨 |

### 4.2 AI エージェントの認証

| 認証方式 | 用途 | 必須/推奨 |
|---------|------|----------|
| API トークン | GitHub API アクセス | 必須 |
| セッション管理 | エージェント識別 | 必須 |
| エージェント ID | ログ・監査用識別 | 必須 |

### 4.3 サービスアカウントの認証

| 認証方式 | 用途 | 必須/推奨 |
|---------|------|----------|
| GitHub App | CI/CD パイプライン | 必須 |
| Deploy Key | 自動デプロイ | 条件付き |
| Webhook Secret | Webhook 検証 | 必須 |

## 5. 認可モデル

### 5.1 認可フロー

```
リクエスト受信
  │
  ├─ 認証確認
  │   ├─ 認証済み → 認可チェックへ
  │   └─ 未認証 → アクセス拒否
  │
  ├─ ロール特定
  │   ├─ ユーザーの所属ロールを取得
  │   └─ AI エージェントの自律レベルを取得
  │
  ├─ 権限チェック
  │   ├─ ロール別権限マトリクスを参照
  │   ├─ リソース別アクセス制御を参照
  │   └─ コンテキスト条件を確認（時間帯、IP 等）
  │
  ├─ 判定
  │   ├─ 許可 → 操作実行
  │   ├─ 承認必要 → 承認ゲートへ
  │   └─ 拒否 → アクセス拒否 + ログ記録
  │
  └─ 監査ログ記録
      └─ すべてのアクセス判定結果を記録
```

### 5.2 リソース分類

| 分類 | 例 | 機密度 |
|------|-----|-------|
| Public | README.md、公開ドキュメント | 低 |
| Internal | 設計文書、アーキテクチャ文書 | 中 |
| Confidential | セキュリティポリシー、監査証跡 | 高 |
| Restricted | 認証情報、暗号鍵、環境変数 | 最高 |

### 5.3 機密度別アクセス制御

| 機密度 | 閲覧 | 編集 | 削除 | AI アクセス |
|--------|------|------|------|------------|
| Public | 全ロール | Operator 以上 | Admin のみ | L1 以上 |
| Internal | Viewer 以上 | Operator 以上 | Admin のみ | L1 以上 |
| Confidential | 特定ロール | Change Manager 以上 | Admin のみ | L1（閲覧のみ） |
| Restricted | Admin のみ | Admin のみ | Admin のみ | アクセス不可 |

## 6. セッション管理

### 6.1 セッションポリシー

| 項目 | 設定値 | 理由 |
|------|--------|------|
| セッションタイムアウト | 8 時間 | 業務時間ベース |
| 非活動タイムアウト | 30 分 | セキュリティリスク低減 |
| 同時セッション数 | 最大 3 | リソース制御 |
| セッション記録 | 必須 | 監査対応 |

### 6.2 AI セッション管理

| 項目 | 設定値 | 理由 |
|------|--------|------|
| Agent Teams 最大同時数 | 設定による | リソース・コスト制御 |
| SubAgent 最大同時数 | 制限なし | 軽量のため |
| セッションログ保持期間 | 90 日 | 監査対応 |

## 7. 特権アクセス管理

特権アクセスの詳細は `PRIVILEGED_ACCESS_POLICY.md` を参照。

### 7.1 特権アクセスの定義

以下の操作を特権アクセスとして分類する。

- リポジトリ設定の変更
- Branch Protection Rules の変更
- GitHub Teams メンバーシップの変更
- CI/CD パイプラインの変更
- セキュリティポリシーの変更
- 監査ログの閲覧・エクスポート

### 7.2 特権アクセスの統制

- 特権操作はすべてログに記録する
- 特権操作には二重承認を適用する
- 特権アクセスは期限付きで付与する
- 定期的な特権アクセス棚卸しを実施する

---

## 8. アクセス制御の監査

### 8.1 定期監査

| 監査項目 | 頻度 | 実施者 |
|---------|------|--------|
| ロール割当の妥当性 | 四半期 | Admin |
| 権限の過剰付与チェック | 月次 | Change Manager |
| 特権アクセスの利用状況 | 月次 | Admin |
| AI 自律レベルの妥当性 | 四半期 | Governance Team |
| 退任者のアクセス無効化 | 発生時即時 | Admin |

### 8.2 監査証跡

以下のイベントを監査ログに記録する。

- ログイン/ログアウト
- 権限変更
- 特権操作の実行
- アクセス拒否イベント
- ロール変更
- AI 自律レベル変更

## 9. インシデント対応

### 9.1 不正アクセス検知時

```
1. 即時対応
   ├─ 該当アカウントの一時停止
   ├─ セッションの強制終了
   └─ 影響範囲の初期評価

2. 調査
   ├─ 監査ログの分析
   ├─ 不正アクセスの経路特定
   └─ 影響を受けたリソースの特定

3. 是正
   ├─ 認証情報のリセット
   ├─ 脆弱性の修正
   └─ アクセス制御の強化

4. 報告
   ├─ インシデントレポート作成
   ├─ 関係者への通知
   └─ 再発防止策の策定
```
