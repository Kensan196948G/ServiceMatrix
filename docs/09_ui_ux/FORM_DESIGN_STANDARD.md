# フォームデザイン標準定義

**ドキュメント番号**: SM-UI-004
**バージョン**: 2.0
**分類**: UI/UX設計仕様 / フォーム設計標準
**作成日**: 2026-03-02
**最終更新日**: 2026-03-02
**準拠規格**: ITIL 4 / ISO/IEC 20000 / WCAG 2.1 AA / J-SOX
**ステータス**: 承認済み

---

## 1. 目的と適用範囲

本ドキュメントは、ServiceMatrix Web UIにおける全フォームの設計標準を定義する。
一貫したユーザー体験の提供と、統治プロセスに必要なデータ品質の確保を目的とする。
WCAG 2.1 AAに準拠したアクセシビリティを設計の基本要件とする。

## 2. フォーム設計原則

### 2.1 基本原則

| 原則ID | 原則名 | 説明 |
|---|---|---|
| F-01 | 最小入力 | 必要最小限のフィールドのみ表示し、入力負荷を最小化する。省略可能な情報は後から追加できる設計にする |
| F-02 | 段階的開示 | 高度な設定項目は折りたたみ/展開で段階的に表示する。初心者は基本項目のみで操作完了できるようにする |
| F-03 | インライン検証 | フィールド離脱時（onBlur）に即座にバリデーション結果を表示する。送信時のまとめてエラー表示は行わない |
| F-04 | 文脈ヘルプ | 各フィールドにツールチップまたはヘルプテキストを付与する。専門用語には必ず説明を添える |
| F-05 | 自動補完 | 既存データからの自動補完・サジェストを可能な限り提供する |
| F-06 | 監査証跡 | フォーム送信は全て監査ログに記録する（送信者・日時・変更内容） |
| F-07 | 二重送信防止 | フォーム送信中はボタンを無効化し、重複登録を防止する |
| F-08 | ドラフト保存 | 複雑なフォームは入力中断時のデータ損失を防ぐドラフト保存機能を提供する |

### 2.2 アクセシビリティ原則（WCAG 2.1 AA）

| 要件 | 実装方法 |
|---|---|
| ラベル関連付け | 全フォーム要素に明示的な `<label>` または `aria-label` を設定 |
| エラー通知 | `role="alert"` + `aria-describedby` でエラーメッセージとフィールドを関連付け |
| フォーカス順序 | Tab移動順序を視覚的・論理的な順序に設定 |
| 必須フィールド表示 | 視覚的な「*」マーク + `aria-required="true"` の両方を付与 |
| カラーへの非依存 | エラー状態はアイコン・テキストでも表示（色のみに頼らない） |
| キーボード操作 | 全操作をキーボードのみで完結できる設計 |
| タイムアウト警告 | 入力タイムアウト前に60秒前の警告メッセージを表示 |
| コントラスト比 | テキスト最小4.5:1、大テキスト最小3:1（WCAG AA基準） |
| エラーの説明 | エラーメッセージには問題の内容と修正方法を明示 |
| 適切なHTML要素 | セマンティックなHTMLを使用（button、input type適切設定） |

## 4. バリデーション設計

### 4.1 フロントエンド・バックエンド二重チェック原則

```
フロントエンド（クライアントサイド）バリデーション:
  目的: 即時フィードバックによるUX向上
  タイミング: onBlur（フィールド離脱時）、onSubmit（送信時）
  実装: React Hook Form + Zod スキーマ

バックエンド（サーバーサイド）バリデーション:
  目的: セキュリティ確保・データ整合性保証
  タイミング: API受信時（全リクエスト必須）
  実装: Zod スキーマ（フロントエンドと同一定義を共有）
  原則: フロントエンドを信頼しない（Never Trust Client）
```

**スキーマ共有設計**:
```typescript
// packages/shared/schemas/incident.ts
// フロントエンドとバックエンドで同一スキーマを共有
import { z } from 'zod';

export const IncidentCreateSchema = z.object({
  title: z.string().min(1, 'タイトルは必須です').max(255, 'タイトルは255文字以内です'),
  description: z.string().min(10, '説明は10文字以上必要です').max(5000),
  impact: z.enum(['high', 'medium', 'low']),
  urgency: z.enum(['high', 'medium', 'low']),
  category: z.string().min(1, 'カテゴリは必須です'),
  affectedCiIds: z.array(z.string().uuid()).optional(),
  assigneeId: z.string().uuid().optional(),
  attachments: z.array(z.instanceof(File)).max(5, '添付は5ファイルまでです').optional(),
});

export type IncidentCreateInput = z.infer<typeof IncidentCreateSchema>;
```

### 4.2 バリデーションルールと表示タイミング

| バリデーション種別 | 表示タイミング | UI表現 |
|---|---|---|
| 必須チェック | onBlur | フィールド直下に赤テキスト + フィールド枠を赤色に |
| 形式チェック（メール・URL等） | onBlur | フィールド直下に赤テキスト |
| 文字数チェック | onChange（カウンター）+ onBlur（エラー） | 80%超過で黄、100%超過で赤カウンター |
| 範囲チェック | onBlur | フィールド直下に赤テキスト |
| 非同期重複チェック | onBlur（APIコール） | ローディングインジケーター → 結果表示 |
| 相関チェック（終了日≧開始日等） | onSubmit | 関連フィールド両方にエラー表示 |
| セキュリティチェック（XSS等） | onSubmit | 無害化または拒否 |

### 4.3 エラーメッセージ標準

| エラー種別 | メッセージパターン | 具体例 |
|---|---|---|
| 必須未入力 | `{フィールド名}は必須です` | 「タイトルは必須です」 |
| 文字数超過 | `{フィールド名}は{N}文字以内で入力してください` | 「説明は5000文字以内で入力してください」 |
| 文字数不足 | `{フィールド名}は{N}文字以上必要です` | 「説明は10文字以上必要です」 |
| 形式不正 | `{フィールド名}の形式が正しくありません（例: {例}）` | 「メールアドレスの形式が正しくありません（例: user@example.com）」 |
| 範囲外 | `{フィールド名}は{MIN}以上{MAX}以下の値を入力してください` | 「リスクスコアは1以上25以下の値を入力してください」 |
| 重複 | `この{フィールド名}は既に使用されています` | 「このCI名は既に使用されています」 |
| 過去日付 | `{フィールド名}は現在以降の日時を指定してください` | 「実施予定日は現在以降の日時を指定してください」 |
| 相関エラー | `{フィールドA}は{フィールドB}より後の日時を指定してください` | 「実施完了予定日は実施予定日より後の日時を指定してください」 |
| サーバーエラー | `入力内容の検証中にエラーが発生しました。しばらくしてから再試行してください` | - |

## 5. アクセシビリティ要件（WCAG 2.1 AA準拠）

### 5.1 フォームアクセシビリティ実装標準

```tsx
// WCAG 2.1 AA準拠のフィールド実装例
<FormField>
  <label
    htmlFor="incident-title"
    className="form-label"
  >
    タイトル
    <span aria-label="必須" className="required-mark">*</span>
  </label>

  <input
    id="incident-title"
    type="text"
    aria-required="true"
    aria-describedby={hasError ? "incident-title-error" : "incident-title-hint"}
    aria-invalid={hasError}
    className={cn("form-input", hasError && "form-input--error")}
    {...register("title")}
  />

  {/* ヘルプテキスト（エラーがない場合） */}
  {!hasError && (
    <p id="incident-title-hint" className="form-hint">
      インシデントの内容が簡潔にわかるタイトルを入力してください
    </p>
  )}

  {/* エラーメッセージ */}
  {hasError && (
    <p
      id="incident-title-error"
      role="alert"
      className="form-error"
      aria-live="polite"
    >
      <span aria-hidden="true">⚠</span>
      {errors.title?.message}
    </p>
  )}
</FormField>
```

### 5.2 フォームエラーサマリー

```tsx
// 複数エラー発生時のエラーサマリーコンポーネント
// フォーム上部に表示し、スクリーンリーダーに読み上げさせる
{hasErrors && (
  <div
    role="alert"
    aria-label="入力エラー"
    className="form-error-summary"
    tabIndex={-1}
    ref={errorSummaryRef} // 自動フォーカス用
  >
    <h3>以下の{errorCount}件のエラーを修正してください:</h3>
    <ul>
      {errors.map(error => (
        <li key={error.field}>
          <a href={`#${error.fieldId}`}>{error.message}</a>
        </li>
      ))}
    </ul>
  </div>
)}
```

### 5.3 キーボードナビゲーション要件

```
Tab順序: 画面上部から下部へ、左から右への自然な流れ
Enterキー: フォーカスのある送信ボタンでフォーム送信
Escapeキー: モーダルフォームを閉じる（確認ダイアログ付き）
スペースキー: チェックボックス・スイッチのトグル
方向キー: ラジオボタングループ・セレクトボックスの選択移動
フォーカスリング: すべてのフォーカス可能要素に明確な視覚的フォーカスリング
フォーカストラップ: モーダルダイアログ内でのフォーカストラップ実装
```

## 9. フォーム操作パターン

### 9.1 送信フィードバック

| 状態 | UI表現 | 継続時間 |
|---|---|---|
| 送信中 | ボタンをローディング状態に変更（二重送信防止） | 処理完了まで |
| 送信成功 | トースト通知（緑）+ 詳細画面へ自動遷移 | 3秒 |
| バリデーションエラー | エラーサマリー表示 + 最初のエラーフィールドへスクロール | 手動解決まで |
| サーバーエラー（4xx） | フォーム上部に赤バナー + エラー詳細（開発環境のみ） | 手動解決まで |
| サーバーエラー（5xx） | フォーム上部に赤バナー + 「しばらくしてから再試行」メッセージ | 手動解決まで |

### 9.2 ドラフト保存

```
対象フォーム: RFC作成、問題登録（入力項目が多いフォーム）
自動保存間隔: 30秒（変更があった場合のみ）
保存先: LocalStorage（ブラウザローカル）+ サーバーサイド（Draftステータス）
復元: ページ再読み込み時に「下書きが保存されています。復元しますか？」ダイアログ表示
有効期限: 24時間（LocalStorage）/ 7日間（サーバーサイドDraft）
ドラフト削除: 送信完了時に自動削除
```

### 9.3 確認画面の表示基準

| フォーム種別 | 確認画面 | 理由 |
|---|---|---|
| インシデント登録 | なし | 迅速性を優先（緊急時対応） |
| RFC作成 | あり（Step5として） | 変更管理の慎重な判断 |
| Emergency Change | あり + 赤警告バナー | 最大リスクの変更 |
| 問題登録 | なし | 定型的な登録フロー |
| CI編集 | あり | CMDB整合性の確保 |
| ユーザー削除・無効化 | あり + 確認テキスト入力 | 不可逆操作の保護 |

### 9.4 キャンセル時の動作

```
入力済みデータがある場合:
  → 「変更を破棄しますか？（入力内容は保存されません）」確認ダイアログ

ドラフト保存機能があるフォームの場合:
  → 「変更を保存しますか？ [保存する] [破棄する] [キャンセル]」

ブラウザの戻るボタン・ページ離脱:
  → beforeunload イベントで同様の確認処理
  ※PWAでは離脱前確認がより確実に動作
```

## 10. フォームとGitHub Issue連携

### 10.1 フォーム送信→GitHub Issue変換

| フォーム | GitHub操作 | ラベル自動付与 | テンプレート |
|---|---|---|---|
| インシデント登録 | Issue新規作成 | `incident`、`priority/P{n}`、カテゴリ | `incident_report.yml` |
| RFC作成 | Issue新規作成 | `change`、`rfc`、変更タイプ | `change_request.yml` |
| 問題登録 | Issue新規作成 | `problem` | `problem_record.yml` |
| ステータス変更 | Issueコメント追加 + ラベル変更 | `status/{status}` | - |
| コメント追加 | Issueコメント追加 | - | - |

### 10.2 フォーム送信のバックエンド処理フロー

```
1. フォームデータ受信（API）
     ↓
2. バックエンドバリデーション（Zod）
     ↓
3. DBへのレコード登録（PostgreSQL）
     ↓
4. GitHub Issue作成（GitHub API）
     ↓（並行処理）
5. 監査ログ記録（非同期）
6. 通知送信（非同期）
7. AI分析キュー登録（非同期）
     ↓
8. フロントエンドへのレスポンス返却（登録ID・GitHubIssue URL含む）
```
